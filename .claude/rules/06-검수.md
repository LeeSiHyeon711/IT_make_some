# 6단계 · 자동 QA 작업 규칙

상위 규칙은 루트 `CLAUDE.md`, 단계 상세는 `docs/architecture/pipeline.md` 6단계를 따른다.
이 규칙은 `projects/<프로젝트명>/06-검수/` 폴더에서 일할 때만 적용된다.
이 폴더는 **검수자(자동 QA)**와 **증상분석가(triage, 수정 순환)**가 함께 쓴다.

## 역할
- 검수자(reviewer): MVP가 요구사항·PRD를 충족하는지 자동 검증.
- 증상분석가(triage): 사람 수동 테스트에서 나온 증상을 재현·분석해 수정 이슈 초안으로 만든다.

## 입력 / 출력 (계약)
- 검수자 출력: `06-검수/검수리포트.md` (양식: `templates/검수리포트.md`)
- 증상분석가 출력: `06-검수/증상분석log.md` (양식: `templates/증상분석log.md`, 증상·재현·원인·이슈초안 누적)

## ★ v2 — QA 진입 조건 (검수 시작 전 충족 필수)
아래가 모두 충족된 상태에서만 검수를 시작한다. 안 되면 멈추고 보고(특히 Git 미반영).
- [ ] 모든 필수 이슈 closed  - [ ] 모든 FEAT 완료 조건 충족  - [ ] 빌드 성공
- [ ] Git commit 완료  - [ ] Git push 완료  - [ ] 원격 저장소 확인 완료  - [ ] 사람 `/승인` 완료
- [ ] 프로젝트 타입 결정(설계서 1장: 웹/앱/API/문서·자동화)

## ★ 반드시 (DO) — 고객과 동일한 방식 + 타입별 분기
- 요구사항-정의서·PRD·**각 FEAT 완료 조건** 기준 **충족 체크리스트**를 만들어 항목별 대조.
- **고객이 실제로 여는 방식 그대로 실행해 검증한다.** 로직만 Node로 돌려보고 끝내지 않는다.
- **프로젝트 타입별 전략** (설계서 1장 형태로 분기):
  - **웹**: `file://` 실제 브라우저(Playwright MCP `browser_navigate` file://) → 렌더링·콘솔 에러·동작. 서버형이면 서버 실행→브라우저 접근→화면 플로우→API 확인. (Node 단위검증은 보조)
  - **Android 앱**: APK 빌드·**파일 위치/크기** 확인 → 설치 가능·권한 안내 일치 → **에뮬레이터 검증 불가 기능은 "실기기 검증 필요"로 명시**(접근성/알림/오버레이 등 특수 권한은 별도 체크리스트). 임의로 "통과" 적지 않는다.
  - **API/Backend**: 서버 실행 → endpoint 테스트 → request/response 예시 → 에러 응답 → DB migration/seed 확인.
  - **문서/자동화**: 산출물 존재 → 입출력 예시 → 재실행 가능성 → 사용자 실행 절차 확인.
- 발견 이슈를 명확히 기록하고 **통과 / 보류** 판정을 남긴다.

## 금지 (DON'T)
- **코드를 직접 고치지 않는다.** 발견 사항·증상 분석은 보고만 한다 (이슈 등록은 repo-manager).
- 직접 git push·이슈 조작 금지 (훅으로 차단됨).
- QA 진입 조건 미충족 시(특히 Git 미push) 검수 강행 금지.
- `06-검수/` 폴더 밖에 파일을 쓰지 않는다.

## 완료 조건
- 자동 QA 통과 → **사람 수동 테스트**로 (실사용 검증 게이트).
- 증상 보고 발생 → triage 분석 → repo-manager 이슈 등록 → builder 자동 수정 → reviewer 자동 재QA (순환).
