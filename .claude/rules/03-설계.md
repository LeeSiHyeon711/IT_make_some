# 3단계 · 설계 작업 규칙 (★ 형태 결정 분기점)

상위 규칙은 루트 `CLAUDE.md`, 단계 상세는 `docs/pipeline.md` 3단계를 따른다.
이 규칙은 `projects/<프로젝트명>/03-설계/` 폴더에서 일할 때만 적용된다.

## 역할
결과물 형태(웹/앱/자동화)와 기술 스택·구조를 **확정**한다. 공방에서 가장 중요한 의사결정.

## 입력 / 출력 (계약)
- 입력: `02-기획/PRD.md`
- 출력: ① `03-설계/설계서.md` + ② `03-설계/features/FEAT-NN-*.md` (이슈 1개당 1개)
- 양식: 설계서는 `templates/설계서.md`, FEAT 문서는 `templates/FEAT.md` 를 복제해 채운다

## 반드시 (DO)
- **형태 결정**: "가장 빠르게 확인시킬 수 있는" 형태를 고른다.
- **스택 결정 + 근거 명시**: 왜 이 스택인지 한 줄이라도 적는다.
- 파일·모듈 구조, 핵심 화면/플로우 스케치를 포함한다.
- ★ **FEAT 문서 생성 (v2)**: 작업 분해를 **이슈 1개 = FEAT 문서 1개**로 펼쳐 쓴다.
  - 파일명 `FEAT-NN-<기능명>.md`, 번호는 등록될 **이슈 번호와 일치**(작업 분해 순서대로 01, 02, …).
  - 10개 섹션을 빠짐없이 채운다 — builder가 **이 문서 하나만 읽고** 구현할 수 있을 만큼 구체적으로(함수·클래스명·수정 파일까지).
  - **이슈 개수 최적화**(헌법 2장 7): 무작정 잘게 쪼개지 말 것. 소형 5~8 / 중형 8~12 / 복잡 12~15개 권장. 관련 작업(예: 데이터 클래스+저장소+Repository → "저장 계층 구현")은 묶는다. 단, UI와 DB 대규모 변경·권한과 비즈니스 로직·서버와 화면·핵심 로직과 디자인은 **섞지 않는다.**

## 금지 (DON'T)
- 실제 코드 작성·빌드 금지 — 구현은 5단계.
- 과한 스택 선택 금지: 정적 프로토타입으로 충분하면 풀스택으로 가지 않는다.
- `03-설계/` 폴더 밖에 파일을 쓰지 않는다. 1~2단계 산출물을 수정하지 않는다.

## 완료 조건
개발자가 FEAT 문서만 보고 그대로 만들 수 있고, GitHub 관리자가 FEAT 1:1로 이슈를 등록할 수 있다.
→ **사람이 설계서 + FEAT 문서를 함께 승인**(방향 결정 + FEAT 승인 게이트). 승인 = 개발 착수 승인이며, 이후 4→5→6은 `/개발착수`로 자동 진행된다.
