# 4단계 · GitHub 관리 작업 규칙 (★ 흐름 영속화)

상위 규칙은 루트 `CLAUDE.md`, 단계 상세는 `docs/pipeline.md` 4단계를 따른다.
이 규칙은 `projects/<프로젝트명>/04-깃허브/` 폴더에서 일할 때만 적용된다.

## 역할
승인된 설계서를 추적 가능한 GitHub 이슈로 분해 등록하고, 이후 이슈 상태를 동기화한다.
**공방 내에서 `gh`/git push를 가진 유일한 단계.**

## 입력 / 출력 (계약)
- 입력: `03-설계/설계서.md`
- 출력: `04-깃허브/이슈등록내역.md` (등록 이슈 목록 + 번호/링크 + 마일스톤 + **repo URL**) + 실제 GitHub 이슈
- 양식: `templates/이슈등록내역.md` 를 복제해 채운다

## repo 정책 (확정)
- **고객 프로젝트마다 별도 repo.** repo 이름 = 프로젝트명. 소유자 = 현재 gh 인증 계정(`gh api user --jq .login`).
- 첫 실행 시 repo가 없으면 `gh repo create <소유자>/<프로젝트명> --private` 로 생성한다. (`gh` 인증은 전제)
- repo URL을 반드시 `이슈등록내역.md` 최상단에 기록한다.

## ★ gh 명령은 반드시 `-R <소유자>/<프로젝트명>` 플래그 사용
`projects/<프로젝트명>/` 은 로컬 git repo가 아니므로 대상 repo를 플래그로 명시해야 한다(빼면 에러).
`gh issue create/list/close/reopen ... -R <소유자>/<프로젝트명>`

## 반드시 (DO) — ★ FEAT 1:1 매칭 등록 (v2)
- **FEAT 문서 1개 = 이슈 1개.** 설계서를 임의로 다시 쪼개지 말고 `03-설계/features/FEAT-NN-*.md` 그대로 등록한다.
- **이슈 번호 = FEAT 번호.** FEAT-01부터 순서대로 등록해 GitHub 이슈 번호(#1, #2, …)를 일치시킨다.
- 이슈 본문은 `templates/이슈본문.md` 형식 — **설계서 전체 복붙 금지**, 매칭 FEAT 문서만 가리킨다.
- 이슈 개수: 소형 5~8 / 중형 8~12 / 복잡 12~15 권장(헌법 2장 7). FEAT 수가 곧 이슈 수.
- 등록 직후 이슈 번호·링크·제목·**매칭 FEAT**·상태를 `이슈등록내역.md`에 기록한다.
- 첫 등록 시 개발 산출물 디렉토리를 git repo로 연결할 준비를 한다(아래 Git 반영).

## ★ v2 — Git 반영 검증 (생애주기 재호출 중 핵심)
v1 빈틈: 이슈는 닫혔는데 코드가 원격에 push되지 않았다. **commit/push 먼저, 이슈 close 나중.**
- builder 완료 보고 → 개발 repo에서 `git add -A` → `git commit -m "#N <작업명>"` → `git push` → `git log -1`/`remote -v`로 원격 반영 확인 → 그제서야 `gh issue close <N> -R …`.
- 동기화할 때마다 `이슈등록내역.md` 상태 + `progress.md`(마지막 commit/push) 갱신.
- 6단계 검수 이슈 발견 → 이슈 **재오픈 또는 신규 등록**(다음 FEAT 번호 이어서).

## ★ v2 — QA 진입 전 체크리스트 출력 (모든 이슈 close 후)
QA로 넘어가기 전 `git status`/`log -1`/`remote -v`/`branch` 확인 후 헌법 2장 9 체크리스트를 출력한다.
하나라도 ❌면 reviewer를 호출하지 않고 사람에게 보고. **사람 `/승인` 전까지 QA 진입 금지.**

## 금지 (DON'T)
- **코드 작성·설계 변경 절대 금지.** 코드는 commit/push만(내용 수정 안 함), 계획을 이슈로 "번역"하고 동기화만 한다.
- 다른 단계 폴더에 새 산출물을 쓰지 않는다(이슈등록내역·progress.md 갱신은 역할 범위).

## 완료 조건
모든 이슈가 등록·close되고 **코드가 원격 repo에 push 확인**되어, 이슈 목록 + 원격 commit만으로 상태를 파악할 수 있다.
→ QA 진입 체크리스트 출력 → 사람 `/승인` → 6단계 자동 QA.
