# 5단계 · 개발 작업 규칙

상위 규칙은 루트 `CLAUDE.md`, 단계 상세는 `docs/pipeline.md` 5단계를 따른다.
이 규칙은 `projects/<프로젝트명>/05-개발/` 폴더에서 일할 때만 적용된다.

## 역할
설계서대로 MVP를 실제로 만든다. (다형적 — 설계가 정한 형태/스택을 따른다)

## 입력 / 출력 (계약) — ★ v2 최소 문서 원칙 (헌법 2장 8)
- 입력(이 3개만): ① 루트 `CLAUDE.md` ② 현재 GitHub 이슈 본문(`gh issue view <N> -R …`) ③ 그 이슈와 1:1 매칭된 `03-설계/features/FEAT-NN-*.md`
- **설계서 전체·다른 단계 문서를 습관적으로 전부 읽지 않는다.** 추가 문서가 필요하면 이유를 빌드노트에 한 줄 적고 해당 문서만 콕 집어 읽는다(불명확하면 공방장/사용자 확인).
- 출력: `05-개발/` (실제 코드/결과물) + `05-개발/빌드노트.md` + `progress.md` 갱신
- 양식: 빌드노트는 `templates/빌드노트.md` 를 복제해 채운다

## ★ 핵심 규칙 — 1 이슈 = 1 FEAT (헌법 2장 7)
- 한 번의 작업(호출)에서는 **열린 이슈 중 하나만** 골라, 그 이슈의 **FEAT 문서 범위 안에서만** 처리한다.
- 절차: ① 이슈 1개 선택·명시(FEAT-NN 읽음 확인) → ② FEAT "구현할 것" 범위로 구현 → ③ 빌드노트 갱신 + **commit 가능 상태로 변경 파일 정리** → ④ GitHub 관리자에게 **완료 보고(변경 파일·빌드 결과 포함)** → ⑤ 호출 종료.
- 여러 이슈를 한 번에 몰아 처리하지 않는다. **끊겨도 그 지점부터 재개 가능**한 것이 목적이다.

> 참고: 개발 단계는 **사람 승인 없이 자동**으로 흐른다(헌법 2장 3). 공방장(`/개발착수`)이 이슈가 0이 될 때까지 너를 자동 재호출한다.
> "1이슈=1작업"은 사람 게이트가 아니라 **원자성·재개성**을 위한 것이다.

## 반드시 (DO)
- FEAT 문서가 정한 형태/스택/수정 파일 범위로만 구현한다.
- 빌드노트에 실행/설치 방법 + **이번 이슈 변경 파일 목록·빌드 결과**를 적는다 (commit/push는 GitHub 관리자가 하므로 commit 가능 상태로 남긴다).

## 금지 (DON'T)
- FEAT 범위 밖 기능 추가·다른 FEAT 선구현·관련 없는 리팩터링 금지 (스코프 고정).
- **직접 이슈 close·git commit·git push 금지** — 상태·commit·push는 GitHub 관리자 몫. 보고만 한다.
- `05-개발/` 폴더 밖에 파일을 쓰지 않는다(예외: `progress.md` 갱신). 이전 단계 산출물을 수정하지 않는다.

## 완료 조건
선택한 이슈 1개가 FEAT 완료 조건을 충족하고, commit 가능 상태로 정리·보고가 끝났다. 남은 이슈는 공방장이 자동 재호출.
모든 이슈 완료 → repo-manager가 commit/push + close → QA 진입 체크리스트 → 사람 `/승인` → 6단계 자동 QA.
