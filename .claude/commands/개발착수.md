너는 IT상상공방의 **공방장(오케스트레이터)** 이다. 직접 산출물을 만들지 말고 에이전트를 순서대로 호출한다.
이 커맨드는 **자동 구간(4→5)** 을 진행하고 **QA 진입 게이트에서 멈춘다**(헌법 2장 3 ②: Git 반영을 사람이 승인해야 QA로 간다).

## 0. 전제 확인
- `projects/$1/03-설계/설계서.md` 와 `03-설계/features/FEAT-*.md` 가 있어야 한다(설계+FEAT 승인 완료). 없으면 중단하고 `/다음단계 $1` 안내.
- `$1` 비면 묻고 멈춘다.

## 1. 4단계 — 이슈 등록 (없으면) ★ FEAT 1:1
`04-깃허브/이슈등록내역.md` 가 없으면 `subagent_type="repo-manager"`(최초 등록 모드) 호출.
→ repo 생성(`gh repo create <소유자>/$1 --private`), **FEAT 문서 1:1로 이슈 등록(번호 일치)**, 개발 repo git init·remote 연결, 내역 기록.

## 2. 5단계 — 개발 자동 루프 (이슈 단위)
열린 이슈가 0이 될 때까지 **반복**한다 (사람 개입 없음):
1. `subagent_type="builder"` 호출 — **열린 이슈 하나만**, 그 이슈의 FEAT 문서 범위로 구현(최소 문서, commit 가능 상태로 정리).
2. `subagent_type="repo-manager"`(Git 반영 모드) 호출 — 방금 완료 이슈를 **commit/push 먼저 → 원격 확인 → 이슈 close** + 내역·progress.md 갱신.
3. 남은 열린 이슈가 있으면 1번으로 (자동 계속). 0개면 루프 종료.

## 3. Git 반영 검증 + QA 진입 게이트 (★ v2 멈춤 지점)
`subagent_type="repo-manager"`(QA 진입 검증 모드) 호출 — `git status`/`log -1`/`remote -v` 확인 후 **QA 진입 전 Git 검증 체크리스트**(헌법 2장 9)를 출력한다.
- 하나라도 ❌(특히 push 미반영)면: 사람에게 보고하고 멈춘다. QA로 넘어가지 않는다.
- 전부 ✅면 **멈추고** 사람에게 안내:
  "개발 완료 + Git 원격 반영 확인됨. GitHub 웹에서 파일을 확인하신 뒤 `/승인 $1` 을 입력하면 자동 QA를 시작합니다."

> 자동 QA(6단계)와 그 이후 수동 테스트는 `/승인 $1` 이후에 진행된다. 이 커맨드는 QA를 직접 호출하지 않는다.
