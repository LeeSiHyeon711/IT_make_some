## 1. 결과물 형태 결정 ★

- **형태**: **웹 애플리케이션 (Next.js 14 App Router 기반 SPA + 서버 API Route)**
- **근거**:
  - PRD는 "브라우저 로컬 저장 기반, 인증 없음, 포트폴리오 수준 UI"를 요구한다 → **웹**이 가장 빠르게 확인시킬 수 있는 형태. 앱(네이티브)은 PRD에서 명시적으로 제외, 자동화는 성격에 맞지 않는다.
  - **순수 정적 SPA가 아니라 서버를 포함하는 이유는 오직 하나 — 설계 결정 #3(AI API 키 보호)** 때문이다. AI 키를 프론트에 노출하지 않으려면 서버 측 프록시가 반드시 필요하다. 그 외 모든 데이터(프로필·기록·대화·사진)는 **서버에 저장하지 않고 100% 브라우저 IndexedDB**에 둔다. 서버는 **상태를 저장하지 않는 AI 중계기**일 뿐이다.
  - Next.js 한 프레임워크 안에서 ① React 클라이언트 UI ② 서버 API Route(`/api/ai/*`) ③ `npm run dev` 단일 실행 ④ Vercel 무료 배포가 모두 해결된다 → builder(Sonnet)가 두 개의 서버를 엮을 필요 없이 단일 코드베이스로 구현. **"과한 스택"이 아니라 키 보호 요건을 만족하는 최소 스택**이다.

> 정적 프로토타입으로 충분하지 않은 유일한 사유 = API 키 보호. 이 요건이 없었다면 정적 SPA로 갔을 것이다. 요건이 있으므로 Next.js로 확정한다.

## 2. 기술 스택 결정 ★

| 영역 | 선택 | 근거 |
|------|------|------|
| 프레임워크 | **Next.js 14 (App Router) + React 18 + TypeScript** | 클라이언트 UI + 서버 API Route를 단일 코드베이스/단일 실행으로. 키 보호 요건 충족. TS로 데이터 구조 명확화 |
| 스타일 | **Tailwind CSS + CSS 변수 테마 토큰** | 따뜻·부드러운 디자인을 빠르게 일관 적용. 디자인 토큰(색/반경/그림자) 중앙화 → 포트폴리오 완성도(AC-16) |
| 한글 폰트 | **Pretendard (CDN/next/font)** | 깔끔·범용. 유아틱하지 않은 한글 가독성 |
| 로컬 저장 | **IndexedDB (`idb` 라이브러리)** | Promise 기반 경량 래퍼로 raw IndexedDB 보일러플레이트 제거 → Sonnet 구현 난이도↓. Blob 저장 지원 |
| 이미지 처리 | **브라우저 Canvas 리사이즈 (무의존)** | 업로드 사진을 최대 1280px/JPEG 0.8로 다운스케일 후 Blob 저장 → 용량 통제 |
| AI 호출 | **`@anthropic-ai/sdk` (서버 Route Handler 내부)** | 키를 서버 env(`ANTHROPIC_API_KEY`)에만 보관. 클라이언트는 `/api/ai/*`만 호출 |
| AI 모델 | **`claude-opus-4-8` (env `ANTHROPIC_MODEL`로 교체 가능)** | 가드레일(단정 회피·의료 표현 회피) 지시 준수 품질 우선. 비용 절감 필요 시 Sonnet급으로 env 교체 |
| 날짜 포맷 | **`date-fns` (경량) + 한국어 로케일** | 날짜 표시/정렬 일관 |
| 상태 관리 | **React Context + 커스텀 훅(무거운 상태 라이브러리 미도입)** | activePet/entries 정도 규모. Zustand 등 불필요 |
| 배포(선택) | **Vercel (정적 + 서버리스 함수)** | API Route가 그대로 서버리스로 동작. MVP 검증엔 로컬 `npm run dev`만으로 충분 |

> **추가 도입 라이브러리는 위 목록으로 한정**한다(idb, @anthropic-ai/sdk, date-fns, tailwind 생태). 그 외 임의 라이브러리 추가 금지(각 FEAT 10항).

### 2-1. AI 키가 없을 때의 동작(QA·오프라인 대비) — 중요
서버 Route Handler는 `process.env.ANTHROPIC_API_KEY`가 **없으면** 실제 API를 호출하지 않고 **결정적 Mock 응답**(가드레일 형식·완충 표현·조건부 병원 권장 포함)을 반환한다.
→ 키 없이도 전체 UI 흐름과 AC-06·11·12·13·14를 검증할 수 있고, 키를 넣으면 실제 AI로 동작한다. QA 진입 장벽을 없앤다.

## 5. 데이터 / 연동

### 5-1. IndexedDB 스키마 (DB명 `pawprint-diary`, version 1)
| 스토어 | keyPath | 인덱스 | 내용 |
|--------|---------|--------|------|
| `pets` | `pet_id` | — | 반려동물 프로필(프로필 이미지 Blob 포함) |
| `entries` | `entry_id` | `by-pet`(pet_id), `by-pet-date`([pet_id, date]) | 발자국 기록(사진 Blob 배열 + ai_summary 포함) |
| `conversations` | `conversation_id` | `by-pet`(pet_id) | AI 질문 대화 |
| `meta` | `key` | — | `activePetId` 등 단일값 |

- **저장 결정 #4**: `ai_summary`는 기록 저장 시 함께 `entries`에 저장(재생성 안 함). FEAT-04는 미포함 저장, FEAT-06이 병합(보정 지시 5).
- **저장 결정 #6**: 모든 entry/conversation은 `pet_id` 외래키. MVP는 activePetId 1개만 운용(전환 UI 없음 · 보정 지시 8).
- **저장 결정 #2(A안)**: 프로필 이미지·일기 사진 모두 Blob으로 저장. 표시는 `URL.createObjectURL` + 사용 후 revoke. **Blob은 AI 요청에 미포함(보정 지시 4).**
- **날짜/정렬(보정 지시 7)**: `date`=`YYYY-MM-DD`, `created_at`=ISO. 목록은 date desc → created_at desc.

### 5-2. 서버 API (상태 비저장 — AI 중계만)
| 엔드포인트 | 메서드 | 입력(JSON) | 출력(JSON) |
|-----------|--------|-----------|-----------|
| `/api/ai/summary` | POST | `{ pet, entry, recent[] }` (사진 제외, 텍스트/상태만) | `{ condition, behavior, observation, memory, vet_note? }` |
| `/api/ai/ask` | POST | `{ pet, question, recent[] }` | `{ answer }` |

## 7. 설계자 메모 (builder·repo-manager 참고)
- **읽기 최소화**: builder는 FEAT 1개로 충분하도록 작성했다. 공유 타입/함수 시그니처는 FEAT-02에 못 박고, 다른 FEAT는 그 이름을 참조만 한다.
- **★ 전역 빌드 규칙(보정 지시 10)**: **모든 FEAT는 작업 종료 시 `npm run build`가 타입/빌드 에러 0으로 통과해야 완료**다. 각 FEAT의 8장 완료 조건에 명시했고, 자동 공정이라도 이 게이트를 건너뛰지 않는다.
- **usePet 시그니처(보정 지시 1)**: FEAT-01은 임시 골격, FEAT-03이 `{ pet?, loading, refresh }`로 교체. 후속 FEAT는 이 시그니처를 기준으로 구현한다.
- **AI 안정화 순서(보정 지시 3)**: FEAT-05(서버/Mock/sanitize)를 06·07보다 먼저 안정화한다.
- **Blob 비전송(보정 지시 4)**: 사진/프로필 이미지 Blob은 IndexedDB 저장 전용이며 AI 요청에 절대 포함하지 않는다.
- **구현 금지**: 이 단계는 설계만. 코드·빌드는 5단계.
- **승인 게이트**: 본 설계서 + 9개 FEAT 문서를 사람이 함께 승인해야 `/개발착수`로 4단계 진행.
