# 09 · 데모 시나리오 (Demo Scenarios)

> 엔드투엔드 예시. **자유 서술 → 소통 에이전트 → 공방 접수 확인 → 균열점 진단 → 첫 검증 미션 → 공방 결과카드**의 전체 흐름을 보여준다.
> 세 시나리오는 `pain_source`별 진단 분기(05 문서 4번)를 의도적으로 다르게 골랐다.

---

## 시나리오 A — SELF (직접 경험형)

### [1] 자유 서술 (사용자)
> "저는 프리랜서 디자이너인데, 클라이언트마다 계약서랑 인보이스를 매번 메일·카톡·노션에 흩어놓다 보니 정산 때 누락이 생겨요. 이걸 한 곳에서 관리하는 웹 서비스를 만들면 좋을 것 같아요. 핵심은 계약별로 입금 여부를 자동 추적하는 거예요."

### [2] 소통 에이전트 — 접수 JSON
```json
{
  "input_summary": "프리랜서가 계약·인보이스·입금 여부를 한 곳에서 관리하는 웹 서비스",
  "service_type": "웹",
  "problem": "계약/인보이스가 여러 채널에 흩어져 정산 때 누락 발생",
  "target_user": "1인 프리랜서(디자이너 등)",
  "pain_source": "SELF",
  "maturity": "SOLUTION",
  "validation_time_budget": "TWO_DAYS",
  "needs_clarification": false
}
```
> 정보가 충분 → **추가 질문 없이** 시간 예산만 확인하고 접수 확인으로.

### [3] 공방 접수 확인 (사용자 노출)
```
🧾 공방 접수 확인
만들고 싶은 것: 계약·인보이스·입금을 한 곳에서 관리하는 웹 서비스
해결하고 싶은 문제: 여러 채널에 흩어져 정산 때 누락이 생김
처음 쓸 사람: 1인 프리랜서
아이디어 출처: 직접 경험
현재 아이디어 상태: 해결책까지 있음
검증에 쓸 수 있는 시간: 2일 이내
이 내용으로 공방 작업을 시작해도 될까요?
1. 좋아요, 진행해주세요   2. 수정하고 싶어요
```
→ 사용자: **1. 진행**

### [4] 균열점 진단 — JSON
> SELF형이므로 "문제가 있나?"를 묻지 않고 **빈도·강도·해결방식 적합성**에 집중.
```json
{
  "diagnosis": {
    "problem_statement": "계약/인보이스 분산으로 정산 누락",
    "target_user_assumption": "프리랜서가 별도 도구를 새로 들여 관리 습관을 바꿀 것이다",
    "context_of_use": "계약 체결·입금 확인 시점",
    "crack_point": "'관리할 곳이 없어서'가 아니라 '입금을 제때 확인·기록하는 습관이 없어서' 누락일 수 있다 — 새 도구가 습관 문제를 풀어주는지가 먼저다",
    "misread_risks": [
      "도구 부재가 원인이라고 단정 (실제론 입력 습관 문제)",
      "자동 입금 추적이 은행 연동 없이는 어려울 수 있음"
    ],
    "positive_signals": [
      "본인이 매달 반복해 겪는 실제 불편",
      "정산 누락은 금전 손실이라 동기가 강함"
    ],
    "diagnosis_focus": "SOLUTION_FIT"
  }
}
```

### [5] 첫 검증 미션 — JSON (2일 이내)
> "만들지 않는 방법 먼저" — 코드 없이 **수동 표**로 습관 가설부터 확인.
```json
{
  "first_experiment": {
    "time_budget": "2일 이내",
    "mission_title": "2주치 계약·입금을 스프레드시트 한 장에 수동 기록해보기",
    "mission_steps": [
      "구글시트에 '계약명/금액/인보이스 발송일/입금 여부' 4칸만 만든다",
      "지난 2주 또는 진행 중 계약을 직접 채워본다",
      "채우는 동안 '누락이 도구 때문인지 습관 때문인지' 메모한다"
    ],
    "success_criteria": [
      "수동 기록만으로 누락이 눈에 띄게 줄어드는지 체감된다"
    ],
    "failure_signals": [
      "시트를 매번 여는 것조차 안 하게 된다(→ 도구가 아니라 습관/알림이 핵심)"
    ],
    "do_not_build_yet": [
      "은행/계좌 자동 연동",
      "로그인·회원 시스템",
      "예쁜 대시보드 UI"
    ],
    "next_step_if_passed": "수동 기록이 효과 있으면, 입력을 더 쉽게 만드는 최소 도구(폼/알림)부터 검토"
  }
}
```

### [6] 공방 결과카드 (사용자 노출)
```
🧾 공방 결과카드
💡 아이디어  계약·인보이스·입금을 한 곳에서 관리하는 웹 서비스
👤 처음 쓸 사람  매달 여러 건을 정산하는 1인 프리랜서
🔍 균열점 진단  누락의 원인이 '관리할 곳이 없어서'가 아니라 '확인·기록 습관이 없어서'일 수 있어요. 이걸 먼저 확인하면 좋아요.
🧪 첫 검증 미션  (투자 가능 시간: 2일 이내)
   2주치 계약·입금을 스프레드시트 한 장에 수동으로 기록해보기
   해야 할 일:
   1. 시트에 계약명/금액/발송일/입금여부 4칸만 만든다
   2. 진행 중·최근 계약을 직접 채운다
   3. 누락이 도구 때문인지 습관 때문인지 메모한다
🧷 왜 이 실험인가요?  작아 보이지만, '도구를 만들면 해결된다'는 가장 위험한 전제를 코드 없이 이틀 만에 확인하는 거예요.
🎯 성공 기준  수동 기록만으로 누락이 줄어드는 게 체감된다
✂️ 지금 만들지 않아도 되는 것  · 계좌 자동 연동  · 로그인  · 대시보드 UI
➡️ 다음 단계  효과가 있으면 입력을 쉽게 하는 최소 도구(폼/알림)부터 검토
```

## 시나리오 B — ASSUMED (시장 추정형)

### [1] 자유 서술
> "노인분들이 약을 자주 까먹는다고 하니까, 복약 시간을 알려주는 앱을 만들면 수요가 클 것 같아요."

### [2] 소통 에이전트 — 질문 1개
> 정보 부족(누가 직접 쓰는지 불명) → **선택지형 질문 1개**:
> "이 앱을 직접 쓰실 분은 누구에 가까울까요? ① 어르신 본인 ② 돌보는 가족 ③ 아직 모르겠어요"
> 사용자: **②**

접수 JSON 요약: `pain_source: ASSUMED`, `target_user: 돌보는 가족`, `maturity: PROBLEM`, `validation_time_budget: TODAY`.

### [3] 공방 접수 확인 → 사용자 **1. 진행**

### [4] 균열점 진단
> ASSUMED형 → **문제를 실제로 겪는 사람이 존재하는지** 본다.
```json
{
  "diagnosis": {
    "crack_point": "'노인이 약을 까먹는다'는 일반론은 맞지만, 이미 가족·약국·기존 알림 앱으로 충분히 해결되고 있어 '새 앱이 필요할 만큼의 불편'이 남아있는지가 먼저다",
    "misread_risks": [
      "일반적 사실(약 까먹음)을 곧 수요로 오해",
      "정작 쓸 사람(가족 vs 본인)이 다를 수 있음"
    ],
    "positive_signals": [
      "대상군이 크고 명확함"
    ],
    "diagnosis_focus": "PROBLEM_EXISTENCE"
  }
}
```

### [5] 첫 검증 미션 (오늘 안에)
```json
{
  "first_experiment": {
    "time_budget": "오늘 안에",
    "mission_title": "실제로 어르신을 돌보는 가족 3명에게 지금 어떻게 챙기는지 물어보기",
    "mission_steps": [
      "주변에서 부모/조부모 약을 챙기는 사람 3명을 찾는다",
      "'지금 어떻게 챙기세요? 불편한 점이 있나요?'만 묻는다",
      "이미 쓰는 방법(달력/알람/약통)과 진짜 불편을 메모한다"
    ],
    "success_criteria": [
      "3명 중 2명 이상이 '현재 방식이 불편하다'고 구체적으로 말한다"
    ],
    "failure_signals": [
      "다들 '지금도 별 문제 없다'고 한다"
    ],
    "do_not_build_yet": [
      "앱 화면·기능 설계",
      "알림 푸시 기술 검토",
      "어르신용 UI"
    ],
    "next_step_if_passed": "불편이 확인되면, 어떤 순간에 가장 불편한지 좁혀 다음 미션 설계"
  }
}
```

### [6] 결과카드 — 균열점은 "수요가 정말 남아있는가", 미션은 오늘 가족 3명 인터뷰. (양식은 08 참조)

## 시나리오 C — IMAGINED (상상 확장형)

### [1] 자유 서술
> "AI가 사용자의 꿈 내용을 분석해서 심리 상태를 알려주는 앱이요. 아침에 꿈을 적으면 AI가 해석해줘요."

### [2] 소통 에이전트
접수 JSON 요약: `pain_source: IMAGINED`, `maturity: SOLUTION`, `target_user` 모호, `validation_time_budget: 30_MIN`.
> 시간 예산이 매우 짧음(30분) → 가장 가벼운 미션 예정.

### [3] 공방 접수 확인 → 사용자 **1. 진행**

### [4] 균열점 진단
> IMAGINED형 → **사용 순간과 첫 사용자가 구체적인지** 본다.
```json
{
  "diagnosis": {
    "crack_point": "'꿈을 매일 적는 행동'이 전제인데, 일어나자마자 꿈을 기록하는 습관을 가진 사람이 거의 없다 — 첫 사용 순간 자체가 성립하는지가 먼저다",
    "misread_risks": [
      "AI 해석 품질보다 '입력 자체가 안 일어나는' 문제가 더 큼",
      "'재미있겠다'와 '계속 쓰겠다'를 혼동"
    ],
    "positive_signals": [
      "호기심 자극이 강해 첫 시도 유입은 쉬울 수 있음"
    ],
    "diagnosis_focus": "CONTEXT_OF_USE"
  }
}
```

### [5] 첫 검증 미션 (30분 이내)
```json
{
  "first_experiment": {
    "time_budget": "30분 이내",
    "mission_title": "나 자신이 내일 아침 꿈을 메모앱에 적을 수 있는지 1회 시도",
    "mission_steps": [
      "오늘 밤 머리맡에 '일어나면 꿈 한 줄 적기' 메모를 둔다",
      "내일 아침 실제로 적는지 본다",
      "안 적었다면 왜 안 했는지 한 줄 남긴다"
    ],
    "success_criteria": [
      "아무 보상 없이도 아침에 꿈을 한 줄이라도 적게 된다"
    ],
    "failure_signals": [
      "알람을 봐도 적기 귀찮아 건너뛴다(→ 입력 습관이 최대 장벽)"
    ],
    "do_not_build_yet": [
      "AI 해석 엔진",
      "심리 분석 로직",
      "앱 자체"
    ],
    "next_step_if_passed": "본인이 적게 되면, 주변 2~3명에게 같은 1일 실험을 시켜 입력 지속성 확인"
  }
}
```

### [6] 결과카드 — 균열점은 "매일 꿈을 적는 행동이 일어나는가", 미션은 본인 1일 자가 실험. AI 해석은 "지금 만들지 않아도 되는 것"으로 분류.

---

## 데모가 보여주는 핵심

1. **같은 6단계라도 `pain_source`에 따라 균열점 각도가 다르다** (SELF=해결방식 / ASSUMED=문제 존재 / IMAGINED=사용 순간).
2. **첫 미션은 항상 "만들지 않는 방법" 우선** — 스프레드시트, 인터뷰, 자가 1일 실험.
3. **시간 예산이 난이도를 정한다** (2일=수동 기록, 오늘=인터뷰 3명, 30분=자가 1회).
4. **결과카드는 짧고 행동 가능** — 오늘/내일 당장 할 수 있는 한 가지.
