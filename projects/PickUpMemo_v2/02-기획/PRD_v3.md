# 제품 요구사항 명세 (PRD) — PickUpMemo_v3

- 작성일: 2026-06-24
- 근거 문서: `01-상담/요구사항-정의서_v3.md` (사람 승인 완료)
- 기준 코드베이스: `projects/PickUpMemo_v2/` (v2 코드 위 업그레이드)

## 1. 제품 개요 — v2 대비 변경점

v2는 배민커넥트 배차 카드에서 메모 매칭 시 오버레이 팝업을 표시하는 Android 앱이다. v3는 이 기반 위에 **픽업지 → 전달지 실제 자전거 경로 거리와 예상 소요시간을 단일 팝업 안에 함께 표시**하도록 업그레이드한다.

| 구분 | v2 | v3 변경 내용 |
|---|---|---|
| 팝업 트리거 | 메모 매칭 성공 시에만 | 픽업지/전달지 추출 성공 시 (메모 유무 무관) |
| 팝업 내용 | 메모 단독 표시 | 메모 + 거리/소요시간 함께 표시 |
| 주소 추출 | StoreExtractor (상호 후보) | AddressExtractor 신규 추가 (주소 전담) |
| 외부 API | 없음 (온디바이스 전용) | 카카오 Local + 카카오모빌리티 Cycling Directions 추가 |
| Android 권한 | 접근성·오버레이 | 기존 + INTERNET 추가 |

v2의 3대 기능(로그 수집 / 메모 CRUD / 메모 팝업)은 **변경 없이 그대로 유지**된다. 자세한 내용은 2장 참조.

---

## 2. 유지 대상 — v2 3대 기능 (변경 없음 / 회귀 금지)

아래 기능은 v3 작업 완료 후에도 v2와 동일하게 동작해야 한다. 어떤 형태로든 회귀가 발생하면 v3는 수용 기준을 통과하지 못한다.

| 기능 ID | 기능명 | 동작 요약 | v3 처리 |
|---|---|---|---|
| V2-1 | 배민 로그 수집 | 접근성 서비스가 `com.woowahan.bros` 화면 텍스트를 수집해 Room DB(`baemin_logs`)에 저장 | 변경 없음 |
| V2-2 | 가게별 메모 CRUD | `storeName / branchName / content / tag` 메모 등록·조회·수정·삭제 | 변경 없음 |
| V2-3 | 가게별 메모 팝업 | 배차 카드 메모 매칭 시 오버레이 팝업(자동 닫힘) 표시 | v3에서 단일 팝업 안에 통합 (트리거 승격, 내용 확장) |

기존 v2 처리 흐름 (보존):
```
화면이벤트 → fullText 조립 → baemin_logs 저장
→ StoreExtractor(픽업지~전달지 사이 상호 후보)
→ MemoMatcher(storeName & branchName 포함)
→ DedupGuard(30초)
→ MemoPopupController.show(memo)
```

v3에서 이 흐름은 유지되며, AddressExtractor와 RouteService가 병행 경로로 추가된다. StoreExtractor와 MemoMatcher는 손대지 않는다.

## 3. 사용자 및 문제 정의

- 사용자: 배민커넥트 배달 라이더 1인 (v2와 동일, 추가 사용자 유형 없음)
- 문제: 배차 수락 전 짧은 시간 안에 거리/시간을 감으로 판단해야 하며, 앱 내 표시 거리는 직선거리이거나 실제 주행 경로와 다를 수 있다
- 목적: 메모 팝업이 뜨는 순간에 실제 자전거 경로 기준 거리와 소요시간을 함께 보여줌으로써 배차 수락 여부를 더 빠르고 정확하게 판단하게 한다

## 4. 기능 명세

### 4-1. 단일 팝업 통합 (신규)

기존 `MemoPopupController` 하나 안에 메모와 거리/시간(route 영역)을 함께 표시한다.

- 별도 `RoutePopupController` 또는 별도 팝업 창을 만들지 않는다
- 팝업 내부는 메모 영역(상단)과 route 영역(하단)으로 구성된다
- route 영역은 비동기로 갱신된다 (팝업 전체를 재생성하지 않음)
- v3.1 오르막 알림을 위한 빈 자리(힐 알림 영역)를 레이아웃에 마련해 둔다 (동작 없음)

### 4-2. 팝업 트리거 승격 (변경)

| 구분 | v2 | v3 |
|---|---|---|
| 팝업 발동 조건 | 메모 매칭 성공 시 | 픽업지/전달지 추출 성공 시 |
| 메모의 역할 | 팝업의 필수 조건 | 팝업 안의 선택적 요소 |

픽업지/전달지를 추출하지 못하면 팝업은 뜨지 않는다. 메모가 없어도 픽업지/전달지 추출에 성공하면 팝업이 뜬다.

### 4-3. 4가지 표시 케이스 (신규)

아래 4가지 케이스 외 조합은 없다. 각 케이스는 서로 배타적이다.

| 케이스 | 메모 | 거리 조회 | 표시 내용 |
|---|---|---|---|
| A | 있음 | 성공 | 메모 내용 + `약 X.Xkm · 약 N분` |
| B | 있음 | 실패 | 메모 내용 + `거리 정보 확인 불가` |
| C | 없음 | 성공 | `등록된 메모 없음` + `약 X.Xkm · 약 N분` |
| D | 없음 | 실패 | debug/test 빌드: 팝업 표시 (`거리 정보 확인 불가`) / release/production 빌드: 팝업 미표시 |

규칙:
- 메모가 있으면 거리 조회 실패 여부와 무관하게 항상 팝업을 표시한다 (케이스 B)
- 케이스 D의 release 빌드 미표시는 실배달 방해 방지 목적이다
- 케이스 D의 debug 빌드 표시는 개발 중 API 실패 상황을 확인하기 위한 목적이다

### 4-4. 비동기 갱신 + 지연 표시 (신규)

- `RouteService`가 비동기로 경로 조회 수행
- 결과가 오면 열려 있는 같은 팝업의 route 영역만 갱신 (팝업 전체 재생성 없음)
- 케이스 C (메모 없음 + 거리 미확정 상태)의 release 빌드는 route 조회 성공 시에만 팝업 표시 (지연 표시)
  - route 조회가 완료되기 전까지는 팝업을 띄우지 않음
  - 조회 성공 후 팝업을 열고 route 영역을 즉시 표시함

### 4-5. 주소 추출 — AddressExtractor 신규 (중요)

`StoreExtractor`는 기존 역할(픽업지~전달지 사이 상호 후보 추출, 메모 매칭 용도)만 유지하고 **확장하지 않는다.** 주소 추출 전담 역할은 신규 `AddressExtractor`가 맡는다. StoreExtractor와 AddressExtractor는 독립적으로 동작한다.

**소스 우선순위**:

| 우선순위 | 소스 | 방법 |
|---|---|---|
| 1순위 (소스 A) | 분리된 bros-textview 토큰열 | `"픽업지"` / `"전달지"` 세그먼트 다음 세그먼트 텍스트를 추출 |
| 2순위 (소스 B, fallback) | 신규배차_카드 desc 한 줄 | 콤마 분할 후 `"픽업지"` 다음 값 / `"전달지"` 다음 값 추출 |

소스 A 추출에 성공하면 소스 B는 사용하지 않는다.

실제 baemin_logs 샘플 기준:
- 픽업지 예시: `푸라닭 신림점` (상호/장소명)
- 전달지 예시: `서울 관악구 남부순환로161나길 13 **** (신림동)` (도로명 주소, 마스킹 포함)

### 4-6. 마스킹 주소 처리 — 전달지 geocode fallback 4단계 (신규)

전달지 주소에 `****` 마스킹이 포함될 수 있다. 좌표 변환(geocode)은 아래 순서로 시도한다:

| 단계 | 시도 내용 |
|---|---|
| 1단계 | 원문 그대로 geocode 시도 |
| 2단계 | `****` 및 괄호 동명 제거 후 재시도 |
| 3단계 | 도로명 일부(앞부분)로 재시도 |
| 4단계 | 카카오 Local API 키워드 검색으로 전환 |

픽업지(상호/장소명)는 카카오 Local API 키워드 검색만 사용한다. 4단계까지 모두 실패하면 `거리 정보 확인 불가`로 처리한다.

### 4-7. 경로 API — RouteService (신규)

- 픽업지 좌표 + 전달지 좌표를 입력으로 카카오모빌리티 Cycling Directions API 호출
- API 응답의 `duration` / `totalTime` 값을 예상 소요시간으로 사용
- 표시 형식: `약 X.Xkm · 약 N분`
- 조회는 비동기 수행, 완료 시 팝업 내 route 영역만 갱신
- 네트워크 실패 / 주소 변환 실패 / 경로 조회 실패는 모두 `거리 정보 확인 불가` fallback

### 4-8. 테스트 fallback — TestActivity 활용 (신규)

주소 추출이 실제 로그에서 불완전할 경우를 대비한 수동 검증 구조가 필요하다.

- 기존 `TestActivity` 또는 동등한 테스트 진입점 활용
- 픽업지/전달지 텍스트를 수동 입력해 전체 흐름(주소 변환 → 경로 조회 → 팝업 표시)을 재현 가능해야 함
- 실제 배민커넥트 앱 없이도 경로 조회 및 팝업 표시를 검증할 수 있어야 함

## 6. 핵심 사용자 흐름 (시나리오)

### 시나리오 A — 메모 있는 배차 카드 (케이스 A, 핵심 가치)

1. 라이더가 배민커넥트에서 신규 배차 카드 화면을 확인한다.
2. PickUpMemo_v2 접근성 서비스가 화면을 감지한다.
3. AddressExtractor가 소스 A(토큰열)에서 픽업지(`푸라닭 신림점`)와 전달지(`서울 관악구 남부순환로161나길 13 **** (신림동)`)를 추출한다.
4. StoreExtractor + MemoMatcher가 기존 흐름대로 `푸라닭 신림점` 메모를 매칭한다.
5. 픽업지/전달지 추출 성공 → 팝업이 즉시 열린다. 팝업에는 메모 내용이 먼저 표시된다.
6. RouteService가 비동기로 카카오 geocode + Cycling Directions API를 호출한다.
7. 결과가 오면 팝업 내 route 영역이 `약 2.3km · 약 18분`으로 갱신된다.
8. 라이더는 팝업 하나에서 메모와 거리/시간을 동시에 확인하고 배차 수락 여부를 판단한다.
9. 팝업이 자동으로 닫힌다 (v2와 동일 자동 닫힘 동작).

### 시나리오 B — 메모 없는 배차 카드 release 빌드 (케이스 C)

1. 라이더가 신규 배차 카드를 확인한다 (메모 미등록 가게).
2. AddressExtractor가 픽업지/전달지를 추출한다.
3. MemoMatcher가 매칭하지 못한다 (메모 없음).
4. RouteService가 비동기로 경로 조회를 수행한다.
5. 경로 조회 성공 시 팝업이 열린다: `등록된 메모 없음` + `약 X.Xkm · 약 N분`.
6. 경로 조회 실패 시(케이스 D release) 팝업이 열리지 않는다 — 실배달 방해 방지.

### 시나리오 C — TestActivity 수동 검증

1. 개발자/테스터가 TestActivity를 실행한다.
2. 픽업지 텍스트(`푸라닭 신림점`)와 전달지 텍스트(`서울 관악구 남부순환로161나길 13 **** (신림동)`)를 수동 입력한다.
3. 실행 버튼을 누르면 주소 변환 → 경로 조회 → 팝업 표시 전체 흐름이 재현된다.
4. 마스킹 주소의 geocode fallback 동작과 route 영역 갱신을 실제 배민 로그 없이 확인한다.

## 7. 비기능 명세

| 항목 | 내용 |
|---|---|
| API 레이어 분리 | 경로 API 호출부는 `RouteProvider` 인터페이스와 `KakaoRouteProvider` 구현체로 분리. 제공자 교체 시 구현체만 교체 |
| API 키 보안 | API 키는 소스 코드 하드코딩 금지. `local.properties` → `BuildConfig` 주입 방식 사용 |

| 예외 처리 통합 | 네트워크 실패 / geocode 실패 / 경로 조회 실패는 모두 `거리 정보 확인 불가` fallback. 앱 크래시 없음 |
| 빌드 타입 분기 | 케이스 D에서 debug/test 빌드와 release/production 빌드 동작이 다름. 빌드 타입별 분기 로직 필요 |
| 온디바이스 유지 | 카카오 Local + 카카오모빌리티 API 외 외부 통신 없음. 서버·OCR·AI 없음 |
| v2 기능 회귀 금지 | V2-1 로그수집 / V2-2 메모 CRUD / V2-3 메모 팝업이 v3 작업 후에도 정상 동작 |
| MVP 원칙 | 속도가 품질보다 우선. "확인 가능한 최소 결과물" 기준 유지 |

| 항목 | v3 범위 | v3.1 범위 |

## 10. 수용 기준 (Acceptance Criteria)

아래를 모두 충족해야 v3 완료로 본다. AC-8 회귀 금지는 절대 조건이다.

| 번호 | 기준 |
|---|---|
| AC-1 | 배민커넥트 배차 카드 화면에서 픽업지/전달지가 추출되면 단일 팝업이 뜬다 |
| AC-2 | 팝업 안에 메모(있을 경우)와 거리/시간 정보가 함께 표시된다 |
| AC-3 | 거리/시간 조회가 비동기로 완료된 후 팝업 내 route 영역이 갱신된다 (팝업 전체 재생성 없음) |
| AC-4 | 거리 조회 실패 시 팝업 안에 `거리 정보 확인 불가` 문구가 표시된다 (메모 있는 케이스 B) |
| AC-5 | 메모 없음 + 거리 실패(케이스 D)에서 debug 빌드는 팝업이 뜨고, release 빌드는 팝업이 뜨지 않는다 |
| AC-6 | TestActivity(또는 동등 테스트 진입점)에서 픽업지/전달지를 수동 입력해 전체 흐름(주소 변환 → 경로 조회 → 팝업 표시)을 재현할 수 있다 |
| AC-7 | API 키가 소스 코드에 하드코딩되어 있지 않다 (`local.properties` → `BuildConfig` 확인) |
| AC-8 | (회귀 금지 절대 조건) 기존 V2-1 로그수집 / V2-2 메모 CRUD / V2-3 메모 팝업이 v3 이후에도 동작한다 |
| AC-9 | v3.1 오르막 알림 인터페이스(`HillAlert`), 위험 도로명 데이터 모델, 팝업 내 빈 자리가 코드에 존재한다 (실제 동작 불필요) |

## 11. 범위 밖 (Out of Scope)

요구사항-정의서 §10과 일치한다.

| 항목 | 사유 |
|---|---|
| v3.1 오르막 주의 실제 동작 (리스트 매칭 로직) | v3.1로 분리 |
| 고도·경사도 API 연동 | v3.1 이후 |
| 카카오모빌리티 외 다른 경로 API 구현체 | 인터페이스 준비 후 추후 교체 가능 구조만. 구현체는 카카오 단독 |
| 배차 수락 후 상세 화면 패턴 감지 (`가게전화`) | v2 MVP에서 제외됨, v3에서도 유지 |
| 타 배달 플랫폼 확장 | v2 결정 유지, v3 범위 밖 |
| 서버 동기화 / 클라우드 저장 | 온디바이스 전용 원칙 유지 |
| 경로 지도 시각화 | 거리/시간 수치 텍스트 표시만. 지도 렌더링 불필요 |
