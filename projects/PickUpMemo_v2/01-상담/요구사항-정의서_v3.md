## 0. 이 문서의 위치

이 문서는 v2 요구사항-정의서(`요구사항-정의서.md`)를 **대체하지 않는다.** v2 산출물은 그대로 보존되며, 이 v3 문서는 v2 대비 **추가/변경 요구사항만** 다루되, 유지 대상 항목을 명시적으로 기록해 다음 단계(기획자)가 전체 그림을 파악할 수 있게 한다.

## 1. 유지 대상 — v2 기존 3대 기능 (절대 훼손 금지)

아래 기능은 v3 작업 후에도 그대로 동작해야 한다. 회귀 발생 시 v3는 수용 기준을 통과하지 못한다.

| 번호 | 기능명 | 동작 요약 |
|---|---|---|
| V2-1 | 배민 로그 수집 | 접근성 서비스가 `com.woowahan.bros` 화면 텍스트를 수집해 Room DB(`baemin_logs`)에 저장 |
| V2-2 | 가게별 메모 저장·조회 | `storeName / branchName / content / tag` 메모를 앱 내에서 등록·조회·수정·삭제(CRUD) |
| V2-3 | 가게별 메모 팝업 | 배차 카드 화면에서 메모가 매칭되면 오버레이 팝업(6초 자동 닫힘)으로 메모를 표시 |

기존 처리 흐름 (보존):

```
화면이벤트 → fullText 조립 → baemin_logs 저장
→ StoreExtractor(픽업지~전달지 사이 상호 후보)
→ MemoMatcher(storeName & branchName 포함)
→ DedupGuard(30초)
→ MemoPopupController.show(memo)
```

## 2. 무엇을 바꾸는가 — v3 목표

### 2-1. 핵심: 경로 거리 + 예상 소요시간 팝업 통합

기존 메모 팝업에 **픽업지 → 전달지 실제 이동거리와 예상 소요시간**을 함께 표시한다.

- 직선거리가 아닌 지도/경로 API 기반 실제 경로 거리
- 자전거/전기자전거에 가까운 경로 (카카오모빌리티 Cycling Directions API 사용)
- 표시 형식: `약 X.Xkm · 약 N분`
- 예상 소요시간은 API 응답의 `duration` / `totalTime` 값을 사용

### 2-2. 확장 골격: v3.1 오르막 주의 알림 (구조만, 동작은 다음 버전)

오르막 위험 도로명 리스트 매칭 방식의 "오르막 주의 알림" 기능을 위한 골격을 마련한다.
- v3에서는 인터페이스 / 데이터 모델 / 팝업 내 자리(레이아웃)만 준비
- 실제 매칭 동작 및 고도/경사도 API 연동은 v3.1에서 구현
- 이 요구사항은 구조 설계 의사결정을 위해 기획서에 명시하며, 개발 이슈로는 분리하지 않는다

## 4. 왜 필요한가 — 문제 / 목적

배달 라이더는 배차 수락 전 짧은 시간 안에 거리와 시간을 감으로 판단해야 한다. 앱 내 표시 거리는 직선거리이거나 실제 주행 경로와 다를 수 있다. v3는 메모 팝업이 뜨는 순간에 실제 자전거 경로 기준 거리와 소요시간을 함께 보여줌으로써, 라이더가 배차 수락 여부를 더 빠르고 정확하게 판단할 수 있게 한다.

## 5. 확정된 UX·동작 결정 (사람 승인 완료 — 재질문 금지)

아래 5개 결정은 이미 고객 확인이 완료된 사항이다. 기획·설계 단계는 이를 전제로 삼는다.

| 번호 | 항목 | 확정 결정 |
|---|---|---|
| ① | **팝업 분리 금지** | 거리/시간은 기존 `MemoPopupController` 기반 **단일 오버레이 팝업** 안에 메모와 함께 표시한다. 별도 `RoutePopupController` 또는 별도 팝업 창을 만들지 않는다. |
| ② | **팝업 트리거 승격** | 기존에는 "메모 매칭 시"에만 팝업이 떴으나, v3는 **픽업지·전달지 추출 성공 시** 메모 유무와 무관하게 단일 팝업을 띄운다. 메모는 팝업 안의 선택적 요소가 된다. |
| ③ | **4가지 표시 케이스** | 아래 §6-1 참조 |
| ④ | **비동기 갱신 방식** | `RouteService`가 비동기로 거리/시간을 조회하고, 결과가 오면 **같은 팝업 내부의 route 영역만** 갱신한다. 팝업 전체를 다시 그리지 않는다. |
| ⑤ | **메모 없는 release 빌드 지연 표시** | 메모가 없고 거리 정보가 비동기로 아직 확정되지 않은 release 빌드는 "route 조회 성공 시에만 표시(지연 표시)"로 처리한다. |

## 6. 기능 요구사항

### 6-1. 팝업 표시 케이스 (4종)

| 케이스 | 메모 | 거리 조회 | 표시 내용 |
|---|---|---|---|
| A | 있음 | 성공 | 메모 내용 + `약 X.Xkm · 약 N분` |
| B | 있음 | 실패 | 메모 내용 + `거리 정보 확인 불가` |
| C | 없음 | 성공 | `등록된 메모 없음` + `약 X.Xkm · 약 N분` |
| D | 없음 | 실패 | **debug/test 빌드**: `거리 정보 확인 불가` 팝업 표시 (디버깅 목적) / **release/production 빌드**: 팝업 미표시 (실배달 방해 방지) |

> 규칙: 메모가 있으면 거리 실패 여부와 무관하게 항상 팝업을 표시한다(케이스 B).

### 6-2. 주소 추출 — 실제 배민 로그 샘플 기준

실제 `baemin_logs` 샘플(Package: `com.woowahan.bros`)에서 확인된 구조:

```
픽업지 다음 텍스트 예: 푸라닭 신림점        (상호/장소명 → 카카오 키워드 검색 대상)
전달지 다음 텍스트 예: 서울 관악구 남부순환로161나길 13 **** (신림동)
                        (도로명 주소, **** 마스킹 포함)
```

**소스 우선순위**:
1. (소스 A — 우선) 분리된 `bros-textview` 토큰열에서 `"픽업지"` / `"전달지"` 세그먼트 다음 세그먼트 텍스트
2. (소스 B — fallback) 신규배차_카드의 `desc` 한 줄을 콤마 분할:
   `desc=배민배달, 조리완료, 픽업지, 푸라닭 신림점, 전달지, 서울 관악구 …, 포인트, 12.6P`

**전달지 주소 변환 fallback 순서**:
1. 마스킹(`****`) 포함 원문 그대로 geocode 시도
2. 실패 시 `****` 및 괄호 동명 제거 후 재시도
3. 실패 시 도로명 일부(앞부분)로 재시도
4. 실패 시 카카오 키워드 검색으로 전환

**픽업지(상호/장소명) 변환**: 카카오 Local API 키워드 검색 사용.

### 6-3. 경로 조회 (RouteService)

- 픽업지 좌표 + 전달지 좌표를 입력으로 카카오모빌리티 Cycling Directions API를 호출한다
- API 응답의 `duration` / `totalTime` 값을 예상 소요시간으로 사용한다
- 조회는 비동기로 수행하며, 완료 시 팝업 내 route 영역만 갱신한다
- 네트워크 실패 / 주소 변환 실패 / 경로 조회 실패는 모두 `거리 정보 확인 불가`로 처리한다

### 6-4. 수동 입력·테스트 fallback (TestActivity 활용)

주소 추출이 실제 로그에서 불완전할 경우를 대비해, 테스트 데이터를 직접 입력해 경로 조회와 팝업 표시를 검증할 수 있는 구조를 마련한다.
- 기존 `TestActivity` 또는 동등한 테스트 진입점을 활용
- 픽업지/전달지 텍스트를 수동 입력해 전체 흐름(주소변환 → 경로조회 → 팝업 표시)을 재현 가능해야 한다

### 6-5. v3.1 오르막 주의 — 골격 요구사항

- `HillAlert` 인터페이스(또는 동등한 추상 레이어) 정의
- 오르막 위험 도로명 리스트를 담을 데이터 모델(로컬 리스트 방식) 정의
- 팝업 레이아웃 내 오르막 알림 영역 자리(빈 자리로 비워둠)
- 실제 매칭 로직 및 고도/경사도 API 연동은 v3 범위 밖 (v3.1로 분리)

## 8. 실제 로그 샘플 근거

v3 주소 추출 설계의 데이터 근거. 실제 `baemin_logs`에서 확인된 샘플:

```
Package: com.woowahan.bros

소스 A (토큰열):
  세그먼트: 픽업지
  세그먼트: 푸라닭 신림점        ← 픽업지 후보
  세그먼트: 전달지
  세그먼트: 서울 관악구 남부순환로161나길 13 **** (신림동)  ← 전달지 후보

소스 B (desc 한 줄 콤마 분할):
  desc=배민배달, 조리완료, 픽업지, 푸라닭 신림점, 전달지, 서울 관악구 남부순환로161나길 13 **** (신림동), 포인트, 12.6P
```

마스킹(`****`): 전달지 주소에 포함될 수 있으며, geocode 시 제거 fallback이 필요함을 위 샘플이 실증한다.

## 9. 수용 기준

아래를 모두 충족해야 v3 완료로 본다.

| 번호 | 기준 |
|---|---|

| AC-8 | **(회귀 금지)** 기존 V2-1 로그수집 / V2-2 메모 CRUD / V2-3 메모 팝업이 v3 이후에도 동작한다 |
| AC-9 | v3.1 오르막 알림 인터페이스/모델/팝업 자리가 코드에 존재한다 (동작 불필요) |

## 11. 용어 정리

| 용어 | 설명 |
|---|---|
| `MemoPopupController` | v2의 오버레이 팝업 제어 클래스. v3에서 거리/시간 표시도 이 컨트롤러 안에 통합 |
| `RouteService` | v3 신규. 픽업지·전달지 좌표를 받아 경로 API를 비동기 호출하고 결과를 반환 |
| `RouteProvider` | v3 신규 인터페이스. API 제공자 교체 가능 구조 |
| `KakaoRouteProvider` | `RouteProvider`의 카카오모빌리티 구현체 |
| `StoreExtractor` | v2 기존. 픽업지~전달지 사이 **상호 후보** 추출(메모 매칭 용도). v3에서 **확장하지 않고 기존 역할 그대로 유지**한다 — 기존 메모 매칭 회귀 위험 최소화 목적 |
| `AddressExtractor` | v3 신규. 픽업지·전달지 **주소 텍스트 추출**을 전담하는 별도 클래스. StoreExtractor를 건드리지 않고, 소스 A(토큰열) 우선·소스 B(desc 한 줄) fallback 방식으로 동작 |
