## v1 대비 개선 메모

- v2 핵심 변경: v1은 알림 리스너 기반이었으나, v2는 접근성 서비스 단독으로 전환하고 Room DB + CRUD UI + 오버레이 팝업을 새로 구현했다.
- 빌드 환경 이슈: Bash 도구 차단 환경으로 에이전트가 직접 빌드하지 못했다. 공방장이 JDK21로 수동 빌드(BUILD SUCCESSFUL, APK 5.8 MB)를 수행했다. 향후 빌드 가능한 환경에서 공정 자동화 여지 있음.
- Bash 차단 환경에서의 정적 QA: 실기기 없이 코드 정적 검증만 수행; AC 위반 0건이었으나 핵심 기능(배차 감지·팝업)은 실기기 확인 필수. 실기기 수동 테스트 게이트로 보완.
- v2.1 확장(FEAT-10~13): 1차 납품 후 사람 실기기 테스트에서 추가 요구 3건(팝업 검증 테스트화면·앱 아이콘·배민 로그 보관함)이 나와, `/증상` 흐름으로 architect가 FEAT-10~13 문서를 정식 작성→승인→개발했다(로그 보관함은 위험도상 저장계층/조회 2개로 분리해 #12·#13). 즉 별도 FEAT 문서 없이 처리한 것이 아니라 4개 문서를 추가했다. 추가로 수동 테스트 증상(입력폼 대비) #14를 수정했다.

## v3 핵심 인사이트

- **AI 협업 분담형 공정**: 무거운 판단(설계 109k Opus)은 Claude Code가 집중하고, 명세가 충분히 구체적인 구현(FEAT-16~23)은 Cursor에 위임한 뒤 Claude Code가 최종 검수만 수행 — "설계 품질 = FEAT 문서 품질"이라는 v2 전략(싼 모델로 구현)을 **외부 도구(Cursor) 위임**으로까지 확장. 인수인계 문서(`05-개발/인수인계_v3_FEAT16-23.md`)가 위임의 계약 역할.
- **버전 보존 규칙**: 모든 산출물 `_v3` 접미사 + 문서 내 버전 헤더로 v2 원본을 무손상 보존(요구사항-정의서/PRD/설계서/FEAT). 이전 결정 추적성 확보.
- **위험 격리 설계가 실전에서 작동**: 카카오 자전거 전용 엔드포인트 불확실성을 KakaoRouteProvider companion 상수 한 곳에 격리. 실기기 테스트에서 드러난 픽업 좌표 변환 실패도 `RouteResolveOutcome`/`RouteFailure` enum 덕에 원인 구분이 즉시 가능했다.
