## 이슈 #2 (FEAT-02) — 메모 저장 계층 (Room)

### 처리일: 2026-06-17

### 작업 개요
Room 기반 로컬 영속 저장 계층을 구현했다.
Entity(Memo), DAO(MemoDao), Database(AppDatabase), 싱글톤 저장소(MemoRepository) 4개 파일 신규 생성.

### 변경 파일 목록 (신규 생성)
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/data/Memo.kt`
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/data/MemoDao.kt`
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/data/AppDatabase.kt`
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/data/MemoRepository.kt`

### 구현 요점
- `Memo`: @Entity(tableName="memos"), 필드 id/storeName/branchName/content/tag?/updatedAt
- `MemoDao`: observeAll() Flow, getAll() suspend, getById() suspend, insert() suspend, update() suspend, delete() suspend
- `AppDatabase`: @Database(entities=[Memo::class], version=1, exportSchema=false)
- `MemoRepository` (object 싱글톤):
  - `init(context)` — DB 빌드만(멱등). suspend 호출 없음.
  - `observeAll()` — Flow<List<Memo>> 직접 반환
  - `add/update/delete` — withContext(IO) + 완료 후 refreshCache() 자동 호출
  - `getById(id)` — withContext(IO)
  - `refreshCache()` — suspend, IO에서 getAll() 호출해 cache 갱신
  - `getCachedSnapshot()` — 동기 반환(접근성 콜백용, 미초기화 시 emptyList)

### 진입점 사용 패턴 (FEAT-04/07에서 따를 것)
```kotlin
// Activity
MemoRepository.init(applicationContext)
lifecycleScope.launch { MemoRepository.refreshCache() }

// AccessibilityService (FEAT-07 전용 scope 사용)
MemoRepository.init(applicationContext)
serviceScope.launch { MemoRepository.refreshCache() }
```

### 빌드 결과
미실행 (Bash 도구 차단 환경). KSP가 Room 코드를 정상 생성하면 빌드 성공 예상.

### 완료 조건 체크
- [x] Memo @Entity — tableName="memos", 6개 필드
- [x] MemoDao @Dao — 6개 함수 (Flow + suspend)
- [x] AppDatabase @Database — entities=[Memo::class], version=1
- [x] MemoRepository object — init 멱등, CRUD suspend+IO, refreshCache, getCachedSnapshot
- [x] init 내부에서 suspend 함수 미호출 (FEAT-02 4장 규칙 준수)
- [x] add/update/delete 후 refreshCache() 자동 갱신
- [x] 입력 검증 없음 (FEAT-04 책임)

## 이슈 #5 (FEAT-05) — 업체명 추출·매칭·중복억제 로직

### 처리일: 2026-06-17

### 작업 개요
접근성 화면 텍스트에서 업체명 후보를 추출하고, 등록 메모와 강한 매칭(상호명+지점명 둘 다 포함)을 판정하며,
동일 메모 ID에 대해 30초 중복 표시를 억제하는 순수 로직 3개 파일을 신규 생성했다.
UI/오버레이/서비스와 분리된 단위 테스트 가능 구조.

### 변경 파일 목록 (신규 생성)
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/matcher/StoreExtractor.kt`
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/matcher/MemoMatcher.kt`
- `05-개발/PickUpMemo_v2/app/src/main/java/com/itmakesome/pickupmemo2/matcher/DedupGuard.kt`

### 구현 요점
- `StoreExtractor` (object): fullText에 "신규배차_카드" OR ("픽업지" AND "전달지") 없으면 null.
  "픽업지" 이후 ~ "전달지" 이전 추출 → id=/desc= 토큰 제거 → /|, 공백 변환 → 연속공백 1개 → trim. 빈값이면 null.
- `MemoMatcher` (object): detected.trim() + store/branch 각 trim(). isNotEmpty()인 경우에만 매칭 시도.
  store AND branch 둘 다 contains일 때 첫 memo 반환. 둘 중 하나만 일치 시 미매칭.
- `DedupGuard` (object): ConcurrentHashMap<Long,Long>, WINDOW_MS=30_000.
  마지막표시 없거나 30초 경과 → true+갱신, 아니면 false. reset() 포함.

### PRD 7-1 매칭 케이스 (명세 충족)
| 등록(상호/지점) | 감지 | 기대 |
|---|---|---|
| 푸라닭/신림점 | "푸라닭 신림점" | 매칭 |
| 푸라닭/봉천점 | "푸라닭 신림점" | 미매칭 |
| 푸라닭/신림점 | "푸라닭 봉천점" | 미매칭 |

### 빌드 결과
미실행 (Bash 도구 차단 환경). 공방장 빌드 검증 필요.

### 최초 권한 설정 절차 (앱 실행 전 필수)

앱은 두 가지 권한이 모두 활성화된 상태에서 동작한다. 순서를 따른다.

**1단계 — 접근성 서비스 활성화**
1. 기기 `설정` → `접근성` → `설치된 앱` (또는 `다운로드한 앱`)
2. **PickUpMemo** 항목 탭
3. 스위치를 **켬(ON)** 으로 전환 → 시스템 경고 대화상자에서 **허용** 선택

   또는 앱 실행 후 메인 화면의 **[접근성 설정]** 버튼으로 바로 이동 가능.

**2단계 — 다른 앱 위에 표시(오버레이) 허용**
1. 기기 `설정` → `앱` → **PickUpMemo** → `다른 앱 위에 표시`
2. **허용** 전환

   또는 앱 메인 화면의 **[오버레이 권한 설정]** 버튼으로 바로 이동 가능.

**3단계 — 앱에서 메모 등록**
1. 앱 실행 → 메인 화면의 **[메모 관리]** 버튼 탭
2. 우하단 **[+] FAB** 탭 → 상호명·지점명·메모내용 입력 → **[저장]**
   - 예시: 상호명 `푸라닭`, 지점명 `신림점`, 메모내용 `계단 3층, 코드 7823`
