## 5. FEAT-16~23 구현 순서 · 작업 요약 · 주의사항 (Cursor용)

> 아래 순서는 의존성 기반 권장 순서다. 각 FEAT는 `03-설계/features/FEAT-NN-*.md`가 정본이며, 요약은 길잡이일 뿐이다.

### FEAT-16 — AddressExtractor (이슈 #17) · 위험도 低
- **목표**: 화면 텍스트에서 픽업/전달 주소 추출.
- **신규 파일**: `matcher/AddressExtractor.kt` — `data class AddressPair(pickup, dest){ key() }`, `object AddressExtractor { fun extract(segments: List<String>, fullText: String): AddressPair? }`.
- **로직**: 소스 A(우선) = `segments`에서 선두 텍스트가 `"픽업지"`/`"전달지"`인 세그먼트의 **다음 세그먼트** 텍스트. 소스 B(fallback) = `신규배차_카드`의 `desc=...` 한 줄을 콤마 분할해 `픽업지`/`전달지` 다음 값. pickup·dest 둘 다 확보돼야 반환, 아니면 null.
- **실제 로그 예시**: 픽업=`푸라닭 신림점`, 전달=`서울 관악구 남부순환로161나길 13 **** (신림동)`.
- **주의**: **StoreExtractor를 수정/대체하지 말 것.** 완전 독립 신규 object.

### FEAT-17 — KakaoRouteProvider (이슈 #18) · 위험도 中
- **목표**: `RouteProvider` 카카오 구현체.
- **신규 파일**: `route/KakaoRouteProvider.kt` — `class KakaoRouteProvider(private val restApiKey: String) : RouteProvider`.
- **geocode(ADDRESS)**: `GET https://dapi.kakao.com/v2/local/search/address.json?query=` → `documents[0].x`(lng)/`y`(lat).
- **geocode(KEYWORD)**: `GET https://dapi.kakao.com/v2/local/search/keyword.json?query=` → 동일.
- **getRoute**: 카카오모빌리티 길찾기 `GET https://apis-navi.kakaomobility.com/v1/directions?origin={lng},{lat}&destination={lng},{lat}` → `routes[0].summary.distance`(m)/`duration`(s) → RouteInfo.
- 헤더 공통 `Authorization: KakaoAK $restApiKey`. **OkHttp 동기 execute()를 `withContext(Dispatchers.IO)`로 감쌈. JSON은 내장 `org.json`.** 모든 예외/비정상 응답 → `null`(throw 금지).
- **주의(★위험 격리)**: 카카오모빌리티 **자전거 전용 REST 엔드포인트 공식 제공이 불확실**. 엔드포인트 URL을 **이 파일 companion object 상수 한 곳에만** 둘 것. 자전거 미지원(404/403) 시 그 상수만 자동차 `/v1/directions`로 교체하면 distance/duration을 그대로 얻음(설계서 7-3). 키 없이 호출 시도 금지(빈 키면 RouteService가 막음).

### FEAT-18 — RouteService (이슈 #19) · 위험도 中
- **목표**: 캐싱 + 5초 timeout + dest 4단계 fallback.
- **신규 파일**: `route/RouteService.kt` — `object RouteService { const val TIMEOUT_MS = 5000L; fun init(apiKey: String); suspend fun resolve(pickup: String, dest: String): RouteInfo? }`.
- **로직**: 캐시 `ConcurrentHashMap<String,RouteInfo>` key=`"$pickup|$dest"`. **API 키 빈 문자열이면 즉시 null**(자동 fallback). 전체(geocode+route)를 **`withTimeoutOrNull(TIMEOUT_MS)`**로 감쌈.
- **dest geocode 4단계 fallback**: ①원문 `geocode(dest,ADDRESS)` → ②`****`·괄호동 제거 후 `ADDRESS` → ③도로명 앞부분(번지 절단) `ADDRESS` → ④`geocode(dest,KEYWORD)`. 각 성공 시 즉시 반환. pickup은 `geocode(pickup,KEYWORD)` 단일.
- **주의**: **timeout 5초 절대 유지**(배차 타이머 30초의 1/6). 모든 실패 경로 → null → 상위에서 "거리 정보 확인 불가".

### FEAT-19 — 단일 팝업 통합 (이슈 #20) · 위험도 高 ★
- **목표**: 기존 `MemoPopupController` **하나** 안에 메모 + route 영역 통합.
- **수정 파일**: `overlay/MemoPopupController.kt`, `res/layout/overlay_memo_popup.xml`.
- **레이아웃**: route 표시용 TextView(`tvPopupRoute`) + v3.1 오르막 자리 TextView(`tvPopupHill`, 기본 GONE) 추가. **기존 ID(tvPopupTitle/tvPopupContent/tvPopupTag) 절대 보존.**
- **컨트롤러**: `private val generation = AtomicLong(0)`, `@Volatile private var currentToken`. 신규 `fun show(context, memo: Memo?, hasRoute: Boolean): Long` + `fun updateRoute(token: Long, route: RouteInfo?)`. **기존 `fun show(context, memo: Memo)`는 `show(context, memo, hasRoute=false)`로 위임해 시그니처/동작 보존.**
  - `memo == null` → 제목 `등록된 메모 없음`(R.string), 내용/태그 GONE.

- **주의**: **별도 RoutePopupController 생성 절대 금지.** 토큰 세대 검증으로 6초 자동닫힘/교체 경합 안전. 오버레이 권한 미허가 시 기존처럼 skip하되 token은 반환.

### FEAT-21 — onAccessibilityEvent 결선 (이슈 #22) · 위험도 高 ★
- **수정 파일**: `service/PickupAccessibilityService.kt` (이 파일만).
- **추가**: `onServiceConnected`에 `RouteService.init(BuildConfig.KAKAO_REST_API_KEY)`(기존 Memo/BaeminLog init 보존). 기존 `serviceScope`(IO) 재사용 — **새 스코프 생성 금지**.
- **결선(설계서 5장 그대로)**:
  1. 기존 라인 보존: fullText/segments 조립 → `maybeLogBaemin` → `StoreExtractor.extract` → `MemoMatcher.match`.
  2. 신규: `addr = AddressExtractor.extract(segments, fullText)`.
  3. 트리거 승격: `if (matched == null && addr == null) return`.
  4. 중복: matched면 `shouldShow(matched.id)`, 아니면 `shouldShow(addr.key())`.
  5. 케이스 분기:
     - **메모 있음(A/B)**: 즉시 `show(matched, hasRoute)` → `hasRoute`면 `serviceScope.launch { updateRoute(token, RouteService.resolve(...)) }`.
     - **메모 없음 + `BuildConfig.DEBUG`(C/D-debug)**: 즉시 `show(null, hasRoute=true)` → resolve → updateRoute(실패해도 "거리 정보 확인 불가" 표시).
     - **메모 없음 + release(C/D-release)**: **지연 표시** — `serviceScope.launch { val r = resolve(...); if (r != null) { token = show(null, true); updateRoute(token, r) } }`. 실패면 아무것도 안 띄움.
- **주의**: 기존 `maybeLogBaemin`·`collectNode`·패키지 화이트리스트(TARGET_PACKAGE) **동작 변경 금지**. 접근성 콜백(메인 스레드)에서 **네트워크 동기 호출 금지**(반드시 serviceScope). `show`/`updateRoute`는 내부에서 mainHandler.post하므로 IO에서 호출 안전.

## 8. API Key 처리 정책 (재확인)
- 카카오 REST API Key는 **발급 완료**됨. **키 값은 채팅/문서/코드에 절대 기록하지 않는다.**
- 키는 **사용자가 `local.properties`에 직접 입력**한다: `KAKAO_REST_API_KEY=발급키` 한 줄. (`local.properties`는 **gitignored** — 커밋되지 않음. 키 유출 방지.)
- gradle이 `local.properties`에서 읽어 `BuildConfig.KAKAO_REST_API_KEY`로 주입. **소스/그래들 하드코딩 금지.**
- **키가 없어도 빌드는 성공해야 한다** — 항목 부재 시 빈 문자열 `""`로 주입되며, `RouteService.resolve`는 빈 키면 즉시 null 반환해 "거리 정보 확인 불가"로 안전 처리(크래시 없음). (FEAT-14에서 이미 구현·검증됨)
- 실제 거리/시간 검증은 사용자가 키 입력 후 수행.

## 9. Cursor에게 전달할 구현 지시문 (복사용)

```
PickUpMemo_v2 안드로이드 프로젝트의 v3 업그레이드를 이어서 구현한다.
작업 루트: projects/PickUpMemo_v2/05-개발/PickUpMemo_v2 (브랜치 main)
정본 명세: projects/PickUpMemo_v2/03-설계/설계서_v3.md + 03-설계/features/FEAT-16~23-*.md
이슈 매핑: 04-깃허브/이슈등록내역.md (이슈 #N = FEAT-(N-1))

구현 순서: FEAT-16 → 17 → 18 → 19 → 20 → 21 → 22 → 23.
각 FEAT는 해당 FEAT-NN 문서대로 구현하고, 끝나면 JDK21로 ./gradlew :app:assembleDebug 빌드가 통과해야 한다.

절대 준수:
- StoreExtractor 수정 금지 (주소 추출은 신규 AddressExtractor 전담)
- MemoPopupController 단일 팝업 유지, 별도 RoutePopupController 생성 금지
- RouteService 전체 timeout 5초(withTimeoutOrNull(5000)) 유지
- debug/release 표시 정책 유지 (케이스 D: debug 표시 / release 미표시)
- FEAT-23은 v3.1 골격만 (실제 오르막 감지/리스트 매칭/팝업 표시/서비스 결선 금지, NoopUphillDetector는 항상 null)
- 기존 v2 기능(로그수집/메모CRUD/메모팝업) 회귀 금지 (7장 체크리스트 통과)
- API 키 하드코딩 금지. local.properties→BuildConfig, 키 없어도 빌드 성공해야 함.

빌드: export JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" 후 ./gradlew :app:assembleDebug
(기본 JDK가 Java25면 Kotlin 플러그인 크래시 → 반드시 JBR21 사용)

커밋 메시지 규칙: "#<이슈번호> FEAT-<번호> <작업명>"
각 FEAT 완료 후 commit/push/이슈 close는 사용자/Cursor가 수행(Claude Code는 하지 않음).
```

## 10. 최종 검수 시 Claude(Claude Code)가 확인해야 할 항목
Cursor 구현 완료 후 Claude Code가 리뷰할 때 점검:
1. **회귀(7장 체크리스트) 전부 통과** — 특히 메모 매칭 팝업이 v2와 동일한지, StoreExtractor/MemoMatcher 불변인지 `git diff`로 확인.
2. **단일 팝업 원칙** — 별도 RoutePopupController가 생기지 않았는지, route가 기존 MemoPopupController 안에 통합됐는지.
3. **timeout 5초** — `RouteService`에 `TIMEOUT_MS=5000` + `withTimeoutOrNull` 적용 확인.
4. **debug/release 분기** — FEAT-21 결선에서 케이스 D가 `BuildConfig.DEBUG`로 분기(release 미표시/지연표시)되는지.
5. **API 키 안전** — 코드/그래들에 키 하드코딩 없음, `local.properties` 미커밋, 빈 키로 빌드·실행 시 "거리 정보 확인 불가" fallback·크래시 없음.
6. **FEAT-23 범위** — 오르막은 골격만(Noop, GONE), 실제 감지/결선/표시 없음.
7. **빌드** — JBR21로 `assembleDebug`/`assembleRelease` 둘 다 성공, APK 생성.
8. **이슈/커밋 정합** — 이슈 #17~#24 close + 각 커밋 원격 push 확인(매핑표와 일치).
9. **마스킹 주소 fallback** — 전달지 `****` 포함 시 4단계 geocode fallback 동작(키 입력 후 실측 권장).
10. 이상 발견 시 코드 직접 수정 대신 **증상 분석 → 수정 이슈** 흐름 또는 사용자 보고(공방 원칙).
