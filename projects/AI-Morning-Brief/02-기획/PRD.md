## 1. MVP 목표

AI 업계 최신 소식을 매일 새벽 자동으로 RSS/공식 블로그 5~8개 소스에서 수집하고, SQLite DB에 누적 저장한 뒤 Claude가 "왜 중요한지", "상상공방에 어떻게 적용할 수 있는지", "오늘의 액션 후보가 무엇인지"를 분석·브리핑한 Markdown 일일 리포트를 로컬에 생성한다. Discord Webhook으로 Claude가 생성한 핵심 브리핑을 발송해 운영자가 아침에 Discord 채널 하나를 열면 이미 분석된 브리핑이 와 있는 상태를 만든다.

V0.1 기본 파이프라인: 수집 → SQLite 저장 → Claude 분석/요약 → Markdown 리포트 생성 → Discord 전송.

Claude 연동 방식(Claude Code headless vs Claude API)은 설계 단계에서 선택한다. Claude 연동에 실패하더라도 raw JSON과 DB 저장은 반드시 완료되며, 리포트에 "AI 분석 실패 / 기본 리포트 생성"을 명시한 후 fallback 리포트를 생성한다.

## 4. 핵심 사용자 시나리오

### 시나리오 1. 정상 자동 실행 (매일 반복)

새벽 3~6시 사이 Mac이 켜진 상태에서 cron 또는 launchd가 메인 스크립트를 자동 실행한다. 스크립트는 설정 파일에 정의된 5~8개의 RSS/공식 블로그 소스를 순서대로 폴링해 기사를 수집한다. 이미 DB에 존재하는 URL은 건너뛰고 신규 기사만 SQLite에 저장한다. 수집된 기사 목록을 Claude에 전달해 각 기사의 중요도, 상상공방 적용 가능성, 오늘의 액션 후보를 분석받는다. Claude 분석 결과를 포함한 섹션별 Markdown 리포트를 생성해 로컬 날짜별 폴더에 저장한다. 마지막으로 Discord Webhook으로 Claude가 작성한 핵심 브리핑(수집 건수, 오늘의 핵심 변화 3~5개, 상상공방 적용 아이디어, 리포트 저장 경로 안내)을 2000자 이내로 전송한다. 운영자는 아침에 Discord 채널을 열어 분석된 브리핑을 확인하고, 더 상세한 내용은 로컬 Markdown 파일을 열어 확인한다.

### 시나리오 2. Claude 연동 실패 시 Fallback 실행

Claude 연동 중 오류가 발생했다 (API 오류, 타임아웃, 키 만료 등). 스크립트는 오류를 로그에 기록하고 fallback 모드로 전환한다. raw JSON과 SQLite DB 저장은 이미 완료된 상태를 유지한다. 키워드 추출 또는 첫 N문장 기반 기본 요약으로 리포트를 생성하며, 리포트 상단에 "AI 분석 실패 / 기본 리포트 생성"을 명시한다. Discord에는 fallback 리포트 기반의 기본 요약을 전송하고 AI 분석 실패 여부를 함께 안내한다. 데이터 유실 없이 다음 실행에서 Claude 재연동이 시도된다.

### 시나리오 3. 실패 복구 (Catch-up) 실행

전날 실행이 실패했다 (Mac 슬립, 네트워크 오류, 스크립트 오류 등). 다음 날 실행 시 스크립트는 DB에 기록된 마지막 성공 시각을 확인한다. 마지막 성공 이후 발행된 기사를 몰아서 수집하고 DB에 저장한다. Claude가 보완 수집된 기사 전체를 분석해 리포트를 생성한다. 리포트와 Discord 메시지에 "보완 수집" 표시와 수집 범위(시작 시각 ~ 종료 시각)를 명시해 운영자가 평소보다 수집 건수가 많더라도 이유를 바로 파악할 수 있다.

## 5. 데이터 저장 정책

### SQLite DB

- 저장 항목: URL (PK), 제목, 발행일, 출처명, 수집 시각, Claude 요약 텍스트, 키워드 태그, 중요도 점수, 상상공방 관련도 점수, Claude 분석 성공 여부
- 메타 정보: 마지막 성공 실행 시각, 마지막 리포트의 수집 범위(시작~종료 시각)
- 중복 방지: URL을 Primary Key로 사용, 삽입 시 중복이면 건너뜀
- 보관 기간: V0.1에서는 삭제 없이 전량 보관 (용량 및 보관 기간 정책은 V0.2 이후 수립)

### Raw JSON

- 소스별 원본 응답을 날짜/소스명 기준으로 파일 저장
- 재처리·디버깅 목적으로 보관 (Claude 연동 실패 시 재분석에도 활용)

### Markdown 리포트

- 날짜별 폴더 구조로 저장 (예: reports/2026-06-21/)
- Claude 분석 결과 포함 버전과 fallback 버전 모두 동일한 경로에 저장 (리포트 상단에 모드 명시)
- 파일명 규칙 및 정확한 디렉토리 구조는 설계 단계에서 확정

### 민감 정보

- Discord Webhook URL, Claude API 키 등은 환경 변수 또는 별도 비공개 설정 파일로 관리
- 설정 저장소에 하드코딩 금지

## 6. 실패 복구 정책

| 실패 유형 | 처리 방법 |
|-----------|-----------|
| 전체 실행 실패 (Mac 슬립, 오류 종료 등) | 다음 실행 시 마지막 성공 시각 이후 기사를 catch-up 수집. 리포트에 보완 수집 명시 |
| 단일 소스 수집 오류 | 해당 소스는 건너뛰고 나머지 소스 정상 진행. 리포트와 Discord 메시지에 오류 소스 목록 명시 |
| Claude 연동 실패 (API 오류, 타임아웃, 키 만료 등) | raw JSON과 DB 저장은 완료 상태 유지. fallback 기본 요약으로 리포트 생성. 리포트 상단에 "AI 분석 실패 / 기본 리포트 생성" 명시. Discord에도 fallback 여부 안내 |
| Discord Webhook 전송 실패 | 리포트 로컬 저장은 완료된 상태로 유지. 전송 실패는 로그에만 기록. 파이프라인 전체 성공 여부는 Discord 전송 결과와 무관하게 DB/파일 저장 기준으로 판단 |
| DB 쓰기 오류 | 오류 로그 기록 후 실행 중단. 다음 실행 시 catch-up으로 재수집 |

catch-up 기준 시각은 DB 메타 정보에 기록된 "마지막 성공 실행 완료 시각"을 사용한다.

## 7. Discord 보고 정책

- 전송 방식: Webhook 단방향 전송 (양방향 Bot 없음)
- Discord 메시지 내용: Claude가 생성한 핵심 브리핑, 2000자 이내
  - 리포트 날짜
  - 수집 범위 (시작 ~ 종료 시각)
  - 신규 수집 건수
  - 오늘의 핵심 변화 3~5개 (Claude 분석 기반 한 줄 요약)
  - 상상공방 적용 가능 소식 유무 및 요약
  - catch-up 실행 여부 안내 (해당 시)
  - Claude 분석 실패 여부 안내 (fallback 모드인 경우)
  - 로컬 리포트 파일 경로 안내
- 전체 Markdown 리포트: Discord에 첨부하지 않고 로컬 파일로만 저장
- Webhook URL: 환경 변수 또는 비공개 설정 파일로 관리, 절대 코드에 하드코딩하지 않음

## 8. MVP 성공 기준

V0.1 MVP는 아래 항목 전부를 충족해야 완성으로 인정한다.

1. 스크립트 수동 실행 시 파이프라인 전체 (수집 → 저장 → Claude 분석/요약 → 리포트 생성 → Discord 전송)가 오류 없이 완료된다.
2. cron 또는 launchd 스케줄로 사람 개입 없이 자동 실행된다.
3. Discord 채널에 Claude가 생성한 핵심 브리핑 메시지가 도착한다 (2000자 이내).
4. 로컬에 날짜별 Markdown 리포트 파일이 생성되며, Claude 분석 결과가 포함되어 있다.
5. SQLite DB에 수집 기사가 URL 중복 없이 쌓인다.
6. Catch-up 흐름이 동작한다 (마지막 성공 이후 기사 재수집, 보완 수집 명시 리포트 생성).
7. Claude 분석 모드와 fallback 모드(Claude 연동 실패 시)가 모두 동작한다.
8. 설정 파일 수정만으로 소스를 추가하거나 제거할 수 있다.
9. `--test` 모드로 실제 RSS/Claude/Discord 없이 파이프라인 전체를 검증할 수 있다.
10. 자동 QA가 아래 6개 항목을 모두 통과한다 (섹션 11 참조).

| 데이터 보관 기간 정책 | 오래된 raw JSON / 리포트 파일 자동 정리 |

## 10. 설계 단계로 넘기는 결정 사항

아래 항목은 PRD 범위를 넘으므로 3단계(설계)에서 결정한다.

- V0.1 RSS 소스 확정 목록 (5~8개 URL)
- 설정 파일 형식 (YAML / TOML / JSON 중 선택)
- Claude 연동 방식 (Claude Code headless vs Claude API)
- Claude 프롬프트 구조 (기사 목록 전달 방식, 분석 요청 항목)
- SQLite DB 스키마 상세 (테이블 구조, 컬럼 타입)
- 로컬 디렉토리 구조 (DB 파일 위치, raw JSON 경로, 리포트 경로)
- 로컬 스케줄러 방식 확정 (cron vs launchd)
- Discord 2000자 초과 시 처리 방식 (텍스트 절삭 vs 섹션 우선순위 기반 생략)
- Fallback 요약 방식 상세 (첫 N문장 / 키워드 추출 등)
- Python 패키지 의존성 및 실행 환경 구성 방법
- `--test` 모드 샘플 데이터 구조 및 fixtures 파일 위치 확정
- `--dry-run` 출력 형식 및 미리보기 범위

## 11. 테스트/검수 인터페이스

이 섹션은 **개발/검수용 내부 인터페이스**를 정의한다. 외부 사용자용 API가 아니며, QA 자동화와 수동 검증에서 사용한다.

### 목적

실제 외부 서비스(RSS 소스, Claude, Discord)에 의존하지 않고 파이프라인의 각 단계를 독립적으로 검증할 수 있어야 한다. 개발 중 빠른 반복과 자동 QA 실행, 개별 실패 시나리오 재현이 가능해야 한다.

### 실행 옵션

| 옵션 | 역할 |
|------|------|
| `--test` | 샘플 고정 데이터(`tests/fixtures/sample_articles.json`)로 수집 단계를 대체. 실제 RSS 요청 없이 수집 → DB 저장 → 분석/요약 → 리포트 생성 흐름 전체 검증 |
| `--dry-run` | 실행 계획과 리포트 미리보기를 출력만 하고, DB 쓰기·파일 저장·Discord 전송을 일절 수행하지 않음. 설정 파일과 소스 목록이 올바른지 사전 확인에 활용 |
| `--no-discord` | Discord Webhook 전송 단계를 건너뜀. 수집·저장·분석·리포트 생성은 정상 실행. Webhook URL이 없는 환경이나 Discord 없이 파이프라인만 검증할 때 사용 |
| `--force-fallback` | Claude 연동 단계를 강제로 실패 처리해 fallback 요약 경로와 "AI 분석 실패 / 기본 리포트 생성" 리포트 생성을 검증 |
| `--from <datetime>` | catch-up 수집 시작 시각을 수동 지정. DB의 마지막 성공 시각 대신 지정 시각을 기준으로 수집 |
| `--to <datetime>` | catch-up 수집 종료 시각을 수동 지정. `--from`과 함께 사용해 임의 기간의 catch-up 흐름을 재현 |

### 샘플 고정 데이터

- 위치: `tests/fixtures/sample_articles.json` (정확한 경로는 설계 단계에서 확정)
- 내용: 파이프라인 각 단계를 검증하기에 충분한 기사 데이터 (최소 5건 이상, 중복 URL 포함 케이스 포함)
- 형식: 실제 RSS 수집 결과와 동일한 구조로 구성해 수집 단계만 교체해도 이후 흐름이 동일하게 실행됨
- 옵션 조합: `--test` + `--no-discord` 조합으로 외부 의존성 없이 파이프라인 전체 통과 가능해야 함

### 자동 QA 최소 검증 항목

자동 QA(6단계)는 아래 6개 항목을 모두 통과해야 검수 완료로 인정한다.

| 번호 | 검증 항목 | 사용 옵션 |
|------|-----------|-----------|
| QA-1 | 샘플 기사가 SQLite DB에 정상 저장되는가 | `--test` |
| QA-2 | 중복 URL이 두 번 저장되지 않는가 (sample_articles.json에 중복 포함) | `--test` |
| QA-3 | Markdown 리포트 파일이 지정 경로에 생성되는가 | `--test` + `--no-discord` |
| QA-4 | Discord 비활성 상태에서도 파이프라인이 오류 없이 완료되는가 | `--test` + `--no-discord` |
| QA-5 | Fallback 리포트가 "AI 분석 실패 / 기본 리포트 생성" 문구와 함께 생성되는가 | `--test` + `--force-fallback` |
| QA-6 | `--from` / `--to` 지정 시 해당 범위의 catch-up 수집 범위가 리포트에 올바르게 명시되는가 | `--test` + `--from` + `--to` |
