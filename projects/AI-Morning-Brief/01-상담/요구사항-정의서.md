# 요구사항 정의서 — AI-Morning-Brief

- 작성일: 2026-06-21
- 고객/요청자: IT상상공방 운영자 (내부 도구)

---

## 1. 무엇을 만드는가

매일 새벽 자동으로 AI 업계 최신 소식을 수집·요약·DB 저장하고, Markdown 일일 리포트를 Discord Webhook으로 전송하는 **개인용 AI 동향 추적 자동화 도구**.
단발성 뉴스 알림이 아니라, 수집 데이터가 매일 쌓여 "지난 N일 간의 AI 흐름"을 나중에 분석할 수 있는 **개인 정보 자산**을 만드는 것이 핵심 목적이다.

---

## 2. 누가 쓰는가 (대상 사용자)

- **주 사용자**: IT상상공방 운영자 1인 (개인용 내부 도구)
- **사용 상황**: 매일 아침, 새벽에 자동 수집된 리포트를 Discord에서 확인한다. 별도 UI를 조작하지 않고 Discord 채널을 열면 이미 리포트가 와 있는 상태를 기대한다.
- **고객용 서비스 아님**: 외부 사용자, 회원가입, 다중 계정은 고려하지 않는다.

---

## 3. 왜 필요한가 (문제 / 목적)

**현재 문제**:
- AI 업계 변화 속도가 빨라 매일 주요 소식을 직접 검색하고 확인하는 데 시간이 소요된다.
- 수집한 정보가 흩어져 있어 "지난달 Claude Code 관련 흐름이 어땠는지" 같은 후향적 분석이 불가능하다.
- 상상공방에 적용 가능한 AI 도구와 사례를 놓치는 경우가 생긴다.

**만들려는 목적**:
- AI 트렌드를 매일 자동으로 수집·정리해 아침에 한 번만 읽으면 되는 구조를 만든다.
- 수집 데이터를 DB에 누적해 장기 트렌드 분석이 가능한 개인 정보 자산을 확보한다.
- 상상공방 v2 전략 수립에 실시간으로 활용 가능한 "내부 레이더" 역할을 한다.

---

## 4. 성공 기준

1. **자동 실행 및 Discord 전송**: 매일 새벽 3시~6시 사이에 사람 개입 없이 자동 실행되어, 정해진 Discord 채널에 일일 리포트가 도착한다.
2. **데이터 누적**: 수집된 기사가 URL 기준 중복 없이 SQLite DB에 쌓이며, 수집 후 1개월이 지나도 과거 데이터가 조회 가능하다.
3. **실패 복구(catch-up)**: 하루 실행이 실패해도, 다음 실행 시 마지막 성공 시각 이후의 소식을 자동으로 몰아서 수집하고 리포트에 "수집 범위"를 명시한다.

---

## 5. 핵심 요구사항 (MVP에서 반드시 동작해야 하는 것)

### 5-1. 수집
- [ ] 설정 파일로 뉴스 소스(RSS, URL 등)를 관리하며, 소스 추가·제거가 설정 파일 수정만으로 가능하다.
- [ ] 수집 대상 관심 분야: OpenAI/ChatGPT/GPT/Codex, Anthropic/Claude/Claude Code, Google Gemini, Cursor, MCP, AI Agent, n8n/자동화 워크플로우, GitHub Actions 기반 AI 개발 자동화, 바이브코딩/AI 개발 프로세스, 상상공방에 적용 가능한 AI 도구·사례.
- [ ] URL 기준으로 중복 수집을 방지한다.
- [ ] 수집한 원본 데이터를 raw JSON 형태로 보관한다.

### 5-2. 저장 (DB)
- [ ] SQLite DB에 기사 메타데이터(URL, 제목, 발행일, 출처), 요약, 태그, 중요도, 상상공방 관련도를 저장한다.
- [ ] 마지막 성공 실행 시각과 마지막 리포트의 수집 범위(시작~끝 시각)를 기록한다.

### 5-3. 요약
- [ ] MVP에서는 Claude API 연동을 고려하되, 연동이 어려운 경우 mock summary 또는 단순 텍스트 요약으로 대체 가능하다.

### 5-4. 리포트 생성
- [ ] 매일 Markdown 형식의 일일 리포트를 생성한다.
- [ ] 리포트 필수 포함 항목:
  - 리포트 날짜
  - 수집 범위 (시작 시각 ~ 종료 시각)
  - 확인한 기사 발행일 범위
  - 신규 수집 건수 / 중요 기사 수
  - 상상공방 적용 가능 소식
- [ ] 리포트 섹션 구성:
  - 오늘의 핵심 변화
  - Claude / Claude Code 관련 소식
  - OpenAI / Codex / GPT 관련 소식
  - MCP / Agent / 자동화 관련 소식
  - 상상공방에 적용할 수 있는 아이디어
  - 나중에 실험해볼 만한 액션 아이템
  - 저장해둘 만한 원문 링크

### 5-5. Discord 전송
- [ ] 생성된 리포트를 Discord Webhook URL로 전송한다.
- [ ] Webhook URL은 설정 파일 또는 환경 변수로 관리한다 (하드코딩 금지).

### 5-6. 실패 복구 (catch-up)
- [ ] 하루 정도 실행에 실패해도 다음 실행 때 마지막 성공 시각 이후의 소식을 몰아서 수집·리포트한다.
- [ ] catch-up 실행 시 리포트에 "수집 범위"와 "보완 수집임을 명시"한다.

## 6. 제약 / 전제

| 항목 | 내용 |
|------|------|
| 구현 언어 | Python |
| 데이터 저장 | SQLite (단일 파일 DB) |
| 리포트 형식 | Markdown |
| 알림 채널 | Discord Webhook (채널 수: 1개) |
| 사용자 수 | 1인 (개인용) |
| UI | 없음 — Discord 수신 + 로컬 파일 확인만 |
| 실행 방식 | 미확정 (설계 단계에서 결정 — cron, GitHub Actions, 기타) |
| 요약 방식 | MVP: mock summary 또는 단순 요약 허용 / 추후 Claude API 연동 고려 |

## 8. 미확정 사항 (설계 단계에서 결정할 질문)

| 번호 | 질문 | 이유 |
|------|------|------|
| Q1 | 자동 실행 환경은 어디인가? (로컬 Mac cron, 서버, Docker, GitHub Actions 등) | 새벽 3~6시 자동 실행 조건에 맞는 실행 환경이 설계 단계에서 결정되어야 한다. |
| Q2 | MVP에서 지원할 뉴스 소스의 구체적 목록과 수집 방식은? (RSS URL, API, 스크래핑 대상 등) | "여러 뉴스 소스, 블로그, RSS, GitHub, 커뮤니티 등"이 언급됐으나, 각 소스의 접근 방식이 다르다. |
| Q3 | Discord 메시지가 2000자 제한을 초과할 경우 어떻게 처리할 것인가? (분할 전송, 파일 첨부, 요약 축약 등) | Discord Webhook 단일 메시지 제한이 있어 설계 단계에서 전략을 정해야 한다. |
| Q4 | 설정 파일 형식은 무엇으로 할 것인가? (YAML, JSON, TOML 등) | 소스 관리·운영 편의성과 직결된다. |
| Q5 | SQLite DB 파일 및 raw JSON, 일일 리포트 파일의 저장 위치와 보관 기간 정책은? | 장기 누적을 전제로 하므로 디렉토리 구조와 파일 네이밍 규칙이 필요하다. |
