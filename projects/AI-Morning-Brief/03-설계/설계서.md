# 설계서 — AI-Morning-Brief (V0.1)

- 작성일: 2026-06-21
- 근거 문서: `02-기획/PRD.md`
- 형태: **자동화 프로세스 (Python CLI 파이프라인 + 로컬 스케줄러)**
- 목표: "검증 가능한 자동화 파이프라인" — 수집 → SQLite 저장 → Claude 분석/요약 → Markdown 리포트 → Discord 전송, 단계별 독립 검증 가능.

## 0. 핵심 설계 결정 요약 (TL;DR)

| 결정 항목 | 선택 | 한 줄 근거 |
|-----------|------|-----------|
| 결과물 형태 | 자동화 프로세스 (헤드리스 CLI) | UI 없이 Discord 수신 + 로컬 파일 확인이 요구사항. 가장 빠르게 검증 가능 |
| 언어 | Python 3.11+ | PRD 명시. RSS/SQLite/HTTP 생태계 성숙 |
| 설정 파일 형식 | YAML | 소스 목록 가독성·주석 가능, 운영자가 직접 추가/제거 쉬움 |
| 설정 분리 | `config.yaml`(일반설정) + `sources.yaml`(소스목록) + `sources.example.yaml`(예시) | 소스 관리와 일반 설정 분리. 실제 URL은 운영자가 sources.yaml에서 수정 |
| DB | SQLite 단일 파일 (`sqlite3` 표준 라이브러리) | PRD 명시. 개인용 단일 사용자, 누적 저장에 충분 |
| RSS 파싱 | `feedparser` | RSS/Atom 양쪽을 관용적으로 파싱, 사실상 표준 |
| HTTP/Discord | `requests` | Discord Webhook POST + 진단용 HTTP 점검에 단순 |
| **Claude 연동** | **Claude API (`anthropic` Python SDK)** | 무인 cron 실행 안정성, 토큰/비용 명시적 제어, JSON 구조화 출력 파싱 안정. (Claude Code headless는 V0.2 확장) |
| **모델명 정책** | config 기본값 + **잘못된 모델명이면 fallback으로 강등(중단 없음)** | 모델명 오타/미존재로 파이프라인이 죽지 않게. README에 "사용 가능한 Anthropic 모델명으로 수정" 안내 |
| **스케줄러** | **launchd** (macOS 네이티브) | Mac 슬립 중 놓친 작업을 깨어날 때 실행 → catch-up 정책과 자연스럽게 맞물림. (cron은 슬립 시 그냥 건너뜀) |
| operator_profile | **Claude 분석 프롬프트의 필수 입력** | 기사 중요도·상상공방 관련도·액션 판단의 "렌즈". 없으면 경고 + 내장 기본 프로필 fallback |
| 분석 실패 정책 | raw JSON·DB 저장은 항상 선행 완료 → fallback 요약으로 파이프라인 지속 | 데이터 유실 0, 다음 실행에서 재분석 |
| **`--dry-run` 무상태** | DB·raw JSON·report·Discord **어떤 것도 변경하지 않음** | 계획/미리보기 전용. 운영 데이터 무해 보장 |
| **소스 접근성 진단** | **`python main.py --check-sources`** (파이프라인 본체와 분리) | 웹 접근 제약 환경에서 실제 RSS URL 검증을 별도 명령으로 분리. 본 파이프라인은 `--test` fixtures로 외부 의존 없이 QA 통과 |

## 1. 최종 V0.1 아키텍처

```
                         ┌─────────────────────────────────────────┐
                         │  main.py (CLI 진입점 / 옵션 파싱)         │
                         │  --test --dry-run --no-discord            │
                         │  --force-fallback --from --to             │
                         │  --check-sources  (진단 분기, 본체와 분리) │
                         └───────────────┬───────────────────────────┘
                                         │ args
                          ┌──────────────┴───────────────┐
              --check-sources?                         (정상 실행)
                    │ yes                                 │ no
                    ▼                                     ▼
        collector.check_sources()          ┌─────────────────────────────────┐
        → 진단표 출력 → 종료(파이프라인 X)  │  pipeline.run(args) (오케스트레이션)│
                                           └─┬───┬───┬───┬───┬───┬────────────┘
       config_loader                         │   │   │   │   │   │
       ┌──────────────────────────────────────┘   │   │   │   │
       │ config.yaml / sources.yaml                 │   │   │   │
       │ operator_profile.md ◄──────────────────────┼───┤   │   │ (analyzer 핵심 입력)
       │ .env (secrets)                              │   │   │   │
       ▼                                             ▼   ▼   ▼   ▼
  [1 수집]      [2 저장]    [3 분석]   [4 리포트]   [5 전송]
  collector ─► storage ─► analyzer ─► reporter ─► notifier
   │  │          │ (SQLite) │            │           │
   │  └─raw JSON │          │            │           │
   ▼            ▼          ▼            ▼           ▼
 data/raw/  morning_brief.db  (Claude    reports/   Discord
                              or fallback) <date>/    Webhook

흐름 불변식:
  - 수집/raw JSON/DB 저장은 분석(Claude) 성공 여부와 무관하게 완료된다. (단, --dry-run은 예외: 아무것도 안 씀)
  - 단일 소스 수집 실패는 건너뛰고 성공 소스만으로 리포트를 계속 생성한다.
  - Claude 실패(모델명 오류 포함) → fallback 요약으로 리포트·전송 계속.
  - Discord 실패 → 로그만, 파이프라인은 DB/파일 기준으로 성공 판정.
  - 성공 시에만 meta.last_success_at 갱신 (catch-up 기준).
```

각 모듈은 **순수 함수 중심**으로 만들어 단독 테스트가 가능하고, 상태 변경(DB 쓰기/파일 저장/전송)은 `pipeline.py`가 조율한다. `--dry-run`은 **수집 직후 계획·미리보기만 출력하고 어떤 상태도 변경하지 않는다(무상태)** — raw JSON 저장조차 하지 않는다. `--check-sources`는 파이프라인을 타지 않고 외부 RSS 접근성만 점검하고 종료한다.

## 3. DB 스키마

SQLite 단일 파일 `data/morning_brief.db`. 테이블 2개 (+ 인덱스).

```sql
-- 기사 테이블: URL이 PK라 중복 수집은 INSERT OR IGNORE로 자동 차단
CREATE TABLE IF NOT EXISTS articles (
    url           TEXT PRIMARY KEY,      -- 중복 방지 키
    title         TEXT NOT NULL,
    published_at  TEXT,                  -- ISO8601, 파싱 실패 시 NULL 허용
    source        TEXT NOT NULL,         -- sources.yaml의 소스 name
    collected_at  TEXT NOT NULL,         -- ISO8601, 수집 시각
    raw_excerpt   TEXT,                  -- 원문 요약/본문 일부 (fallback 요약 재료)
    summary       TEXT,                  -- Claude 또는 fallback 요약 (분석 후 채움)
    tags          TEXT,                  -- JSON 배열 문자열 예: ["claude","mcp"]
    importance    INTEGER DEFAULT 0,     -- 0~5 중요도 (미분석 0)
    relevance     INTEGER DEFAULT 0,     -- 0~5 상상공방 관련도 (미분석 0)
    analyzed      INTEGER DEFAULT 0      -- 1=Claude 분석 성공, 0=미분석/fallback
);
CREATE INDEX IF NOT EXISTS idx_articles_published ON articles(published_at);
CREATE INDEX IF NOT EXISTS idx_articles_collected ON articles(collected_at);

-- 메타 key-value: catch-up 기준 시각 등 파이프라인 상태
CREATE TABLE IF NOT EXISTS meta (
    key   TEXT PRIMARY KEY,
    value TEXT
);
-- 사용 키:
--   last_success_at         : 마지막 성공 실행 완료 시각 (catch-up 기준, ISO8601)
--   last_report_range_start : 마지막 리포트 수집 범위 시작
--   last_report_range_end   : 마지막 리포트 수집 범위 끝
```

**중복 제거**: `INSERT OR IGNORE INTO articles(url,...) VALUES(...)` 로 URL 충돌 시 무시. `insert_article`은 `cursor.rowcount`로 신규 삽입 여부(True/False)를 돌려준다.

**기사 dict 표준 형태** (모듈 간 계약):
```python
{
  "url": str, "title": str, "published_at": str|None,
  "source": str, "collected_at": str, "raw_excerpt": str
}
```
분석 결과는 별도 dict로 합쳐 DB에 update: `{"url","summary","tags"(list),"importance","relevance","analyzed"}`.

## 5. Claude 분석 프롬프트 구조

Claude API 단일 호출로 (1) 기사별 메타 분석 + (2) 전체 브리핑을 **하나의 JSON**으로 받는다. 파싱 안정성을 위해 JSON 출력을 강제한다.

### 5-1. 프롬프트 조립 (`analyzer._build_prompt`)

```
[system]
당신은 IT상상공방 운영자의 'AI 동향 분석가'다. 아래 <운영자 프로필>을 분석의 렌즈로
삼아 각 기사의 중요도와 상상공방 적용 가능성을 평가하라.
반드시 지정된 JSON 스키마만 출력하라. 그 외 텍스트 금지.

<운영자 프로필>
{operator_profile.md 전체 내용}   ← 핵심 입력 (섹션 6)
</운영자 프로필>

[user]
다음은 오늘 수집된 기사 목록이다. 각 기사를 평가하고 전체 브리핑을 작성하라.

<기사 목록 (JSON)>
[
  {"idx":0,"title":"...","source":"...","published_at":"...","url":"...","excerpt":"..."},
  ...
]
</기사 목록>

요구 출력 스키마:
{
  "articles": [
    {"url": "<원본 url 그대로>", "summary": "<한국어 2~3문장>",
     "tags": ["키워드", ...], "importance": <0-5 정수>, "relevance": <0-5 정수>}
  ],
  "briefing": {
    "headline_changes": ["오늘의 핵심 변화 3~5개"],
    "sangsang_ideas": ["상상공방 적용 아이디어"],
    "action_items": ["실험해볼 액션 후보"],
    "summary_text": "<Discord용 핵심 브리핑, 1800자 이내 한국어>"
  }
}
```

- `importance` = 업계 영향도, `relevance` = 상상공방(운영자 프로필 기준) 적용 가능성.
- `summary_text`는 Discord 본문 재료(notifier가 메타 정보를 덧붙여 2000자 내로 마감).
- 모델/토큰 한도는 `config.yaml: analysis.*`에서 조정. 모델명이 잘못되면 호출이 실패하지만 fallback로 내려가 파이프라인은 계속된다(섹션 8).

### 5-2. 응답 파싱

- `anthropic` SDK 응답에서 텍스트 추출 → `json.loads`.
- 코드펜스(```json … ```)가 섞여 와도 견디도록 첫 `{` ~ 마지막 `}` 슬라이스 후 파싱(방어적).
- 파싱 실패/누락 필드 → **fallback로 강등**(섹션 8). 부분 누락 필드는 기본값(importance/relevance=0, summary=raw_excerpt 일부)으로 보정.

## 6. operator_profile.md 사용 방식 ★

`config/operator_profile.md`는 **선택 참고문서가 아니라 Claude 분석의 핵심 입력 문서**다.

1. **로드**: `config_loader.load_operator_profile(path)`가 파일 전체를 문자열로 읽어 `(text, is_default)`를 반환.
2. **주입**: `analyzer.analyze(...)`가 이 문자열을 `_build_prompt`의 `<운영자 프로필>` 블록에 그대로 넣는다. Claude는 이 프로필을 기준으로 importance/relevance/액션을 판단한다.
3. **부재 정책 (명시적 규칙)**:
   - 파일이 없거나 비어 있으면 → **WARNING 로그** 출력 + 모듈 내장 `DEFAULT_PROFILE` 상수를 사용해 파이프라인을 **중단 없이 계속**한다.
   - 이때 리포트 상단과 Discord 메시지에 `"⚠ 운영자 프로필 없음 — 기본 프로필로 분석"`을 명시한다(운영자가 품질 저하를 인지하도록).
   - 무인 실행 안정성을 위해 "기본 프로필 fallback"을 채택한다(오류 중단 대신). 단 fallback 사용 사실은 항상 가시화한다.

**`config/operator_profile.md` 기본 내용(템플릿)** — FEAT-01에서 생성:

## 7. 실행 플로우

`pipeline.run(args)` 표준 순서 (정상/catch-up 공통):

```
0. --check-sources 분기: 지정 시 collector.check_sources()만 실행·출력하고 종료(섹션 9-1)
1. 설정 로드: load_config()(+경로 env override), load_secrets(), load_operator_profile(), load_sources()
2. 수집 범위 결정 (catch-up):
     start = args.from  > meta.last_success_at  > (now - default_lookback_hours)
     end   = args.to    > now
     catchup_flag = (meta.last_success_at가 default_lookback보다 오래됨) or (--from/--to 지정)
3. 수집: collector.collect(sources, start, end, raw_dir, timeout, test_fixtures=..., save_raw=NOT dry_run)
     - enabled=true 소스만, 소스별 RSS fetch (또는 --test면 fixtures 로드)
     - published_at이 [start,end]에 드는 항목만 채택
     - raw JSON 저장은 save_raw=True일 때만 (──dry-run이면 save_raw=False라 raw도 안 씀──)
     - 반환: (articles, source_errors)
   ── [--dry-run이면 여기서 계획+미리보기만 출력 후 종료. DB·raw JSON·report·Discord 변경 0 (무상태)] ──
4. 저장: storage.insert_article() 루프 → 신규만 DB 삽입 (중복 skip), new_count 집계
5. 분석: analyzer.analyze(new_articles, profile, api_key, model, ..., force_fallback=args.force_fallback)
     - 성공: mode="claude", 기사별 summary/tags/importance/relevance + briefing
     - 실패/강제/키없음/모델명오류: mode="fallback" (섹션 8)
     - storage.update_analysis()로 DB 갱신
6. 리포트: reporter.build_report(...) → reporter.save_report() → reports/<date>/report.md
     - mode, catchup_flag, 수집범위, failed_sources, new_count 반영
7. 전송: args.no_discord가 아니면 notifier.send_discord(webhook, message)
     - webhook 없음/실패 → 경고 로그만, 파이프라인 성공 유지
8. 성공 마감: meta.last_success_at=now, last_report_range_*=(start,end) 갱신
9. 종료코드 0
```

### CLI 옵션 매핑

| 옵션 | 효과 |
|------|------|
| `--test` | 3단계 수집을 `tests/fixtures/sample_articles.json` 로드로 대체 (RSS 미사용) |
| `--dry-run` | 수집 후 계획·미리보기만 출력. **DB·raw JSON·report·Discord 어떤 것도 변경하지 않음(무상태)** |
| `--no-discord` | 7단계 전송 skip (나머지 정상) |
| `--force-fallback` | 5단계 Claude 호출을 강제 실패 처리 → fallback 경로 |
| `--from <ISO>` / `--to <ISO>` | 2단계 수집 범위 수동 지정 (catch-up 재현) |
| `--check-sources` | 파이프라인을 타지 않고 소스 접근성 진단만 실행·출력 후 종료 (섹션 9-1) |

## 8. 실패 복구 플로우

| 실패 유형 | 처리 | 데이터 영향 |
|-----------|------|-------------|
| 단일 소스 수집 오류(타임아웃/파싱오류/HTTP오류) | 해당 소스 skip, `source_errors`에 기록, 나머지 진행. 리포트·Discord에 실패 소스 목록 명시 | 다른 소스 정상 저장 |
| Claude 연동 실패(API 오류/타임아웃/키 없음/**모델명 오류·미존재 모델**/JSON 파싱 실패) or `--force-fallback` | `_fallback_analyze()`로 강등: summary=raw_excerpt 앞 N문장, tags=제목 키워드 추출, importance/relevance=0, briefing=규칙기반 요약. 리포트 상단 `"AI 분석 실패 / 기본 리포트 생성"` 명시 | **raw JSON·DB 저장은 이미 완료(분석 단계 이전)** → 유실 0 |
| Discord 전송 실패(webhook 없음/HTTP 오류) | 경고 로그만. 파이프라인 성공 판정은 DB/파일 기준 | 리포트 로컬 저장 완료 유지 |
| DB 쓰기 오류 | 오류 로그 + 실행 중단(종료코드 1). `last_success_at` **미갱신** → 다음 실행이 catch-up | 부분 저장분은 유지, 미완으로 간주 |
| 전체 실행 실패(슬립/크래시) | `last_success_at` 미갱신 상태 → 다음 실행이 그 시각부터 catch-up 수집. 리포트에 보완 수집 + 범위 명시 | — |

**불변식**: raw JSON 저장과 DB 삽입은 Claude 분석 *이전*에 끝난다(단 `--dry-run`은 둘 다 생략). 따라서 Claude/Discord가 죽어도 데이터는 남는다.

**fallback 요약 상세** (`_fallback_analyze`):
- summary: `raw_excerpt`에서 첫 2문장(없으면 제목).
- tags: 제목·excerpt에서 사전 키워드(claude, openai, gpt, mcp, agent, gemini, cursor, automation…) 매칭.
- briefing.summary_text: "기본 리포트" 헤더 + 신규 건수 + 중요 키워드 매칭 상위 제목 나열.

## 9. 테스트/검수 플로우

### 9-1. 소스 접근성 진단 (`--check-sources`)

외부 RSS 검증을 **파이프라인 본체에서 분리한 별도 명령**으로 둔다. 자동 QA는 외부 네트워크에 의존하지 않고 `--test` fixtures로 통과하며, 실제 URL 접근성은 운영자가 이 명령으로 별도 확인한다.

- 명령: `python main.py --check-sources`
- 동작: `sources.yaml`(없으면 `sources.example.yaml`)의 각 소스를 enabled 여부와 무관하게 점검(enabled=false는 표시만), 결과를 표로 콘솔 출력. 종료코드는 모두 성공이면 0, 하나라도 실패면 1.
- 진단 결과 항목(소스당 1행):

| 필드 | 의미 |
|------|------|
| `source_name` | 소스 이름 |
| `url` | 점검 대상 URL |
| `enabled` | sources.yaml에서 켜짐 여부 |
| `http_status` | HTTP 응답 코드(예 200/404), 연결 실패 시 None |
| `ok` | 응답 성공 여부(2xx + 파싱 가능) |
| `timeout` | 타임아웃 발생 여부(bool) |
| `parseable` | feedparser로 RSS/Atom 파싱 가능 여부(엔트리 ≥1) |
| `entry_count` | 파싱된 엔트리 수 |
| `error` | 에러 메시지(없으면 빈 문자열) |
| `checked_at` | 점검 시각 ISO8601 |

출력 예:
```
SOURCE                 ENABLED  HTTP  OK  TIMEOUT PARSE ENTRIES  ERROR
Hugging Face Blog      true     200   ✓   no      yes   30
GitHub Blog            true     200   ✓   no      yes   20
Google Research        true     404   ✗   no      no    0       HTTP 404
...
요약: 성공 6 / 실패 1  → 실패 소스: [Google Research]
```

> 진단은 **읽기 전용**이다(DB/리포트/Discord에 아무것도 쓰지 않음). 점검 결과로 운영자가 `sources.yaml`을 직접 수정한다. 진단 자체는 파이프라인을 중단시키지 않으며, 진단을 돌리지 않아도 정상 실행에서는 부분 실패 정책으로 죽은 소스를 건너뛴다.

### 9-2. 테스트 데이터와 운영 데이터 분리 ★

**pytest는 운영용 `data/morning_brief.db`·`data/raw/`·`reports/`를 절대 건드리지 않는다.** 격리 메커니즘:

- `test_pipeline.py`는 각 테스트 실행 시 환경변수를 세팅해 운영 경로를 덮어쓴다:
  - `MORNINGBRIEF_DB=data/test_morning_brief.db`
  - `MORNINGBRIEF_RAW_DIR=data/raw/test`
  - `MORNINGBRIEF_REPORTS_DIR=reports/test`
- `config_loader.load_config()`가 이 env들을 읽어 `paths.*`를 override한다(섹션 2 참조, FEAT-01 구현).
- 테스트는 `subprocess`로 `main.py`를 호출하며 위 env를 `os.environ` 복사본에 주입한다.
- 리포트 검증 경로도 `reports/test/<date>/report.md`를 본다.
- 테스트 종료 후 `data/test_*`·`reports/test/`는 정리하거나 무시(gitignore). 운영 누적 데이터는 보존된다.

### 9-3. V0.1 성공 기준 = 아래 4개 명령 전부 통과 ★ (reviewer 필수 검증)

자동 QA(6단계) reviewer는 아래 4개 명령을 그대로 실행해 전부 통과해야 V0.1을 완성으로 인정한다.

```bash

#       cp config/sources.example.yaml config/sources.yaml   # 최초 1회
```

### C. 트리거 — 시간/날짜 + meta 커서 (확정)
- 별도 launchd 잡 `com.itsangsang.morningbrief.weekly`, **일요일 04:40**.
- 대상 주차 = `target_iso_week(now) = (now-2일).isocalendar()`의 (ISO연도, 주차). 일요일 정시 및 +1/+2일 지연 모두 같은 주(월~일)를 가리켜 **자가치유**(Mac sleep 후 wake 보완 실행 대응).
- 멱등: `meta.last_weekly_iso_week`(예 `"2026-W26"`)와 같으면 skip, 다르면 생성 후 갱신. `--week YYYY-W##`는 강제 재생성(커서 일치여도 생성).
- **왜 카운터가 아니라 커서인가**: 카운터(일간 7회==7)는 Mac sleep 미실행·전소스 실패 빈 리포트일·`--from` 재실행에서 일요일 정렬이 깨지고 자가치유 불가 → 기각. 시간/날짜+커서는 늦게 깨어나도 같은 주를 가리키며 커서가 중복을 막는다.
- 커서는 **새 컬럼이 아니라 기존 `meta` 테이블 key-value row**(`last_weekly_iso_week`)로 둔다(스키마 불변).

### D. 정량 신호 레이어 (v2 핵심 — 집계에서만 나오는 신호)
`weekly.aggregate_week`가 코드로 계산해 Claude 합성의 "근거"로 투입(서술이 숫자에 정합):
1. **강도 추이**: 일자별 (건수 / 고중요≥8 / 평균 중요도), 수집 ≤1건 일은 ⚠️ 데이터 품질 경고(이슈 #13의 빈 리포트일과 연결).
2. **태그 빈도 × 지속성**: (총 등장 / 등장 일수). **5일↑ = 지속 줄기**(구조적), 고빈도·저지속 = **단발 버스트**(노이즈).
3. **소스 신호 품질**: 소스별 (건수 / 평균 중요도). 고볼륨·저신호(평균<3.0) = **노이즈 후보**(#12 노이즈 필터·소스 튜닝 입력).
4. **공방 관련 픽**: `relevance ≥ 7`(글로벌 중요도 정렬이 놓치는 공방 맞춤 신호).
5. **일간 아이디어 7일치 통합**: 일간 "상상공방 적용 아이디어" 섹션을 `.md`에서 추출, 반복 등장(우선)→1회성(하위).
- 상위기사: 제목 정규화 중복제거 후 importance desc, relevance desc 상위 ~35건을 합성에 투입.
- **임계값 스케일 주의**: 확정 스펙은 라이브 importance/relevance **0~10** 스케일을 전제(고중요≥8, 공방 픽 relevance≥7, 노이즈 평균<3.0). 설계서 본문 3장의 "0~5" 표기는 V0.1 초기값이며, 임계값은 `weekly.py` 상단 상수(HIGH_IMPORTANCE 등)로 중앙화해 스케일 변경 시 상수만 조정한다.

| # | 결정 항목(#12 요구) | 선택 | 근거(2~3줄) |
|---|---------------------|------|-------------|
| 1 | 분석 기간 밖 기사(경계·published_at 없을 때) | **published_at 우선 경계 + collected_at 폴백 + 범위 외 분리·건수 명시** | 월간 흐름은 "그 달 발행된 사건"이 기준이라 published_at으로 경계를 확정(수집일은 재수집·지연으로 왜곡). published_at 파싱 실패분은 버리면 유실이라 collected_at 월내면 채택. 범위 외는 삭제 않고 '참고 항목'으로 분리·건수 명시(투명성). |

### M-D. #12-1 기간 경계 (`partition_by_period`)
- `get_articles_by_range`로 collected_at 범위를 넓게 조회한 뒤, published_at으로 월 경계를 재검증한다: 월내 채택 / 월 밖 = out_of_period('범위 외 참고 항목') / published_at NULL·파싱불가 → collected_at 월내면 채택(폴백).
- 전문 말미 "## 12 범위 외 참고 항목"에 out_of_period 제목·published_at을 상한 20건 나열하고, 상단 배지에 "범위외 P건"으로 건수 명시.
- **근거**: 결정1 참조 — 발행일 기준 경계로 흐름의 시점을 확정하되, 파싱 실패 유실을 폴백으로 막고, 제외분을 삭제 않고 분리·건수화해 "분석 대상 N건"의 정의를 투명하게 한다.

### M-E. #12-3 2단계 필터 (`first_stage_filter` + Claude 2차)
- **1차(코드)**: 제목·요약·태그에 AI/개발/에이전트 키워드 화이트리스트 매칭 **또는** importance/relevance≥1이면 통과, 아니면 rejected. `classify_noise(rejected)`가 유형별 건수 집계.
- **2차(Claude)**: 프롬프트가 "1차 통과분을 받았고, 2차로 상상공방 적용 가능성 기준으로 흐름·액션 우선순위를 정하되 삭제하지 말 것"을 명시 → `workshop_actions`·`flow_themes` 랭킹에 반영.
- **근거**: 결정3 참조 — 대량·규칙적 노이즈 제거는 코드가 싸게, 프로필 해석이 필요한 적용성 판단은 AI가. 2차를 삭제가 아닌 정렬로 둬 유실을 막고 단계별 건수를 추적한다.

### M-F. #12-5,6 소스 편중 표시 + 지배 방지 (`source_stats` + `cap_source_bias`)
- **표시**: `source_stats` = 소스별 (건수·점유율(0~1)·평균중요도·노이즈여부). 전문 "## 6 소스 편중도" 표에 점유율%로 렌더.
- **지배 방지 1(대표)**: `cap_source_bias`가 소스당 상한 `cap = max(3, round(len×0.15))`을 적용해 각 소스 상위 cap건만 대표 후보로 남김.
- **지배 방지 2(태그)**: `aggregate_month`가 태그 count를 raw 등장수가 아니라 **고유 (source, day) 조합 수**로 계산 → 한 소스가 하루 도배해도 1. day_span(지속성)은 그대로.
- **근거**: 결정5 참조 — 대표는 상한, 태그는 (소스,일) 정규화의 두 축으로 편중을 억제하고, 편중 표를 병기해 소스 튜닝 근거를 제공한다.

### M-G. #12-4 대표 기사 압축 (`select_top_articles_monthly`)
- 제목 정규화 중복제거 → `cap_source_bias` → (importance, relevance, cross_source_count) desc 상위 `TOP_LIMIT=40`. 각 항목에 교차출처수를 부가해 신뢰도 라벨 근거로 동봉.
- **근거**: 결정4 참조 — 토큰 한도 안에서 30일 흐름을 대표하도록 40건으로 압축, 편중 정규화·교차출처 우선으로 신호를 집중한다.

### M-H. #12-2 신뢰도 라벨 (`synthesize_monthly`)
- 프롬프트가 규칙 명시: 교차출처≥2 또는 공식 릴리스=`[확정]`, 단일 출처·전망=`[추정]`, 미확인·논란=`[주의]`. Claude가 `flow_themes`·`notable_events` 각 항목과 `narrative` 핵심 주장 앞에 접두. fallback은 `_label_by_cross_source`로 교차출처수 기반 기계 부여.
- **근거**: 결정2 참조 — 의미 판단은 AI, 근거(교차출처)는 코드, 형식 검증·카운트는 코드. 무인 fallback에서도 라벨이 비지 않게 규칙 부여.

### M-I. #12-7 노이즈 제외 상단 명시 (`build_monthly_report`)
- 헤더 직후 배지 한 줄: `> 📊 수집 N · 1차통과 M · 대표 K · 범위외 P · 노이즈 Q · 신뢰도 [확정 a/추정 b/주의 c]`. "## 0 분석 기반과 제외 내역"에 노이즈 유형별 건수 표(정치사회/인프라투자/비AI소비자/단순릴리스/홍보성/반도체지정학/윤리논평/기타)와 범위 외 건수.
- 신뢰도 카운트는 synthesis의 flow_themes+notable_events+narrative에서 `[확정]/[추정]/[주의]` 등장 수를 코드가 집계.
- **근거**: 결정6 참조 — 제외 기준·건수를 상단 수치로 못박아 신뢰도를 확보하고, 상세는 본문 표·압축은 다이제스트로 분리한다.
