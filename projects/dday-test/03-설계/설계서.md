## 1. 결과물 형태 결정 ★

- **형태**: 웹 (정적 단일 페이지, 클라이언트 전용)
- **근거**: 요구사항은 "날짜 입력 → D-Day 즉시 표시"가 전부이며 저장·로그인·서버 연산이 일절 없다(PRD 3장, 요구사항정의서 6장). 모든 계산은 브라우저 안에서 끝나므로 백엔드·DB·빌드 파이프라인 없이 정적 HTML 파일 하나로 "가장 빠르게 확인시킬 수 있다". 고객이 명시적으로 '웹페이지'를 요청한 점과도 일치한다. 앱(설치 필요)·자동화(트리거 대상 없음)는 이 요구에 과하거나 부적합하다.

## 2. 기술 스택 결정 ★

| 영역 | 선택 | 근거 |
|------|------|------|
| 프론트엔드 | 순수 HTML + CSS + Vanilla JavaScript (단일 `index.html`) | 기능이 단순(입력 1, 텍스트 출력 1)해 프레임워크·번들러가 불필요하다. 의존성 0으로 즉시 브라우저에서 열어 검증 가능 — MVP 속도 최우선. |
| 날짜 입력 UI | HTML5 `<input type="date">` (브라우저 기본 날짜 선택기) | PRD 6장에서 기본 date picker 사용을 이미 결정. 라이브러리 없이 달력 팝업·형식 검증을 무료로 제공한다. |
| 날짜 계산 | 표준 `Date` API (UTC 자정 정규화) | 외부 라이브러리(dayjs 등) 없이 일수 차이 계산이 충분히 가능. 의존성 추가 불필요. |
| 백엔드 / DB | 없음 | 저장·인증·다중 항목 모두 범위 밖(P2). 서버를 두면 과잉 설계. |
| 빌드 도구 | 없음 (파일을 브라우저로 직접 오픈) | 트랜스파일·번들 대상이 없다. `index.html` 더블클릭으로 동작. |
| 배포 | (선택) 정적 호스팅 — 단일 HTML이므로 어디든 그대로 업로드 가능 | MVP 검증 단계에서는 로컬 오픈으로 충분, 배포는 납품 단계 재량. |

> 과한 스택 금지 원칙 준수 — 정적 단일 파일로 전 기능이 충족되므로 프레임워크·서버·DB를 도입하지 않는다.

## 3. 설계자 확정 결정 (PRD 7장 위임 사항) ★

PRD 7장이 설계로 넘긴 3개 항목을 아래와 같이 확정한다.

| 항목 | 확정 | 근거 |
|------|------|------|
| 날짜 선택기 UI 스타일 | `<input type="date">` 의 **브라우저 기본 달력 팝업** 사용 (인라인 캘린더 직접 구현 안 함) | 인라인 캘린더는 직접 구현 비용이 크다. 기본 팝업은 데스크톱·모바일 모두에서 OS 네이티브 UX를 제공해 MVP에 최적. |
| 타임존(자정 기준 계산) | **클라이언트 로컬 타임존** 기준. 입력 날짜와 오늘 날짜를 모두 각자의 연·월·일만 취해 자정으로 정규화 후 비교 | 사용자는 자기 지역의 "오늘"을 기준으로 며칠 남았는지를 기대한다. 로컬 연/월/일을 그대로 쓰면 타임존 경계 오차 없이 직관과 일치한다. UTC 변환 시 발생하는 ±1일 오류를 피하기 위해 `getFullYear/getMonth/getDate`로 날짜 구성요소만 추출해 계산한다. |
| 결과 갱신 시점 | **날짜/이름 변경 즉시 갱신** (버튼 없음). `input`/`change` 이벤트로 실시간 반영 | PRD 5장 "새로고침 없이 즉시 갱신" 비기능 요구를 직접 충족. 버튼 단계가 줄어 UX가 단순해진다. |

## 4. 파일 / 모듈 구조

코드 산출물은 5단계에서 `05-개발/` 아래에 생성된다. 구조는 단일 파일을 기본으로 하되, 가독성을 위해 CSS/JS를 분리할 수 있다(개발 재량). 권장 구조:

```
05-개발/
├── index.html      # 마크업: 제목, 이벤트명 입력, 날짜 입력, 결과 표시 영역
├── style.css       # 레이아웃·반응형(모바일 한눈에) 스타일
├── script.js       # D-Day 계산 + 즉시 갱신 이벤트 바인딩
└── 빌드노트.md      # 5단계 개발자가 작성 (실행 방법 등)
```

> 최소화 우선이면 `index.html` 한 파일에 `<style>`·`<script>` 인라인도 허용. 단 로직 가독성을 위해 분리를 권장한다.

### 핵심 로직 모듈 (script.js 내 함수 단위)
- `calcDday(targetDate, today)` — 두 날짜의 로컬 연·월·일을 자정 정규화해 일수 차 반환(정수).
- `formatLabel(diff, eventName)` — diff와 이벤트명을 받아 표시 문자열 생성:
  - diff > 0 → `"D-{diff}"`
  - diff === 0 → `"D-0"`
  - diff < 0 → `"D+{|diff|}"`
  - eventName 있으면 `"{eventName} {위 라벨}"`, 없으면 라벨만.
- `render()` — 입력값을 읽어 위 두 함수를 호출하고 결과 영역에 출력. 날짜 미입력 시 안내 문구.

## 5. 화면 / 플로우 스케치

단일 페이지, 세로 1열 구성 (모바일 한눈 대응):

```
┌─────────────────────────────┐
│        D-Day 카운터          │   ← 제목
│                             │
│  이벤트 이름 (선택)          │
│  [ 토익시험            ]    │   ← text input (선택)
│                             │
│  목표 날짜                   │
│  [ 2026-07-05      📅 ]     │   ← input type=date (달력 팝업)
│                             │
│  ┌───────────────────────┐  │
│  │     토익시험 D-21      │  │   ← 결과 표시 (즉시 갱신)
│  └───────────────────────┘  │
└─────────────────────────────┘
```

**플로우**
1. 페이지 접속 → 결과 영역은 "날짜를 선택하세요" 안내 표시.
2. 사용자가 날짜 선택(또는 이벤트명 입력) → `input`/`change` 이벤트 발생.
3. `render()` 실행 → `calcDday` → `formatLabel` → 결과 영역 텍스트 갱신(새로고침 없음).
4. 케이스별 출력: 미래 `D-21` / 당일 `D-0` / 과거 `D+4`, 이벤트명 입력 시 접두로 붙음.

## 6. 데이터 / 연동

- 외부 API·서버·DB·로컬스토리지 없음. 모든 상태는 입력 필드의 현재 값(in-memory)뿐.
- 입력 데이터: 이벤트명(string, 선택), 목표 날짜(`YYYY-MM-DD` string).
- 출력 데이터: 표시 문자열 1개.
