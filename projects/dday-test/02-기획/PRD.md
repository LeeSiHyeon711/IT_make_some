# 제품 요구사항 명세 (PRD) — dday-test

- 작성일: 2026-06-14
- 근거 문서: `01-상담/요구사항-정의서.md`

---

## 1. 개요

목표 날짜(연·월·일)를 입력하면 오늘 기준으로 남은 일수 또는 지난 일수를 D-숫자 형식으로 즉시 보여주는 단일 D-Day 카운터다. 고객 본인(개인 용)이 시험일·마감일 같은 특정 날짜를 매번 직접 계산하는 번거로움 없이, 입력 한 번으로 즉시 확인할 수 있도록 한다. 저장·로그인·다중 항목 관리는 이번 범위 밖이다.

---

## 2. 기능 목록 + 우선순위

| 우선순위 | 기능 | 설명 |
|----------|------|------|
| P0 (필수) | 날짜 입력 | 사용자가 목표 날짜(연·월·일)를 입력한다 |
| P0 (필수) | D-Day 계산 | 입력 날짜와 오늘 날짜를 비교해 일수 차이를 계산한다 |
| P0 (필수) | 결과 표시 | 미래는 D-숫자, 당일은 D-0, 과거는 D+숫자 형식으로 화면에 표시한다 |
| P0 (필수) | 단일 페이지 구성 | 입력과 결과 확인이 한 화면 안에서 완결된다 |
| P1 (있으면 좋음) | 이벤트 이름 표시 | 목표 날짜에 이름(예: 토익시험)을 붙여 "토익시험 D-7" 형태로 함께 표시한다 |
| P2 (나중) | 복수 D-Day 관리 | 여러 날짜를 목록으로 저장·관리한다 |
| P2 (나중) | 브라우저/서버 저장 | 입력한 날짜를 저장해 재방문 시 자동 복원한다 |
| P2 (나중) | 알림 기능 | D-Day가 특정 일수 이하가 되면 알림을 보낸다 |

---

## 3. MVP 스코프 (이번 사이클)

**포함 (P0 전체 + P1)**
- 날짜 선택기를 통한 목표 날짜 입력
- 오늘 날짜 기준 D-Day 계산 및 D-숫자 / D-0 / D+숫자 형식 표시
- 선택적 이벤트 이름 입력 → 입력 시 "이벤트명 D-7" 형태로 표시, 미입력 시 "D-7"만 표시
- 단일 페이지 구성 (입력 + 결과 한 화면)

**제외 (나중 / P2)**
- 복수 D-Day 목록 관리
- 브라우저 또는 서버 저장
- 알림·공유 기능

---

## 4. 핵심 사용자 흐름 (시나리오)

**시나리오 1 — 미래 날짜 확인 (일반 케이스)**
1. 사용자가 화면에 접속한다.
2. 날짜 선택기에서 목표 날짜(예: 2026-07-05)를 선택한다.
3. 이벤트 이름 입력란에 "토익시험"을 입력한다(선택 사항).
4. 화면이 즉시 "토익시험 D-21"을 표시한다.

**시나리오 2 — 당일 확인**
1. 사용자가 오늘 날짜(2026-06-14)를 목표 날짜로 선택한다.
2. 화면이 즉시 "D-0"을 표시한다.

**시나리오 3 — 지난 날짜 확인 (과거 케이스)**
1. 사용자가 이미 지난 날짜(예: 2026-06-10)를 목표 날짜로 선택한다.
2. 화면이 즉시 "D+4"를 표시한다.

---

## 5. 비기능 요구

- 별도 로그인·회원 가입 없이 즉시 접근 가능해야 한다.
- 날짜를 바꿀 때마다 결과가 즉시(새로고침 없이) 갱신되어야 한다.
- 모바일 화면에서도 입력과 결과가 한눈에 보여야 한다(최소 반응형 대응).

## 6. 미확정 사항 기획자 결정 내역

요구사항 정의서 7장의 미확정 사항을 아래와 같이 MVP 기준으로 결정했다. 결정 근거를 함께 남긴다.

| 항목 | 결정 | 근거 |
|------|------|------|
| 날짜 입력 방식 | 브라우저 기본 날짜 선택기(date picker) 사용 | 별도 UI 라이브러리 없이 즉시 구현 가능하고, 직접 텍스트 입력 대비 날짜 형식 오류를 방지해 MVP 속도 및 안정성에 유리하다 |
| 이벤트 이름 표시 | MVP에 포함, 선택적 입력 필드로 추가 | 텍스트 한 줄 추가로 구현 비용이 거의 없으나 사용성이 뚜렷이 향상된다. 입력하지 않아도 기본 동작(D-숫자 표시)에 영향을 주지 않는다 |
| 자정 기준 계산 | 날짜 단위(자정 기준)로만 계산, 시간 무시 | D-Day는 "몇 일 남았는가"를 묻는 것이므로 시간대·시각 차이를 무시하고 날짜만 비교하는 것이 사용자 기대에 부합한다. 시간대 처리 세부 사항은 3단계 설계에서 결정 |

---

## 7. 다음 단계(설계)로 넘기는 결정 사항

- 고객이 '웹페이지' 형태를 명시적으로 요청했음을 참고 정보로 전달한다. 최종 형태 결정은 설계자 몫이다.
- 날짜 선택기의 UI 구체적인 스타일(달력 팝업 vs 인라인 등)은 설계에서 결정한다.
- 자정 기준 날짜 계산 시 클라이언트 시간대(브라우저 로컬 타임존) 사용 여부는 설계에서 결정한다.
- 결과 갱신 시점(날짜 선택 즉시 vs 버튼 클릭 후)은 설계에서 결정한다.
