# PickUpMemo v1 고객 전달문

안녕하세요. PickUpMemo v1 개발이 완료되어 전달드립니다.

---

## 이번에 만든 것

**한 줄 요약**: 배민커넥트 배차 화면의 업체 상호를 안드로이드가 제공하는 방법으로 읽어올 수 있는지 검증하는 로그 수집 앱입니다.

이 앱의 존재 이유는 단 하나입니다. "안드로이드 접근성 API와 알림 데이터로 배민커넥트 배차 화면의 업체 상호(식당명)를 실제로 읽어올 수 있는가?" 라는 질문에 답하는 것입니다.

메모도, 팝업도, 로그인도 없습니다. 그것들은 이 질문에 "예"라는 답이 나온 뒤에 만들 것들입니다. 이번 v1은 그 답을 구하기 위한 도구입니다.

---

## 만들어진 기능 목록

| 기능 | 설명 |
|------|------|
| 접근성 서비스 수집 | 폰 화면에 표시되는 모든 텍스트(업체명·주소·버튼 텍스트 등)를 자동으로 읽어 로그에 기록 |
| 알림 수집 | 시스템 알림(배차 알림 포함)의 제목·내용·앱 이름·발생 시각을 자동으로 기록 |
| 로그 보기 화면 | 수집된 로그를 최신순으로 조회. 앱 이름으로 필터링 가능. 전체 삭제 가능 |
| 로그 내보내기 | 버튼 한 번으로 txt 파일 생성 → 카카오톡·Gmail·Google Drive 등으로 바로 공유 |
| 데모 기능 | "교촌치킨 죽전점" 텍스트가 담긴 가짜 배차 화면 + 테스트 알림 생성으로 수집 동작 즉시 확인 가능 |
| 권한 상태 표시 | 메인 화면에서 접근성·알림 권한 허용 상태를 실시간으로 확인 |

## 검증해야 할 핵심 질문과 성공·실패 판단 방법

이 앱으로 확인해야 할 핵심 질문은 두 가지입니다.

**질문 1. 배민커넥트 배차 화면의 업체 상호가 접근성 로그에 잡히는가?**

확인 방법: 실제 배차가 왔을 때 [로그 내보내기]로 txt 파일을 꺼내, 그 안에 업체명이 아래처럼 들어있는지 찾습니다.

```
Type: Accessibility
Text: (업체 상호명) | desc=... | id=...
```

**질문 2. 배민커넥트 배차 알림의 내용이 알림 로그에 잡히는가?**

확인 방법: 같은 txt 파일에서 배민커넥트 패키지명과 함께 아래 형식의 기록이 있는지 찾습니다.

```
Type: Notification
Title: (배차 알림 제목)
Body: (업체 상호명 포함 내용)
```

### 결과에 따른 다음 행동

| 결과 | 판단 | 다음 행동 |
|------|------|-----------|
| Accessibility 또는 Notification 로그에 업체 상호가 잡힘 | 검증 성공 | v2 개발 착수 (아래 설명) |
| 어느 쪽에도 업체 상호가 안 잡힘 | 검증 실패 | OCR(화면 캡처 인식) 등 대안 방식 검토 |

데모 기능(접근성 테스트 화면 + 테스트 알림)으로 수집 자체가 동작한다는 것은 이미 확인된 상태입니다. 남은 건 "배민커넥트라는 실제 앱이 화면/알림 데이터를 안드로이드 시스템에 충분히 노출하는가"의 확인입니다.

## 다음 단계 (v2) 로드맵

이번 v1 로그를 분석한 결과에서 업체 상호 데이터가 확인되면, 그것을 근거로 아래 v2 개발을 시작합니다.

**PickUpMemo v2 — 업체명 기반 메모 자동 팝업**

배차가 왔을 때 그 업체에 대해 본인이 미리 적어둔 메모(예: "주차 어려움", "문 잠금", "2층 직접 전달")가 화면에 자동으로 떠오르는 기능입니다.

v2에서 새로 만들 것들:
- 업체명별 메모 작성·조회 기능
- 배차 감지 시 메모 자동 팝업
- 메모 데이터 기기 내 저장

v1에서 검증된 수집 방식(접근성 또는 알림)을 그대로 가져다 씁니다. 즉 v1의 로그 파일이 v2의 설계 근거가 됩니다.

## 이번에 일부러 만들지 않은 것들 (그 이유)

| 만들지 않은 것 | 이유 |
|---------------|------|
| 메모 기능 / 팝업 기능 | v2의 핵심 기능. 수집이 되는지 먼저 확인한 뒤 만들어야 의미 있음 |
| 서버 / 클라우드 저장 | 1인 검증 도구에 서버는 불필요한 비용·복잡도. 데이터는 폰 내부에만 보관 |
| 회원가입 / 로그인 | 혼자 쓰는 앱에 계정 관리는 과함 |
| OCR(사진 인식) | 접근성 API 방식이 먼저. 실패했을 때의 대안 |
| 앱 디자인 다듬기 | 수집·내보내기가 동작하는 것이 목표. 디자인은 v2에서 |

이 판단들은 처음 요구사항 정의 때 함께 결정한 내용이며, v2를 빠르게 시작하기 위해 v1을 최대한 가볍게 만들기 위한 선택입니다.

---

## 검증 방법과 현재 상태

- 자동화 테스트(에뮬레이터 기반)는 환경 제약으로 진행하지 않았습니다.
- 실제 안드로이드 기기에서 직접 설치·실행하여 수동 테스트를 완료했습니다.
- 접근성 수집, 알림 수집, 로그 보기, 내보내기, 데모 기능 모두 실기기에서 정상 동작을 확인했습니다.
