- [전체 삭제] → 확인 다이얼로그 [삭제] → 인메모리·파일 로그 모두 삭제, 목록 0건.

**6. 내보내기**
- 메인 [로그 내보내기] → compaction 후 `pickupmemo_log_yyyyMMdd_HHmmss.txt` 생성 + 공유 시트(카카오톡/Gmail/Drive 등). 공유받은 txt를 열어 블록 형식(`[시각]`/`Package:`/`Type:`/`Text:` 또는 `Title:`/`Body:`)이 사람이 읽기 좋은지 확인(0건이어도 빈 txt로 공유 동작).

> 위 단계별 상세 동작/경계 케이스는 아래 "접속 / 확인 경로"에 이슈별로 더 자세히 적혀 있다.

## 접속 / 확인 경로
- 앱 실행 → 메인(허브) 화면 표시. 상단에 상태 3종(접근성 서비스 실행 중/중지, 접근성 권한 허가/미허가, 알림 접근 권한 허가/미허가)이 onResume마다 갱신된다. [접근성 설정 열기]/[알림 접근 설정 열기]로 시스템 설정 이동 → 토글 ON → 뒤로가기 하면 상태가 즉시 "허가/실행 중"으로 바뀌는지 확인(설계서 4-3 시나리오 A). [접근성 테스트 화면]은 placeholder Activity로 이동(G-2 #14). **[로그 보기]는 LogViewActivity로 이동해 수집된 로그를 최신순 RecyclerView로 표시(F-1 #11 완료)** — 로그 0건이면 안내 문구가 뜨고, 권한을 켜고 외부 앱 화면/알림으로 로그가 쌓인 뒤 다시 들어가면(또는 화면 재진입 시) 최신순으로 갱신된다. [테스트 알림 생성]은 후속 이슈(#13) 안내 토스트.
- **패키지명 필터 + 전체 삭제(#12 검증 경로)**: [로그 보기] 진입 → 상단 입력란에 패키지명(예: `com.sample.app` 또는 일부 문자열)을 넣고 [적용]을 누르면 해당 패키지명을 포함하는 로그만 최신순 표시되는지 확인(비우고 [적용]하면 전체로 복귀, 무결과면 "필터('x')에 해당하는 로그가 없습니다" 안내). 필터를 적용한 뒤 앱을 종료했다 다시 [로그 보기]로 들어가면 필터가 빈 값(전체)으로 초기화되는지 확인(PRD 4-3). [전체 삭제] → 확인 다이얼로그에서 [삭제] 선택 시 인메모리·파일 로그가 모두 비워지고 목록이 0건 안내로 바뀌며 "로그를 모두 삭제했습니다" 토스트가 뜨는지 확인(설계서 5-3③).
- **로그 내보내기(#10 검증 경로)**: 접근성/알림 권한을 켜고 외부 앱 화면·알림으로 로그가 쌓인 뒤 메인에서 [로그 내보내기] → compaction 후 `pickupmemo_log_yyyyMMdd_HHmmss.txt`가 생성되고 공유 시트(카카오톡/Gmail/Drive 등)가 뜨는지 확인(설계서 4-4 시나리오 C). 공유받은 txt를 열어 설계서 5-2 형식(블록별 `[시각]`/`Package:`/`Type:`/`Text:` 또는 `Title:`/`Body:`)으로 사람이 읽을 수 있는지 확인. 로그가 0건이어도 빈 txt로 공유 시트는 동작.
- **접근성 테스트 화면(#14 검증 경로)**: 먼저 메인에서 [접근성 설정 열기]로 **접근성 권한을 ON**(PickUpMemo 서비스 토글). 그다음 메인 [접근성 테스트 화면] → 모의 배차 화면("업체 상호 / 교촌치킨 죽전점 / 전달지 주소: 경기 용인시 ... / [배차 수락] [거절]")이 표시되는지 확인(버튼은 데모라 누르면 토스트만). 뒤로가기 후 [로그 보기]로 들어가면 방금 그 화면의 텍스트가 `Type: Accessibility` 블록(예: `Text: 교촌치킨 죽전점 | desc=업체상호 | id=tv_store_name`, Package: com.itmakesome.pickupmemo)으로 수집돼 있는지 확인 — 이게 G-2 데모의 목적(화면 텍스트를 접근성 서비스가 수집하는지 눈으로 확인). 접근성 권한이 꺼져 있으면 화면은 떠도 로그에는 남지 않는다.
- **테스트 알림 + 정책 충돌 해소(#13 검증 경로)**: 먼저 메인에서 [알림 접근 설정 열기]로 **알림 접근 권한을 ON**(이게 켜져야 리스너가 자기 알림을 수집). 그다음 [테스트 알림 생성] → (API 33+ 첫 실행이면 알림 권한 요청 다이얼로그가 뜨고 [허용]) → 상태바에 제목 "신규 배달 요청" / 내용 "교촌치킨 죽전점 / 배달료 3,900원" 알림이 표시되고 "테스트 알림을 생성했습니다" 토스트가 뜨는지 확인. 이어 [로그 보기]로 들어가면 방금 그 테스트 알림이 `Type: Notification` / `Title: 신규 배달 요청` / `Body: 교촌치킨 죽전점 / 배달료 3,900원` 블록(Package: com.itmakesome.pickupmemo)으로 **로그에 수집돼 있는지** 확인 — 이게 #7의 자기 앱 제외 정책을 전용 채널 예외로 해소한 핵심 결과다. (알림 접근 권한이 꺼져 있으면 알림은 떠도 로그에는 남지 않는다 — 권한 ON 필수.) API 33+에서 권한 거부 시 "알림 권한이 없어 테스트 알림을 표시할 수 없습니다" 토스트.

## 알려진 한계 / 미구현 (다음 이슈)

### ★ #15(H-1) 완료 — 전체 개발 종료 + 알려진 한계 (사람 수동 테스트 전 참고)
- **전체 15개 이슈(A~H) 개발 완료.** 남은 개발 이슈 0. 디버그 APK 빌드 최종 확인 완료(`BUILD SUCCESSFUL`, app-debug.apk ≈ 6.5MB). placeholder 전부 실제 구현으로 교체됨.
- **알려진 한계(설계상 의도 — MVP 검증 도구 범위)**:
  - 권한은 코드로 못 켠다(시스템 바인드 서비스). 사용자가 시스템 설정에서 접근성·알림 접근을 직접 ON 해야 수집 시작(설계서 2-3/4-3). 권한 OFF면 화면/알림은 떠도 로그에 안 남는다.
  - 수집 한도 5,000건(인메모리 deque 기준). 초과 시 가장 오래된 것부터 drop(설계서 5-3). 무한 보관 아님.
  - 저장은 앱 내부 txt 파일 1개(외부 노출 없음). DB·서버·클라우드·메모·OCR·로그인 없음(설계서 2-2 배제 확정). 인터넷 권한 미선언이라 외부 전송 불가 — 데이터 반출은 [로그 내보내기] 공유 시트 수동 방식만.
  - 알림 수집 시 자기 앱(PickUpMemo) 알림은 노이즈로 제외하되 **테스트 전용 채널(`pickupmemo_test_delivery`)만 예외 통과**(#13). 그래서 데모 테스트 알림은 수집되지만, 그 외 자기 앱 알림은 안 남는다.
  - 접근성 수집 범위는 전체 앱 대상이라 정상 사용 중 다양한 앱 화면 텍스트가 함께 쌓일 수 있다(검증 단계 의도). 패키지명 필터로 좁혀서 본다.
  - UI는 검증 최소 수준(Material 3 표준 위젯, 디자인 다듬기 최하위 우선순위). 기능 확인용이지 상용 완성도 아님.
  - 기기/OS 제조사별 접근성·알림 설정 경로 명칭이 조금씩 다를 수 있다(설정 화면 이동 Intent는 표준 액션 사용).
