# 설계서 — PickUpMemo (v1 기술 검증 로그 수집기)

- 작성일: 2026-06-15
- 근거 문서: `02-기획/PRD.md`, `01-상담/요구사항-정의서.md`

---

## 1. 결과물 형태 결정 ★

- **형태**: 앱 (Android Native 단일 앱, 디버그 APK)
- **근거**:
  - 검증 대상 데이터(접근성 트리 텍스트 / 시스템 알림)는 **Android OS의 `AccessibilityService`와 `NotificationListenerService`로만 접근 가능**하다. 웹/자동화로는 원천적으로 수집이 불가능하므로 형태 선택의 여지가 없다.
  - "배민커넥트 화면에서 업체 상호를 읽을 수 있는가"를 실제 기기에서 한 번에 확인시키려면, 실 기기에 설치되는 네이티브 앱이 가장 빠르고 유일한 검증 수단이다.
  - PRD/요구사항정의서에서 고객이 플랫폼(Android Native), 언어(Kotlin), UI(Material 3), 최소 버전(API 26)을 이미 명시했다. 설계자는 이를 **실제 빌드 가능한 버전 조합으로 확정**한다.

## 2. 기술 스택 결정 ★

### 2-1. 버전 조합 (실제 빌드 가능 조합으로 확정)

| 영역 | 선택 | 근거 |
|------|------|------|
| 언어 | Kotlin 1.9.24 | 고객 명시. AGP 8.5.x와 검증된 안정 조합. |
| 빌드 도구 | Android Gradle Plugin (AGP) 8.5.2 + Gradle 8.7 | 함께 검증된 안정 페어. 빌드 실패 위험 최소화(최신 RC 회피). |
| JDK | JDK 17 | AGP 8.x 요구사항. Android Studio 번들 JDK 사용. |
| compileSdk / targetSdk | 34 (Android 14) | API 33+ 알림 권한(POST_NOTIFICATIONS) 정상 처리 위해 34 권장. |
| minSdk | 26 (Android 8.0) | 고객 명시. `NotificationListenerService`/`AccessibilityService` 모두 안정 동작. |
| UI 방식 | **XML View 레이아웃 + Material Components (Material 3 테마)** | 화면 3개·버튼 위주의 단순 검증 도구. Compose보다 의존성·빌드 변수 적어 "확인 가능한 최소"에 부합. |
| Material 3 | `com.google.android.material:material:1.12.0` (`Theme.Material3.DayNight.NoActionBar`) | 고객 명시한 Material 3 충족. 표준 위젯만 사용. |
| 기타 androidx | `androidx.core:core-ktx:1.13.1`, `androidx.appcompat:appcompat:1.7.0`, `androidx.constraintlayout:constraintlayout:2.1.4` | FileProvider·AppCompatActivity·레이아웃에 필요한 표준 라이브러리만. |
| 로그 저장 | 앱 내부 저장소 단일 txt 파일 + 인메모리 캐시 (표준 `java.io`) | DB·서버 금지. 5,000건 한도 검증 도구엔 파일이 가장 단순. ORM/Room 불필요. |
| 파일 공유 | `androidx.core.content.FileProvider` + `Intent.ACTION_SEND` | API 24+ 외부 공유 표준. 별도 라이브러리 없음. |
| 비동기 | Kotlin Coroutines (`kotlinx-coroutines-android:1.7.3`) | 파일 IO를 메인 스레드 밖에서 처리. 단순 launch/withContext만. |

> **과한 스택 금지 원칙 적용**: Room/Retrofit/Hilt/Compose/DataBinding 등 **도입하지 않는다.** 표준 SDK + Material 3 + 최소 androidx만 사용한다. 외부 네트워크 라이브러리는 비기능 요구(외부 통신 없음)에 의해 명시적으로 배제한다.

### 2-2. 절대 구현 금지 항목 재확인 (설계에서 배제 확정)

회원가입/로그인, 서버·DB서버·클라우드, AI/OCR, 메모 기능, 팝업 기능 — **본 설계 어디에도 포함하지 않는다.** 인터넷 권한(`INTERNET`)도 선언하지 않는다(공유 Intent는 권한 불필요).

### 2-3. 포그라운드 서비스 사용 여부 결정

- **사용하지 않는다.** `AccessibilityService`와 `NotificationListenerService`는 **시스템이 바인딩·유지하는 서비스**다. 사용자가 OS 설정에서 권한을 켜 둔 동안 시스템이 살려두므로, 별도 포그라운드 서비스/상시 알림이 필요 없다.
- 따라서 `FOREGROUND_SERVICE` 권한과 상시 알림 채널을 만들지 않는다(불필요한 복잡도 제거). PRD 6장 "백그라운드 유지"는 이 두 시스템 서비스의 기본 생명주기로 충족된다.

### 4-2. 데이터 수집 흐름

```
[배민커넥트 등 임의 앱 화면]
        │ (화면 텍스트 변화)
        ▼
ScreenAccessibilityService.onAccessibilityEvent()
        │  packageName, eventType, source 노드 트리 순회
        │  → text / contentDescription / viewIdResourceName 추출
        ▼
LogRepository.add(LogEntry(ACCESSIBILITY, ...))

[시스템 알림 발생]
        │
        ▼
LogNotificationListenerService.onNotificationPosted(sbn)
        │  title, text, packageName, postTime 추출
        ▼
LogRepository.add(LogEntry(NOTIFICATION, ...))

LogRepository.add()
   → 인메모리 deque에 추가(앞이 최신), 5000 초과 시 가장 오래된 것 drop
   → LogFileStore에 1줄 append (코루틴, IO 스레드)
```

### 4-3. 권한 설정 플로우 (시나리오 A)

```
MainActivity 상태 "미허가" 표시
  → [접근성 설정 열기] → Settings.ACTION_ACCESSIBILITY_SETTINGS
  → 사용자가 PickUpMemo 토글 ON → 뒤로가기
  → MainActivity.onResume → PermissionChecker 재판정 → "허가/실행 중"으로 갱신
(알림 접근도 동일: Settings.ACTION_NOTIFICATION_LISTENER_SETTINGS)
```

### 4-4. 내보내기 플로우 (시나리오 C)

```
[로그 내보내기]
  → LogFileStore 현재 로그를 export 파일로 기록
     파일명: pickupmemo_log_yyyyMMdd_HHmmss.txt   (PRD 4-1)
     위치: filesDir/exports/  (FileProvider 노출 경로)
  → FileProvider.getUriForFile() 로 content:// URI 획득
  → Intent.ACTION_SEND (type=text/plain) + FLAG_GRANT_READ_URI_PERMISSION
  → 공유 시트 (카카오톡 / Gmail / Drive 등)
```

## 5. 데이터 / 저장 설계

### 5-1. LogEntry 모델

| 필드 | 타입 | 설명 |
|------|------|------|
| timestampMillis | Long | 수집 시각 (정렬·포맷용) |
| type | LogType | ACCESSIBILITY / NOTIFICATION / SYSTEM |
| packageName | String | 출처 앱 패키지명 (필터 기준) |
| title | String? | 알림 제목 / (접근성은 비움 또는 Activity 등) |
| body | String | 핵심 텍스트 (화면 텍스트·desc·viewId 묶음 또는 알림 내용) |

### 5-2. 로그 텍스트 포맷 (PRD 6장 일관 형식)

각 로그 1건 = 파일에 아래 한 블록(append). 블록 구분은 빈 줄.
```
[2026-06-15 14:33:12]
Package: com.sankuai.meituan.banma   (예시)
Type: Accessibility
Text: 교촌치킨 죽전점 | desc=업체상호 | id=tv_store_name
```
- 알림은 `Type: Notification` / `Title:` / `Body:` 라인 구성.
- 파싱이 아니라 육안 분석이 목적이므로 사람이 읽기 쉬운 고정 형식 유지.

### 5-3. 5,000건 한도 구현 방식 (PRD 4-2) — 단순 설계 확정

- **인메모리 `ArrayDeque<LogEntry>`(최대 5,000)를 "건수 한도의 단일 기준(source of truth)"으로 삼는다.** 추가 시 5,000 초과하면 가장 오래된 항목을 즉시 drop.
- **파일은 인메모리의 영속 미러**다. 접근성 이벤트는 빈번하므로 **매 건 전체 파일 재작성은 금지**(성능). 대신:
  - 추가 시: 파일에 **1블록 append만** (빠름).
  - 압축(compaction)은 제한된 시점에만 전체 재작성: ① 앱 시작 시 로드 후, ② **내보내기 직전**, ③ 전체 삭제 시.
  - 파일이 무한 증가하지 않도록 **히스테리시스**: append 누적 줄 수가 한도의 약 1.2배(예: 6,000블록)를 넘으면 인메모리 deque 기준으로 파일을 재작성해 5,000으로 정리.
- **내보내기 파일은 항상 압축 후 생성** → 내보낸 txt는 5,000건 한도를 정확히 지킨다.
- 앱 재시작 시: 파일을 읽어 마지막 5,000블록만 deque에 적재(앞 초과분 무시).
- 동시성: `LogRepository.add()`는 동기화(`synchronized` 또는 단일 디스패처)로 접근성/알림 두 서비스의 동시 호출을 직렬화한다.

> 대안(인메모리+전건 재작성)은 이벤트 빈도가 높아 성능 위험. append+제한적 compaction이 검증 도구에 가장 단순하면서 안전하다.

### 5-4. 저장 위치

- 내부 로그 파일: `filesDir/pickupmemo_log.txt` (앱 전용, 외부 노출 안 됨)
- 내보내기 임시 파일: `filesDir/exports/pickupmemo_log_<timestamp>.txt` → FileProvider로만 공유

## 6. AndroidManifest / 권한·서비스 등록 설계

### 6-1. 권한 (uses-permission)

| 권한 | 용도 | 비고 |
|------|------|------|
| `android.permission.POST_NOTIFICATIONS` | 테스트 알림 발생 (API 33+ 런타임 요청) | API 33 미만은 선언만으로 동작 |
| (INTERNET) **선언 안 함** | 외부 통신 없음(PRD 6장) 명시적 배제 | — |
| (FOREGROUND_SERVICE) **선언 안 함** | 시스템 바인드 서비스라 불필요 | 2-3 결정 |

> `BIND_ACCESSIBILITY_SERVICE`, `BIND_NOTIFICATION_LISTENER_SERVICE`는 uses-permission이 아니라 **각 서비스 선언의 `android:permission` 속성**으로 시스템에 등록한다(아래).

### 6-2. 서비스 등록

```xml
<!-- 접근성 서비스 -->
<service
    android:name=".service.ScreenAccessibilityService"
    android:exported="true"
    android:permission="android.permission.BIND_ACCESSIBILITY_SERVICE">
    <intent-filter>
        <action android:name="android.accessibilityservice.AccessibilityService" />
    </intent-filter>
    <meta-data
        android:name="android.accessibilityservice"
        android:resource="@xml/accessibility_service_config" />
</service>

<!-- 알림 수신기 서비스 -->
<service
    android:name=".service.LogNotificationListenerService"
    android:exported="true"
    android:permission="android.permission.BIND_NOTIFICATION_LISTENER_SERVICE">
    <intent-filter>
        <action android:name="android.service.notification.NotificationListenerService" />
    </intent-filter>
</service>
```

### 6-3. accessibility_service_config.xml (수집 범위)

- `android:accessibilityEventTypes="typeWindowStateChanged|typeWindowContentChanged|typeViewTextChanged"`
- `android:accessibilityFeedbackType="feedbackGeneric"`
- `android:canRetrieveWindowContent="true"`
- `android:accessibilityFlags="flagReportViewIds|flagIncludeNotImportantViews|flagRetrieveInteractiveWindows"`
- `android:packageNames` **미지정**(모든 앱 대상) — 검증 단계라 배민커넥트 포함 전체 수집.
- `android:notificationTimeout="100"`
- `android:description="@string/accessibility_service_desc"` (설정 화면 설명 문구)

### 6-4. FileProvider

```xml
<provider
    android:name="androidx.core.content.FileProvider"
    android:authorities="${applicationId}.fileprovider"
    android:exported="false"
    android:grantUriPermissions="true">
    <meta-data
        android:name="android.support.FILE_PROVIDER_PATHS"
        android:resource="@xml/file_paths" />
</provider>
```
`file_paths.xml`: `<files-path name="exports" path="exports/" />`

### 6-5. Activity 등록

`MainActivity`(LAUNCHER), `LogViewActivity`, `AccessibilityTestActivity` 3개. 테마 `Theme.Material3.DayNight.NoActionBar`.

| AccessibilityTestActivity | 고정 텍스트(업체 상호·교촌치킨 죽전점·주소·수락·거절) 표시만 |
| ScreenAccessibilityService | 접근성 이벤트→노드 트리 순회→text/desc/viewId/패키지 추출→Repository.add |
| LogNotificationListenerService | onNotificationPosted→title/text/패키지/시각 추출→Repository.add |
| LogRepository (싱글톤) | 인메모리 5000 cap deque 관리 + 파일 동기화 진입점, 조회/삭제 API |
| LogFileStore | 내부 txt 파일 append/compact/read/clear, 내보내기 파일 생성 |
| PermissionChecker | 접근성 활성 여부(Settings.Secure enabled_accessibility_services 파싱), 알림 접근 여부(NotificationManagerCompat 또는 enabled_notification_listeners) |
| LogExporter | 압축→export 파일→FileProvider URI→ACTION_SEND 공유 시트 |
| TestNotifier | 알림 채널 생성 + 런타임 POST_NOTIFICATIONS 처리 + 고정 내용 알림 발생 |
| TimeFormat | `yyyy-MM-dd HH:mm:ss`(로그) / `yyyyMMdd_HHmmss`(파일명) 포맷 |

## 8. 빌드 · 실행 방법 개요

1. **Android Studio**(최신 안정판, 번들 JDK 17)로 프로젝트 루트 `PickUpMemo/` 열기 → Gradle Sync(인터넷으로 의존성 1회 다운로드).
2. **디버그 APK 빌드**: `Build > Build Bundle(s)/APK(s) > Build APK(s)` 또는 터미널 `./gradlew assembleDebug` → `app/build/outputs/apk/debug/app-debug.apk`.
3. **설치**: 실 기기 USB 연결 후 Run, 또는 `adb install app-debug.apk`(기기 "출처를 알 수 없는 앱" 허용 필요).
4. **권한 설정**: 앱 실행 → 메인 화면 버튼으로 ① 접근성 ② 알림 접근 권한 ON.
5. **검증**: 테스트 알림/접근성 테스트 화면으로 수집 확인 → 실제 배민커넥트 배차 수신 → 로그 내보내기.

> 5단계 개발자는 위 버전 조합으로 Gradle wrapper(8.7)·AGP(8.5.2)를 설정한다. 별도 추가 결정 불필요.
