# 제품 요구사항 명세 (PRD) — PickUpMemo

- 작성일: 2026-06-15
- 근거 문서: `01-상담/요구사항-정의서.md`

---

## 1. 개요

배민커넥트 배차 화면에서 업체 상호(식당명)를 Android 접근성 API와 알림 데이터로 읽어올 수 있는지 검증하기 위한 기술 검증 전용 로그 수집기다. 개발자 본인(1인)이 단일 Android 기기에 설치해 실제 배차 상황에서 데이터 수집 여부를 확인한다. 이 앱은 "데이터를 확보할 수 있는가"를 증명하는 도구이며, 예쁜 화면이나 서비스 앱 수준의 완성도를 목표로 하지 않는다.

## 3. MVP 스코프 (이번 사이클)

### 포함 (P0 + P1 전체)

- 접근성 서비스: 화면 텍스트·ContentDescription·View ID·패키지명·Activity 수집 및 로그 기록
- 알림 수신기 서비스: 알림 제목·내용·패키지명·시각 수집 및 로그 기록
- 로그 기기 내부 저장 (txt 형식, 외부 전송 없음)
- 메인 화면 (서비스·권한 상태 표시 + 로그 보기·내보내기 버튼)
- 로그 내보내기 (공유 Intent, txt 파일)
- 테스트 알림 생성 버튼 (데모)
- 접근성 테스트 화면 (데모)
- 로그 조회 화면 (최신순 목록 + 패키지명 필터 + 전체 삭제)

### 제외 (이번엔 만들지 않는다)

- 메모 기능 (v2)
- 팝업 기능 (v2)
- 서버/클라우드 저장 (v2 이후)
- AI/OCR (검증 실패 시 v2 대안 검토)
- 회원가입/로그인 (해당 없음)
- 디자인 완성도 (기능 동작 확인 가능한 최소 수준)

## 4. 미확정 3항목 — 기획 단계 결정

요구사항 정의서에서 다음 단계로 넘긴 미확정 항목 3개를 기술 검증 도구 특성에 맞게 결정한다.

### 4-1. 로그 내보내기 파일명 자동 생성 규칙

**결정**: `pickupmemo_log_YYYYMMDD_HHmmss.txt`

- 예시: `pickupmemo_log_20260615_143312.txt`
- 내보내기 버튼을 누른 시각을 기준으로 자동 생성한다.
- 별도 파일명 입력 화면은 만들지 않는다.
- 이유: 검증 도구는 파일을 여러 번 내보내는 반복 사용 패턴이 있다. 타임스탬프를 파일명에 포함하면 별도 설명 없이 수집 시점을 식별할 수 있다.

### 4-2. 로그 보관 한도

**결정**: 최대 5,000건 유지. 초과 시 가장 오래된 로그부터 자동 삭제.

- 이유: 기술 검증은 단기간 집중 수집 후 즉시 내보내는 패턴이다. 5,000건은 수천 건의 접근성 이벤트를 수집해도 여유 있는 한도이며, 기기 저장 용량 소모를 방지한다. 보관 한도 초과를 사용자에게 별도 알리지 않고 자동 순환한다.

### 4-3. 패키지명 필터 설정 영속성

**결정**: 앱 재시작 시 필터 초기화 (전체 로그 표시 상태로 복귀).

- 이유: 기술 검증 도구는 세션마다 관심 패키지가 달라질 수 있다. 이전 필터가 그대로 남아 있으면 "왜 데이터가 안 나오지?"류의 혼선이 생긴다. 매 세션마다 의도적으로 필터를 입력하게 하는 것이 검증 도구에 더 안전하다.

## 5. 핵심 사용자 흐름 (시나리오 3개)

### 시나리오 A — 첫 설치 및 권한 설정

1. 사용자가 APK를 설치하고 앱을 실행한다.
2. 메인 화면에서 접근성 권한 허가 상태가 "미허가"로 표시된다.
3. 사용자가 안내에 따라 Android 접근성 설정 화면으로 이동해 PickUpMemo 접근성 서비스를 활성화한다.
4. 메인 화면으로 돌아오면 접근성 서비스 상태가 "실행 중"으로 바뀐다.
5. 같은 방식으로 알림 접근 권한을 허용한다.
6. 메인 화면의 두 권한 상태가 모두 허가됨으로 표시된다 → 수집 준비 완료.

### 시나리오 B — 데모 기능으로 수집 동작 확인

1. 사용자가 메인 화면에서 "테스트 알림 생성" 버튼을 누른다.
2. 기기에 "신규 배달 요청 / 교촌치킨 죽전점 / 배달료 3,900원" 알림이 표시된다.
3. 사용자가 로그 조회 화면으로 이동해 방금 수신된 알림 로그가 기록됐는지 확인한다.
4. 이어서 메인 화면에서 "접근성 테스트 화면"으로 이동한다.
5. 모의 배차 화면("교촌치킨 죽전점" 텍스트 포함)이 열리고, 잠시 후 로그 조회 화면에서 해당 텍스트가 접근성 로그로 기록됐는지 확인한다.
6. 두 방식 모두 기록 확인 → 수집 기능 정상 동작 검증 완료.

### 시나리오 C — 실제 배차 후 로그 내보내기 및 분석

1. 사용자가 실제 배민커넥트 앱을 사용하며 배차를 수신한다.
2. 배차 화면이 표시되는 동안 접근성 서비스가 화면 텍스트를 자동으로 수집한다.
3. 배차 알림이 도착하면 알림 수신기가 알림 데이터를 자동으로 기록한다.
4. 사용자가 메인 화면에서 "로그 내보내기" 버튼을 누른다.
5. `pickupmemo_log_20260615_143312.txt` 형식의 파일이 생성되고 공유 Intent가 열린다.
6. 사용자가 카카오톡 또는 Gmail로 파일을 전송해 PC에서 열어본다.
7. 파일 안에 "교촌치킨 죽전점" 또는 이를 유추할 수 있는 텍스트가 있으면 검증 성공 → v2 개발 착수 결정.

## 6. 비기능 요구

- **외부 통신 없음**: 수집 데이터는 기기 내부 저장소에만 기록한다. 인터넷 권한은 로그 전송 용도로는 사용하지 않는다.
- **백그라운드 유지**: 접근성 서비스와 알림 수신기는 앱이 백그라운드에 있거나 화면이 꺼진 상태에서도 지속 동작해야 한다.
- **로그 형식 일관성**: 모든 로그 항목은 `[YYYY-MM-DD HH:mm:ss] / Package: / Type: / Text or Title / Body` 구조를 유지해 육안 분석이 가능해야 한다.
- **권한 상태 즉시 반영**: 사용자가 권한을 변경하고 앱으로 돌아왔을 때 메인 화면 상태 표시가 즉시 갱신되어야 한다.
- **디자인 수준**: 기능 동작 확인이 가능한 최소 수준. 시각적 완성도는 요구하지 않는다.

## 7. 다음 단계(설계)로 넘기는 결정 사항

- Android Native + Kotlin + Material 3 + minSdk 26 (API 26, Android 8.0)은 고객이 명시한 환경이다. 설계자가 최종 확정한다.
- 접근성 서비스(`AccessibilityService`)와 알림 수신기(`NotificationListenerService`)는 Android 시스템 권한이 필요하다. 권한 요청 흐름 설계는 3단계에서 구체화한다.
- 로그 파일 저장 위치(내부 저장소 경로)와 공유 Intent 구현 방식(FileProvider 등)은 설계 단계에서 결정한다.
- 백그라운드 서비스 유지를 위한 포그라운드 서비스 사용 여부는 설계 단계에서 결정한다.
- 납품 형태: Android Studio에서 열 수 있는 전체 프로젝트 소스 + 디버그 APK 빌드 가능 상태. 설치·권한·테스트 절차 안내 문서 포함.
