# 회고 — Todo-Developer (v3 Workflow 검증)

> 실험 종료 후 Sonnet이 작성. 각 항목은 **관찰된 사실 + 판단**으로 채운다.

## 1. 성공 조건 달성 여부

| 조건 | 달성 | 근거 |
|------|------|------|
| Builder가 기능 구현 가능 | ✅ | 5개 기능 + 엔터키 추가까지 결국 모두 구현. 단, 4라운드 수정이 필요했음(⑤ 참조) |
| Reviewer가 실제 문제 발견 | 🔺 부분 | taskList 섀도잉을 "중간" 심각도로는 짚었으나(③), 더 치명적인 onclick 스코프 버그는 "문제없음"으로 오판. 게다가 컨텍스트 길이 제약(400에러)으로 전체 파일 리뷰 자체가 불가능했음 |
| Architect가 구조적 문제 발견 | 🔺 부분 | 구조 적절성·범위 준수는 정확히 판단했으나, 실제 앱을 마비시키는 치명 버그 2건(classList 크래시, innerHTML+= 리스너파괴)은 "구조" 관점 밖이라 애초에 못 잡음(정적 검토의 한계) |
| Sonnet이 최종 의사결정 수행 | ✅ | Reviewer 오판 정정(③→④), Architect 의견 수용 여부 판단(⑤→⑥), 로컬 LLM 전원이 놓친 치명 버그 2건을 실행 테스트로 직접 발견·수정(⑥) |
| 로컬 LLM이 토큰 절감에 기여 | ✅ | 코드 생성/재작성(초안+3라운드, 매회 6KB 내외)을 전부 위임 — Sonnet이 직접 타이핑했다면 훨씬 컸을 출력 토큰을 아꼈음 |
| Workflow 무중단 완료 | ✅ | 400 에러·타임아웃 등 여러 장애가 있었지만 전부 우회/재시도로 극복, 중단 없이 ⑦까지 완주 |

**종합 판정: 부분 성공.** 워크플로우 자체(Build→Review→Architect→Final Approval 루프, 기록 체계)는 무중단으로 완주했고 로컬 LLM 위임으로 코드 생성 부담은 확실히 줄었다. 그러나 **로컬 리뷰어/아키텍트 리뷰만으로는 실제 동작하는 앱을 보장하지 못했다** — 최종적으로 앱을 마비시킬 치명 버그 4건 중 3건(onclick 스코프, classList 빈문자열, innerHTML+= 리스너파괴)을 Sonnet이 직접 코드 재검토·실행 테스트로 잡아야 했다. "로컬 LLM에게 위임 + Sonnet은 병합·판단만" 이라는 애초 설계보다 Sonnet의 개입 비중이 컸다.

## 2. 역할별 평가

### Builder (local_builder, qwen2.5:14b)
- 잘한 점: 요구사항을 코드로 옮기는 1차 변환은 빠르고 성실함(형식 준수, 지시한 수정사항을 문자 그대로 반영). 5기능 + 엔터키까지 결국 다 구현.
- 한계: (1) 인라인 이벤트 핸들러(onclick/onkeydown) 패턴을 습관적으로 반복 — R1에서 고쳐줘도 R3에서 새 기능을 짤 때 같은 패턴으로 회귀함(④ 참조). (2) `label.innerHTML += text` 같은 흔한 안티패턴을 스스로는 못 피함. (3) 300초 타임아웃 1회 발생.

### Reviewer (local_reviewer, deepseek-coder-v2:16b)
- 실제로 잡아낸 문제: taskList 매개변수명 섀도잉(단, 심각도 과소평가 — "혼동 가능" 수준으로만 봄, 실제로는 크래시).
- 놓친 것 / 오탐: onclick 스코프 버그를 "문제없음"으로 명백히 오판(치명적 false negative). 근본 원인은 **컨텍스트 길이 제약** — system prompt+파일+질문 합계가 2~3KB만 넘어도 400 에러가 나서, 전체 파일을 못 보고 잘라낸 스니펫만으로 판단했기 때문. 이 환경에서는 "현재 코드 파일을 함께 넘겨야 실효 있는 리뷰"(실험프로토콜 5장 권장)가 사실상 불가능했다.

### Architect Reviewer (local_architect_reviewer, qwen3:30b-a3b)
- 구조적으로 기여한 점: 단일 파일 구조 적절성, MVP 범위 준수 여부를 정확하고 간결하게 판단. 우려했던 300초 타임아웃도 이번엔 발생하지 않음(6.5KB 파일, 1회 호출로 정상 응답).
- 한계: "구조/범위" 관점만 보므로 애초 역할 밖이지만, 실제 기능이 동작하는지는 전혀 검증하지 못함(그 몫이 아니었다는 게 정확한 평가). qwen3가 "느리다"는 알려진 리스크는 이번엔 재현되지 않았다.

### Sonnet (오케스트레이터)
- 의사결정 품질: Reviewer의 오판을 놓치지 않고 직접 코드로 재검증해 정정했고(③→④), 로컬 LLM 3종이 전혀 잡지 못한 치명 버그 2건(⑥)을 실행 테스트로 발견 — 최종 승인 전 반드시 "실행해서 확인"해야 한다는 원칙이 결정적으로 작동함.
- 의견 병합 방식: Reviewer/Architect 의견을 그대로 받아쓰지 않고 매번 Sonnet이 직접 코드를 열어 근거를 재확인한 뒤 반영/기각을 결정 — 로컬 모델의 판단을 "1차 후보"로만 쓰고 최종 검증은 항상 Sonnet이 함.

## 3. 워크플로우 자체 평가
- 흐름이 자연스러웠는가: 큰 틀(①~⑦)은 자연스러웠으나, ③↔④ 루프가 프로토콜상 "3라운드"인데 회귀 버그 때문에 사실상 4라운드까지 늘어났고, ⑤ 이후에도 Sonnet 단독의 "⑥ 실행테스트 기반 수정" 라운드가 프로토콜에 아예 정의돼 있지 않았음(즉흥적으로 만들어 채움).
- 병목: (1) local_reviewer의 컨텍스트 길이 제약으로 인한 400 에러 디버깅(약 6분 소요) (2) 실행 기반 검증 단계가 워크플로우에 없어서 Sonnet이 즉석에서 jsdom 스모크테스트를 만들어야 했음 (3) claude-in-chrome 확장 미연결로 계획했던 실브라우저 검증이 막혔다가 사용자가 Playwright 존재를 알려줘서 우회.
- 기존 방식(Sonnet 단독) 대비 생산성 체감: 코드 타이핑량은 확실히 줄었으나(로컬 LLM이 매 라운드 6KB 안팎 생성), 버그를 "찾아내는" 핵심 작업은 결국 Sonnet 몫이었다. 즉 이 구조는 "타이핑 절감"에는 유효하지만 "판단 절감"에는 아직 미흡하다.

## 4. 토큰 절감 효과
- 정량 측정은 안 됐음(도구가 토큰 사용량을 반환하지 않아 workflow_log에도 "미측정"으로 남김).
- 정성: 초안 1회 + 수정 3회, 매회 약 6KB 분량의 완성된 HTML/CSS/JS를 Sonnet이 직접 작성하지 않고 로컬 모델에 위임함 — 출력 토큰 기준으로는 상당한 절감으로 추정됨.
- 절감과 품질의 트레이드오프: 이번 실험에서 뚜렷하게 드러남 — 코드 생성은 위임해도 되지만, **버그 탐지·최종 검증은 위임 폭을 줄이고 Sonnet이 실행까지 해봐야** 신뢰할 수 있는 결과가 나왔다.

## 5. 한 줄 결론
- v3 Workflow는 계속 쓸 만한가? — **조건부로 그렇다.** "코드 생성은 로컬 LLM에 위임 + Sonnet은 실행 기반 검증을 반드시 수행"으로 역할을 재정의하면 쓸 만하다. 현재처럼 "Reviewer/Architect의 정적 판단만 믿고 넘어가는" 구조로는 실제 동작하는 결과물을 보장하지 못한다 — improvement.md의 "실행 기반 검증 단계 신설" 제안이 v4의 핵심이 되어야 한다.
