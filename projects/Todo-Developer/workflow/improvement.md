## v4 워크플로우로 반영할 우선순위 제안
1. **실행 기반 검증 단계를 정식으로 신설한다.** Review(정적 리뷰)와 Architecture Review(구조 검토)만으로는 실제 동작을 보장 못 함이 이번 실험에서 확인됨(치명 버그 2건을 jsdom/Playwright 실행 테스트로만 발견). Build 직후 또는 Final Approval 직전에 "Sonnet이 headless 실행 테스트를 돌린다"를 프로토콜에 명문화.
2. **local_reviewer(deepseek-coder-v2:16b)의 컨텍스트 길이 제약을 감안한 리뷰 전략을 세운다.** 이 환경에서 system prompt+파일+질문 합계가 2~3KB만 넘어도 400 에러가 나므로, "전체 파일 첨부" 대신 "의심 구간만 잘라 첨부"를 기본 전략으로 삼거나, 애초에 이 역할엔 컨텍스트가 더 큰 모델을 배정하는 걸 검토.
3. **Builder에게 "직전에 고친 패턴"을 다음 요청에도 명시적으로 재상기시킨다.** 한 번 고친 버그 유형(인라인 이벤트 핸들러 스코프)이 새 기능 추가 시 재발했음 — 매 Build 수정 요청에 "지난 라운드에서 지킨 규칙: …"을 프롬프트에 포함시키는 걸 표준 절차로.
