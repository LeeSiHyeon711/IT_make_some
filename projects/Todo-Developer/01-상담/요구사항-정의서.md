# 요구사항 정의서 — Todo-Developer

## 1. 프로젝트 성격

**IT상상공방 v3 Workflow 검증 실험.** Todo 페이지는 실험용 소재일 뿐, 진짜 목표는 Sonnet + 로컬 LLM(Builder/Reviewer/Architect)의 협업 워크플로우가 실제 프로젝트에서 동작하는지 검증하는 것이다.

## 2. 만들 것 (MVP · 고정)

단일 웹페이지(Single Page). 웹앱 아님.

| 기능 | 설명 |
|------|------|
| Todo 추가 | 입력 후 목록에 항목 추가 |
| Todo 삭제 | 항목 개별 삭제 |
| 완료 체크 | 항목 완료/미완료 토글 |
| 필터 | 진행중 / 완료 (전체 포함 가능) |
| 저장 | LocalStorage에 상태 영속화, 새로고침 후 유지 |

여기까지만. 그 외 기능(우선순위, 마감일, 정렬, 다중 목록, 서버 연동 등) **추가 금지**.

## 3. 비기능 요구

- 디자인: 심플한 MVP 수준. 화려함 불필요.
- 기술: 순수 HTML/CSS/JS 단일 파일 권장(빌드 도구·프레임워크 없이). 과설계 금지.
- 브라우저에서 파일 열어 바로 동작(정적).

## 4. 검증 대상 (진짜 목표)

- Builder: 구현 능력·코드 품질·요구사항 이해·속도
- Reviewer: 실제 버그 발견·요구사항 누락 탐지·품질 개선 제안·비판적 리뷰
- Architect: 구조 문제·MVP 범위 유지·과설계 방지·기술 선택·확장성
- Sonnet(Claude): 최종 의사결정·의견 병합·반영 여부 판단
- Workflow: 역할 분리 동작·Claude 협업 가능성·흐름의 자연스러움·토큰 절감·생산성

## 5. 성공 조건

Builder 구현 / Reviewer 문제 발견 / Architect 구조 문제 발견 / Sonnet 최종 의사결정 / 로컬 LLM의 토큰 절감 기여 / Workflow 무중단 완료 — 6개 모두 만족 시 성공.

## 6. 제약·주의

- 실행 주체는 **Sonnet**. 요구사항 정의서·틀은 Opus가 작성, 실행 개입 없음.
- 모든 단계는 `workflow/workflow_log.md`에 기록(10개 항목). 기록이 곧 산출물.
- 구동 절차는 `실험프로토콜.md`를 따른다.
