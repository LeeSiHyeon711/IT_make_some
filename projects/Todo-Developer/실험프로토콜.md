# 실험 프로토콜 — v3 Workflow 구동 지침 (Sonnet 전용)

> 이 문서는 **실행 주체인 Sonnet**이 실험을 시작하기 전에 반드시 읽고, 진행 중 그대로 따르는 구동 지침이다.
> Opus(틀 제작자)는 이 문서를 만들었을 뿐 실행에 개입하지 않는다. 종료 후 평가만 한다.

---

## 0. 대전제

- 목적은 **Todo 앱 완성이 아니라 v3 협업 워크플로우 검증**이다. 기록이 곧 산출물이다.
- **모든 단계는 `workflow/workflow_log.md`에 즉시 기록**한다. 기록 안 된 단계는 없었던 것으로 본다.
- MVP 범위(단일 페이지 · 5기능)를 넘는 구현은 금지. 범위 초과 제안이 나오면 Architect 검토에서 걸러 Reject 기록한다.

## 1. 역할 ↔ 로컬 LLM 도구 매핑 (검증 완료)

| 워크플로우 역할 | MCP 도구 | 모델 | 파일 첨부 버전 |
|-----------------|----------|------|----------------|
| Builder | `mcp__local-llm__ask_builder` | local_builder (qwen2.5:14b) | `ask_builder_with_files` |
| Reviewer | `mcp__local-llm__ask_deepseek` | local_reviewer (deepseek-coder-v2:16b) | `ask_deepseek_with_files` |
| Architect Reviewer | `mcp__local-llm__ask_qwen3` | local_architect_reviewer (qwen3:30b-a3b) | `ask_qwen3_with_files` |

- 코드 리뷰/구조 검토 시에는 **현재 코드 파일을 함께 넘겨야** 실효 있는 리뷰가 나온다 → `*_with_files` 사용 권장.
- 이 도구들은 deferred 상태일 수 있다. 첫 사용 전 **한 번의 `ToolSearch`로 필요한 것을 모두 로드**한다:
  `select:mcp__local-llm__ask_builder_with_files,mcp__local-llm__ask_deepseek_with_files,mcp__local-llm__ask_qwen3_with_files`

## 2. 워크플로우 루프

```
① Planning (Sonnet)
      ↓  요구사항 → 구현 계획/파일 구조 확정, PRD.md 작성
② Build (local_builder)
      ↓  계획대로 단일 페이지 구현 초안
③ Review (local_reviewer)  ── 버그/요구사항 누락/품질
      ↓
④ Build 수정 (local_builder)   ← ③⑤ 지적을 반영
      ↓  (③↔④ 필요 시 여러 번 반복)
⑤ Architecture Review (local_architect_reviewer) ── 구조/과설계/범위/확장성
      ↓
⑥ Build 수정 (local_builder)   ← ⑤ 반영
      ↓
⑦ Final Approval (Sonnet)  ── 의견 병합, 반영 여부 최종 판단
      ↓
   Complete → 05-개발 코드 확정 → workflow/ 3종 문서 완결 → push
```

- ③↔④ 반복은 허용한다. 단 **무한 루프 방지**: 같은 지적이 2회 반복되거나 3라운드를 넘으면 Sonnet이 판단해 종료한다.
- 각 로컬 LLM 호출 **직전에 타이머 시작, 직후 종료 시각 기록** → `9. 시간` 필드에 소요 시간 적는다.

## 3. 단계별 기록 의무 (workflow_log.md의 10개 항목)

각 단계마다 아래 10개를 모두 채운다. 비면 "N/A" 명시.

1. 단계명  2. 담당 모델  3. 입력(핵심 요청)  4. 출력(핵심 응답 요약)
5. 반영 여부 (**Accepted / Modified / Rejected** + 이유)
6. Reviewer 의견  7. Architect 의견  8. Claude(Sonnet) 최종 판단(반영/미반영 근거)
9. 시간(소요)  10. Claude 토큰 사용량(Input/Output, 가능 범위)

> 5·8번이 이 실험의 핵심이다. "왜 반영했는가 / 왜 안 했는가"를 반드시 근거와 함께 남긴다.

## 4. 반드시 별도로 수집하는 개선 관찰 (→ improvement.md)

진행 중 아래가 관찰되면 즉시 improvement.md에 적는다:
역할 충돌 · 프롬프트 개선점 · 로컬 LLM 한계 · Claude 한계 · 워크플로우 병목 · 새 아이디어 · 개선 가능한 구조.

## 5. 알려진 리스크 / 운영 팁

- **qwen3(Architect)는 느리다.** 실제로 300초 read timeout이 관측됨. Architect 호출은 프롬프트를 짧고 명확하게, 코드 파일만 첨부한다. 타임아웃 시 1회 재시도 후에도 실패하면 로그에 "미수신"으로 남기고 진행한다.
- 로컬 LLM은 지시한 출력 형식을 벗어나거나(예: 시작 문구 무시) 마커를 누락할 수 있다. **형식보다 내용(실제 버그/구조 지적)의 유효성**으로 평가한다.
- 로컬 LLM 응답은 `MODEL_RAW` 마커로 감싸져 올 수 있다(서버 측 주입). 내용만 취해 요약한다.
- Sonnet은 로컬 LLM에 **일을 위임**하는 것이 핵심이다. 토큰 절감이 성공 조건이므로, 직접 다 짜지 말고 Builder에게 시키고 검수·병합만 한다.

## 6. 종료 절차

1. `05-개발/`에 최종 단일 페이지(`index.html` 등) 확정.
2. `workflow/workflow_log.md` 전 단계 완결.
3. `workflow/retrospective.md`(회고), `workflow/improvement.md`(개선) 작성.
4. `progress.md`를 "실험 완료"로 갱신.
5. 원격 repo `Todo-Developer`에 push.
6. 사용자에게 완료 보고 → 이후 사용자가 **Opus를 평가자로 호출**한다.
