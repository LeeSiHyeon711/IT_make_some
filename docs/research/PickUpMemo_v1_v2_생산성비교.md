# PickUpMemo v1 → v2 생산성 비교 (포트폴리오용)

- 정리일: 2026-06-17
- 핵심 질문: **Builder를 Opus→Sonnet + FEAT 1:1 매칭 + 이슈 압축 적용 시, Builder 평균 토큰이 얼마나 감소하는가?**
- 결론(요약): Builder 단일 호출 **전체 평균 ≥58.6k → 42.9k**, 즉 **최소 −26.8%이며 실제 개선폭은 그 이상**. (v1의 미기록 빌더 작업이 기록분보다 토큰을 더 많이 썼으므로 58.6k는 v1 실제 평균의 **하한값**이다.) 모델 단가 인하(Opus→Sonnet)까지 더하면 Builder 단계 **비용 절감폭은 훨씬 큼**. 단, 절감의 대가는 **설계 단계로 이동**(architect 토큰 대폭 증가).

## 3. 핵심 비교 — Builder 전체 평균 토큰 (대표 지표)

| 지표 | v1 (Opus) | v2 (Sonnet) | 변화 |
|------|-----------|-------------|------|
| **Builder 호출당 전체 평균** | **≥ 58.6k** (하한) | **42.9k** (실측 14개) | **≤ −15.7k / ≥ −26.8%** |

- v2 = 전체 14개 호출의 실측 평균 **42.9k** (확정값).
- v1 = 기록된 5개로 산출한 **58.6k이며, 미기록 작업이 더 무거웠으므로 실제 평균은 그 이상**. 따라서 v1을 58.6k로 잡은 −26.8%는 **보수적 하한**이고, 실제 절감폭은 더 크다.
- 결론 메시지: **"Builder 호출당 평균 토큰을 최소 27%(보수적), 실제로는 그 이상 줄였다."**

> 참고(무거운 작업끼리 비교): v2 상위 5개 평균은 55.2k다. v1 기록 5개(58.6k)와 비교하면 그 자체로도 낮지만, 이 5개 비교는 v1 미기록분(더 무거움)을 빼놓은 것이라 **개선폭을 오히려 축소해 보이게 한다.** 대표 지표는 위의 **전체 평균(≥−26.8%)** 을 사용한다.

### 비용 관점 (토큰 × 단가)
토큰 수 감소(−27%)는 **FEAT 1:1 매칭 + 최소 문서 읽기(헌법 8조) + 이슈 압축**에서 나온다(모델과 무관).
여기에 Builder 모델을 **Opus→Sonnet**으로 내리면서 **토큰당 단가**가 크게 낮아진다. 두 효과가 곱으로 작용하므로 Builder 단계의 실제 **비용 절감폭은 토큰 절감폭(−27%)보다 훨씬 크다.**
(정확한 비용 배수는 당시 모델 단가표 기준으로 별도 산정 필요 — 여기서는 토큰 실측만 확정.)

## 4. 절감의 대가 — 무거운 판단은 설계 단계로 이동

v2의 전략(CLAUDE.md 4-1)은 "**무거운 판단을 설계 게이트로 몰고, 실행(Builder)은 싸게**"이다. 그 결과 토큰 부담이 Builder→architect로 이동했다.

| 단계 | v1 | v2 | 비고 |
|------|----|----|------|
| 상담(consultant) | 21.6k | 34.1k | Sonnet |
| 기획(planner) | 23.0k | 21.3k | Sonnet |
| 설계(architect) | 31.8k | **228.6k** | Opus, FEAT 13개 작성(1차 112.2k + v2.1 추가 116.4k) |
| 납품(delivery) | 59.8k | 67.3k | Sonnet |
| 자동 QA(reviewer) | — | 89.7k | Sonnet (v1엔 분리 단계 없었음) |

- 설계가 무거워진 이유 = **Builder가 "FEAT 문서 1개만 읽고 그대로 구현"** 할 수 있을 만큼 상세히 설계를 펼쳤기 때문. 즉 **싼 모델(Sonnet)로 구현하려면 FEAT 품질이 곧 결과 품질**이라, 설계 투자(Opus)가 늘어난다.
- 트레이드오프 해석: 한 번 잘 만든 FEAT 문서는 재사용·재개·검증에 유리하고, 반복 호출되는 Builder를 싸게 유지한다. **반복 비용(Builder)을 1회성 비용(설계)으로 치환**한 구조.

## 6. 방법론·한계 (정직성 노트)

- v2 토큰은 이번 단일 세션의 builder subagent_tokens **전체 14개 실측**. v1은 기록이 남은 5개 기준 58.6k인데, **미기록 v1 빌더 작업이 기록분보다 더 많이 소모**했으므로 58.6k는 v1 실제 평균의 **하한**이다. 따라서 −26.8%는 보수적 수치이며 실제 개선폭은 더 크다. (v1 전체 실측이 없어 상한은 미상 — 단 방향은 "더 큼"으로 확정.)
- 시간·총 토큰·Builder당 정확 평균 중 일부는 v1/v2 모두 외부 타이머 부재로 "측정 불가" 항목 존재.
- v2는 v1(로그 수집기)보다 **기능 범위가 넓다**(메모 CRUD + 팝업 + 로그 보관함 + 테스트 화면). 이슈 수 단순 비교는 부적절하며, **호출당 토큰**이 더 공정한 지표.
- 환경 제약: 이번 세션은 하위 에이전트 Bash가 차단되어 빌드·git·gh를 공방장이 대행함(별도 보고서: `~/Desktop/공방_Git권한이슈_보고서.md`). builder 토큰 자체에는 영향 없음(코드 작성만 수행).
