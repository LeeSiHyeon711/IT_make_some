## OPS-01 · repo-manager와 공방장의 git 권한 분리 구조 정리

- **상태**: 설계 승인(기본 L1.5) — 적용 이슈 A~J로 분해 등록됨([설계 문서](./OPS-01-git-flow-design.md))
- **등록일**: 2026-06-21
- **영향 범위**: `.claude/agents/repo-manager.md`, 공방장(오케스트레이터) 커맨드(`/개발착수`·`/검수`·`/납품`·`/이슈동기화`), `docs/pipeline.md`, 루트 `CLAUDE.md`(헌법 2장 9 Git 게이트)

### 배경
현재 하위 에이전트는 git commit/push 권한이 없고, repo-manager가 git 작업 계획을 보고하면 공방장 에이전트가 실제 명령을 실행하는 구조로 동작한다. 이 방식은 안전하지만 동선이 번거롭고 자동화 흐름이 끊긴다.

### 검토할 방향
1. 하위 에이전트는 **코드 수정과 테스트까지만** 담당한다.
2. repo-manager는 **git diff 확인, 커밋 메시지 작성, push 계획 수립**을 담당한다.
3. 공방장은 **사용자 승인 후 실제 commit/push 실행**을 담당한다.
4. **위험 명령어는 반드시 사용자 승인**을 거친다.
5. **commit/push 자동화 수준을 단계별로 정의**한다.

### 목표
- 권한 분리의 안전성은 유지한다.
- 반복적인 git 명령 실행 동선을 줄인다.
- `/개발착수`, `/검수`, `/납품` 단계에서 git 작업 흐름을 명확히 한다.

### 현황 메모(검토 시작점)
- 현재 헌법: GitHub 쓰기(`gh`/push)는 repo-manager만 수행하고, 공방장은 직접 `gh`/push 하지 않는다(`/이슈동기화` 원칙). 단, QA 진입 전 Git 반영 검증과 실행 게이트는 사람 `/승인`에 묶여 있다(헌법 2장 9).
- repo-manager 정의상 "공방 내에서 `gh`/git push를 쓸 수 있는 유일한 에이전트"로 명시되어 있어, 위 검토 방향 3(공방장이 실제 실행)과 충돌·중복 소지가 있다 → 권한 주체를 한 곳으로 합칠지, 계획(repo-manager)·실행(공방장) 이원화를 유지할지 결정 필요.

### 결정 필요 사항(검토 산출물)
- [x] 실제 commit/push 실행 주체 → **repo-manager 단일 유지**(공방장 직접 실행 안 함) — 설계 문서 C1
- [x] 자동화 레벨 정의 → **기본 L1.5**, L2는 승격 목표 — 설계 문서 C2
- [ ] "위험 명령어" 목록은 설계 문서 5장에 1차 정의(차단 방식 C5 미결)
- [x] 단계별 git 동선 → 설계 문서 Q3·4-3·4-4에 정리 완료

- **라벨**: ops, sangsang-os, git-flow, pending, high, **high-risk**

- **대상 파일**: `.claude/hooks/`(신규 `block-dangerous-git.py` 또는 `block-github-write.py` 확장), 관련 에이전트 훅 설정.

- **완료 조건**: 위험 명령은 차단되고 L1.5 정상 명령은 통과.
- **검증 방법**: 위험 명령 샘플 차단 + 정상 명령 통과를 더미로 확인(J와 연계).
