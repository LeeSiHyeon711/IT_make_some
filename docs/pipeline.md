# IT상상공방 생산 라인 상세 (Pipeline)

7단계 각각의 **입력·처리·출력·완료조건**을 정의한다. 에이전트와 규칙 파일은 이 문서를 기준으로 구현한다.
용어는 루트 `CLAUDE.md`, 구조는 `docs/architecture.md` 참조.

---

## 단계 1 · 상담 (Consultation)

- **목적**: 고객의 막연한 상상을 "확인 가능한 요구사항"으로 변환.
- **입력**: 고객의 최초 요청 (자연어, 종종 모호함)
- **처리**:
  1. 핵심 질문 — 무엇을 만들고 싶은가 / 누가 쓰는가 / 왜 필요한가 / 성공의 기준은
  2. 모호한 점은 반드시 되묻는다 (가정 금지)
  3. 결과물 형태는 **단정하지 않는다** (웹/앱/자동화 결정은 3단계 몫)
- **출력**: `01-상담/요구사항-정의서.md`
- **완료 조건**: 고객이 "맞다"고 확인할 수 있을 만큼 요구사항이 구체적이다.

## 단계 2 · 기획 (Planning)

- **목적**: 요구사항을 만들 수 있는 제품 명세(PRD)로 구체화.
- **입력**: `01-상담/요구사항-정의서.md`
- **처리**:
  1. 기능 목록화 → 우선순위 부여
  2. **MVP 스코프 컷**: "이번에 꼭 보여줄 것" vs "나중 것" 분리
  3. 사용자 흐름(핵심 시나리오) 1~3개 정의
- **출력**: `02-기획/PRD.md`
- **완료 조건**: 무엇을 만들지가 명확해 설계자가 형태/스택을 고를 수 있다.

## 단계 3 · 설계 (Design) — ★ 형태 결정 + FEAT 문서 분기점

- **목적**: 결과물 형태·스택·구조 확정 + **이슈별 FEAT 문서 생성**. 공방에서 가장 중요한 의사결정(Opus).
- **입력**: `02-기획/PRD.md`
- **처리**:
  1. **형태 결정**: 웹 / 앱 / 자동화 중 "가장 빠르게 확인시킬 수 있는" 것
  2. 스택 결정 + 근거 명시
  3. 파일·모듈 구조, 화면/플로우 스케치
  4. ★ **FEAT 문서 생성(v2)**: 작업 분해를 **이슈 1개 = FEAT 문서 1개**로 펼친다. `03-설계/features/FEAT-NN-*.md`, 번호는 이슈 번호와 일치. `templates/FEAT.md` 10개 섹션을 채운다.
  5. **이슈 개수 최적화**: 무작정 쪼개지 말 것. 소형 5~8 / 중형 8~12 / 복잡 12~15. 관련 작업은 묶되(예: 데이터+저장소+Repository=저장 계층), UI와 DB·권한과 로직·서버와 화면·로직과 디자인은 섞지 않는다.
- **출력**: `03-설계/설계서.md` + `03-설계/features/FEAT-NN-*.md`
- **완료 조건**: builder가 **FEAT 문서 하나만 읽고** 그대로 만들 수 있고, repo-manager가 FEAT 1:1로 이슈를 등록할 수 있다.
- **게이트**: 사람이 설계서 + **FEAT 문서를 함께 승인**(FEAT 승인 게이트 — 싼 모델 구현의 품질 보증선).
- **주의**: 과한 스택 선택 금지. 정적 프로토타입으로 충분하면 풀스택으로 가지 않는다.

## 단계 4 · GitHub 관리 (Issue Tracking) — ★ 흐름 영속화

- **목적**: 설계자만 아는 전체 방향을 **추적 가능한 GitHub 이슈로 박제**해, 중간에 끊겨도 재개 가능하게 만든다.
- **입력**: `03-설계/설계서.md`
- **처리**:
  1. ★ **FEAT 1:1 등록(v2)**: FEAT 문서 1개 = 이슈 1개, **이슈 번호 = FEAT 번호**. 이슈 본문은 `templates/이슈본문.md`(설계서 복붙 금지, FEAT만 가리킴).
  2. 첫 실행: repo 생성 + **개발 디렉토리 git init·remote 연결**.
  3. 등록 결과를 `이슈등록내역.md`에 기록 (번호·링크·제목·매칭 FEAT·상태).
  4. ★ **Git 반영(v2)**: builder 완료 보고 → **commit/push 먼저 → 원격 확인 → 이슈 close.** (v1 "원격 미반영" 빈틈 차단)
  5. ★ **QA 진입 체크리스트**: 모든 이슈 close 후 헌법 2장 9 체크리스트 출력 → 사람 `/승인` 전까지 QA 미진입.
  6. **코드 내용 변경·설계 변경 금지** — commit/push와 이슈 번역·동기화만.
- **출력**: `04-깃허브/이슈등록내역.md` + 실제 GitHub 이슈 + 원격 repo의 commit
- **완료 조건**: 모든 이슈가 등록·close되고 **코드가 원격에 push 확인**됨.
- **권한**: 공방 내에서 `gh`/git commit·push를 가진 유일한 단계.

## 단계 5 · 개발 (Development)

- **목적**: FEAT 문서대로 동작하는 MVP 제작. (Sonnet)
- **입력(최소 3개, v2)**: ① `CLAUDE.md` ② 현재 GitHub 이슈 ③ 그 이슈의 `FEAT-NN` 문서. **설계서 전체를 습관적으로 읽지 않는다(헌법 2장 8).** 추가 문서가 필요하면 이유를 빌드노트에 적고 그것만 읽는다.
- **처리**:
  1. FEAT "구현할 것" 범위로 구현 (형태/스택은 FEAT/설계가 정함)
  2. FEAT 범위 밖·다른 FEAT 선구현·관련 없는 리팩터링 금지 (스코프 고정)
  3. 실행 방법 + 변경 파일 목록을 빌드노트에 기록, **commit 가능 상태로 정리**
  4. 진행 상태·변경 파일을 GitHub 관리자에게 보고 (직접 commit/close/push 안 함)
  5. `progress.md` 갱신(완료/진행/남은 이슈)
- **출력**: `05-개발/` (코드/결과물) + `05-개발/빌드노트.md` + `progress.md`
- **완료 조건**: FEAT 완료 조건 충족 + commit 가능 상태. (commit/push/close는 repo-manager)

## 단계 6 · 자동 QA (Review) + 수동 테스트 + 수정 순환

### 6-0. QA 진입 게이트 (★ v2 신설, 사람)
자동 QA 시작 전 **반드시 충족**: 모든 이슈 closed + 모든 FEAT 완료 조건 + 빌드 성공 + **Git commit/push 원격 반영 확인** + 사람 `/승인`.
repo-manager가 헌법 2장 9 체크리스트를 출력하고, 사람이 GitHub 웹에서 파일을 확인한 뒤 `/승인 <프로젝트>`. 미충족이면 reviewer 미호출.

### 6-a. 자동 QA (검수자, `/승인` 후 자동) — ★ 타입별 분기
- **목적**: 결과물이 요구사항·PRD·**FEAT 완료 조건**을 충족하는지 자동 검증.
- **입력**: `05-개발/` + 1~4단계 문서
- **처리**:
  1. 충족 체크리스트 작성 (요구사항 + 각 FEAT 완료 조건)
  2. **★ 프로젝트 타입별 실행 검증(v2)**:
     - 웹: `file://` 실제 브라우저(Playwright) / 서버형은 서버+화면+API
     - Android 앱: APK 빌드·위치/크기, **에뮬레이터 불가 기능은 "실기기 검증 필요" 명시**(특수 권한 별도 체크리스트)
     - API/Backend: endpoint·request/response·에러·DB seed
     - 문서/자동화: 산출물·입출력 예시·재실행 가능성
  3. 발견 이슈 기록 — **직접 고치지 않는다**
- **출력**: `06-검수/검수리포트.md`
- **보류 시**: repo-manager 등록 → builder 자동 수정 → repo-manager commit/push/close → 자동 재QA (자동 수렴).

### 6-b. 사람 수동 테스트 (★ 실사용 검증 게이트)
- 자동 QA 통과 후 **사람이 직접** 결과물을 사용해 본다. 자동이 못 잡는 실사용·환경 문제를 찾는다.
- 이상 없으면 `/납품`. 증상이 있으면 `/증상 <프로젝트> "증상"`.

### 6-c. 증상 수정 순환 (증상분석가, 자동)
- **목적**: 사람이 보고한 증상을 자동으로 수정.
- **처리**: triage가 **고객과 동일한 방식으로 재현** → 원인 분석 → `06-검수/증상분석log.md` 기록 → repo-manager 이슈 등록 → builder 수정 → reviewer 재QA.
- 수렴할 때까지 6-b ↔ 6-c 를 반복한다.

## 단계 7 · 납품 (Delivery)

- **목적**: 고객이 직접 확인·실행할 수 있게 패키징.
- **입력**: 전 단계 전체 + **사람 수동 테스트 통과**
- **처리**:
  1. `실행안내.md` — 고객이 결과물을 직접 보는/돌리는 법 (비개발자 눈높이)
  2. `고객전달문.md` — 무엇을 만들었는지 요약 + 다음 단계 제안
  3. ★ `증거자료체크리스트.md`(v2) — 포트폴리오·회고용 캡처 목록 자동 생성(공정/개발/실행/납품)
  4. ★ 루트 `생산성요약.md`(v2) — 단계별 토큰/시간을 모아 최종 요약(측정 불가는 정직히 표기)
- **출력**: `07-납품/실행안내.md`, `07-납품/고객전달문.md`, `07-납품/증거자료체크리스트.md`, 루트 `생산성요약.md`
- **완료 조건**: 고객이 추가 설명 없이 결과물을 확인할 수 있고, 포트폴리오 증거·생산성 요약이 준비됨.

## 단계 간 인계 규칙 (요약)

| 넘기는 단계 | 인계 산출물 | 받는 단계 | 진행 |
|-------------|-------------|-----------|------|
| 1 상담 | 요구사항-정의서.md | 2 기획 | ▣ 게이트 |
| 2 기획 | PRD.md | 3 설계 | ▣ 게이트 |
| 3 설계 | 설계서.md + FEAT-NN 문서들 | 4 GitHub 관리 | ▣ 게이트(설계+FEAT 승인 = 개발 착수) |
| 4 GitHub 관리 | 이슈등록내역.md + GitHub 이슈(FEAT 1:1) | 5 개발 | ⚙ 자동 |
| 5 개발 | 05-개발/ + 빌드노트.md + progress.md | 4 Git 반영(commit/push/close) | ⚙ 자동 |
| 4 Git 반영 | 원격 commit + QA 진입 체크리스트 | QA 진입 승인 | ▣ 게이트(`/승인`) |
| 6 자동 QA | 검수리포트.md(타입별) | 사람 수동 테스트 | ⚙ 자동 |
| 사람 수동 테스트 | (증상 보고 / OK) | 6-c 수정순환 / 7 납품 | ▣ 게이트 |
| 6-c 증상분석 | 증상분석log.md → 이슈 | 자동 수정→commit/push→재QA | ⚙ 자동(순환) |
| 7 납품 | 실행안내·고객전달문·증거체크리스트·생산성요약 | 고객 | ▣ 게이트 후 |

각 단계는 자기 폴더 밖에 쓰지 않는다(예외: progress.md·생산성요약.md는 루트 공용 추적 파일).
게이트는 **방향 결정(1~3) + QA 진입 승인(Git 반영 후 /승인) + 실사용 검증(수동 테스트)** 세 곳이고, 그 사이는 자동이다.
