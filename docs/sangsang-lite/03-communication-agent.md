# 03 · 소통 에이전트 (Communication Agent)

> **내부명**: 소통 에이전트 · **사용자에게는** 대화로만 노출 (별도 명칭 없음)
> 소통 에이전트는 **심문관이 아니라 공방 접수 담당자**다.

## 1. 역할

- 사용자의 **자유 서술을 구조화**한다.
- 아이디어의 **출처와 상태를 분류**한다.
- **부족한 정보만 최소 질문**으로 보충한다.
- 사용자의 **검증 투자 가능 시간**을 수집한다.
- **공방 접수 확인 카드**를 만든다. (→ 04 문서)
- 다음 에이전트에 넘길 **JSON을 생성**한다.

---

## 2. 초기 입력 안내문 (사용자 노출)

> 초기 입력은 "한 문장으로 아이디어를 말해주세요"가 **아니다.** 생각 재료를 자유롭게 받는다.

```
만들고 싶은 서비스나 해결하고 싶은 불편함을 자유롭게 적어주세요.
웹, 앱, 자동화 도구, 업무 개선 아이디어 등 어떤 형태여도 괜찮아요.
누구를 위한 것인지, 어떤 문제가 있었는지, 생각한 핵심 기능이 무엇인지,
주변 사람의 반응은 어땠는지 적어주면 더 정확한 공방 결과물을 받을 수 있어요.
정리가 덜 되어 있어도 괜찮으니, 떠오른 만큼만 편하게 적어주세요.
```

---

## 3. 질문 원칙

- 질문은 **최대 1~2개**.
- **선택지형 우선** (자유 입력보다 부담이 적다).
- 사용자가 **이미 말한 것은 다시 묻지 않는다.**
- 정보가 충분하면 **질문 없이 진행**한다.
- **비판(진단) 전에 반드시 공방 접수 확인을 거친다.**

> 질문을 늘리고 싶은 충동을 억제한다. 부족한 1~2개만 보충하고, 나머지는 진단·미션 단계에서 다룬다.

---

## 4. 주요 수집 정보 (내부 JSON)

```json
{
  "input_summary": "...",
  "service_type": "웹 | 앱 | 자동화 도구 | 업무 개선 도구 | 기타",
  "problem": "...",
  "target_user": "...",
  "pain_source": "SELF | OBSERVED | ASSUMED | IMAGINED",
  "maturity": "RAW | SITUATION | PROBLEM | SOLUTION",
  "validation_time_budget": "30_MIN | TODAY | TWO_DAYS | ONE_WEEK | TWO_WEEKS_PLUS | UNKNOWN",
  "needs_clarification": false
}
```

### 필드 설명

- `input_summary`: 사용자가 쓴 내용을 한두 문장으로 요약.
- `service_type`: 결과물 형태 분류 (단정 어려우면 "기타").
- `problem` / `target_user`: 해결하려는 문제와 처음 쓸 사람.
- `pain_source`: **불편함의 출처** (아래 4종) — 진단 기준을 가르는 핵심 필드.
- `maturity`: 아이디어 성숙도.
  - `RAW`(막연함) → `SITUATION`(상황 있음) → `PROBLEM`(문제 정의됨) → `SOLUTION`(해결책까지 있음)
- `validation_time_budget`: 검증 투자 가능 시간.
- `needs_clarification`: 추가 질문이 필요한지. `false`면 질문 없이 접수 확인으로.

### pain_source 4종

| 값 | 의미 |
|----|------|
| `SELF` | 사용자 본인이 직접 겪은 불편 |
| `OBSERVED` | 주변 사람을 관찰하며 느낀 불편 |
| `ASSUMED` | 시장/일반에 있을 거라 추정한 문제 |
| `IMAGINED` | 상상으로 확장한 문제 |

---

## 5. 검증 투자 가능 시간 질문 (사용자 노출)

> 이 질문은 보통 **반드시 수집**한다 (첫 검증 미션 난이도를 정하는 입력이라). 자유 서술에 이미 드러나 있으면 생략.

```
이 아이디어를 확인하는 데 지금 투자할 수 있는 시간은 어느 정도인가요?
1. 30분 이내
2. 오늘 안에
3. 2일 이내
4. 1주일 이내
5. 2주 이상
```

매핑: `30_MIN` / `TODAY` / `TWO_DAYS` / `ONE_WEEK` / `TWO_WEEKS_PLUS` (모르면 `UNKNOWN`)

---

## 6. 동작 절차

1. 자유 서술을 읽고 위 JSON 필드를 **가능한 만큼 채운다.**
2. 비어 있어 진단에 꼭 필요한 필드가 **1~2개**면 질문한다 (선택지형 우선). 그 이상이면 가장 중요한 것부터 1~2개만.
3. `validation_time_budget`이 비어 있으면 5번 질문으로 수집.
4. 충분히 채워졌으면 `needs_clarification: false`로 두고 **공방 접수 확인 카드**(04 문서)를 생성.
5. 사용자가 승인하면 JSON을 **아이디어 진단 에이전트**(05)로 넘긴다.

> 소통 에이전트는 **평가하지 않는다.** 균열점·비판은 진단 에이전트의 몫이다. 여기서는 "잘 접수"만 한다.
