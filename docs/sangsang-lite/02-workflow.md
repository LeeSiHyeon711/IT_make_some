# 02 · 전체 워크플로 (Workflow)

## 1. 실행 방식 — MVP는 순차 실행

> **MVP에서는 병렬 실행을 제외하고 순차 실행으로 구현한다.**
> 병렬 실행 / 오케스트레이션은 **후속 고도화 단계**로 남긴다.

각 단계는 앞 단계의 출력(JSON 또는 확인된 텍스트)을 입력으로 받아 **순서대로** 실행된다.
동시에 여러 에이전트를 돌리지 않는다 — MVP는 단순함과 예측 가능성을 우선한다.

---

## 2. 전체 흐름

```
[1] 아이디어 자유 서술
        │  (사용자가 떠오른 만큼 자유롭게 입력)
        ▼
[2] 소통 에이전트
        │  - 자유 서술에서 핵심 정보 추출
        │  - 부족한 정보만 최대 1~2개 질문
        │  - 검증 투자 가능 시간 수집
        ▼
[3] 공방 접수 확인              ◀── ▣ 사람 승인 게이트 (유일한 게이트)
        │  - 이해한 내용을 사용자 언어로 요약
        │  - 승인 또는 수정
        ▼
[4] 아이디어 진단 에이전트
        │  - 균열점 진단 (1개)
        │  - 착각 가능성 (최대 2개)
        │  - 좋은 신호 (최대 2개)
        ▼
[5] 첫 실험 설계 에이전트
        │  - 균열점을 확인할 최소 검증 미션
        │  - 사용자 시간 예산에 맞춰 난이도 조정
        ▼
[6] 출력 에이전트
        │  - 공방 결과카드 작성 (편집만, 재분석 없음)
        ▼
   🧾 공방 결과카드
```

---

## 3. 단계별 입력 / 출력 계약

| 단계 | 입력 | 출력 |
|------|------|------|
| [1] 자유 서술 | 사용자 자연어 | 원문 텍스트 |
| [2] 소통 에이전트 | 원문 + (질문 답변) | 접수 JSON (`input_summary`, `service_type`, `pain_source` 등) |
| [3] 공방 접수 확인 | 접수 JSON | 사용자 승인/수정 결과 |
| [4] 아이디어 진단 | 승인된 접수 JSON | 진단 JSON (`crack_point`, `misread_risks`, `positive_signals`) |
| [5] 첫 실험 설계 | 접수 JSON + 진단 JSON + 시간 예산 | 실험 JSON (`first_experiment`) |
| [6] 출력 에이전트 | 위 모든 JSON | 공방 결과카드 (사용자 노출 텍스트) |

> **JSON은 내부 통신용**이다. 사용자에게는 [3] 공방 접수 확인 카드와 [6] 공방 결과카드만 노출된다.

---

## 4. 게이트 정책

- **사람 승인 게이트는 [3] 공방 접수 확인 한 곳뿐**이다.
- AI가 아이디어를 잘못 이해한 채 진단에 들어가는 것을 막고, 이후 균열점 진단에 대한 반발감을 줄인다.
- [4]~[6]은 접수 확인 이후 **자동으로** 흐른다 (MVP 순차 실행).

---

## 5. MVP 범위 vs 후속 고도화

### MVP 범위 (이번에 구현)

- 6단계 **순차** 파이프라인
- 소통 에이전트의 **최소 질문(1~2개)** + 시간 예산 수집
- 공방 접수 확인 승인 게이트
- 균열점 **1개** 진단
- 시간 예산 기반 첫 검증 미션
- 공방 결과카드 출력

### 후속 고도화 후보 (이번엔 제외)

- **병렬 실행 / 오케스트레이션** (여러 진단 관점 동시 실행 등)
- 진단 관점 자동 분기 고도화 (`diagnosis_focus` 다중)
- 본 공방(IT상상공방) 1단계로의 자동 연결
- 검증 결과 재투입(미션 실패 → 재진단 루프)
- 사용자 히스토리/세션 저장
