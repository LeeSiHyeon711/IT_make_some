# IT상상공방 설계도 (Architecture)

공방의 멀티에이전트 오케스트레이션 구조를 정의한다. 이 문서는 **에이전트 구현의 청사진**이다.
용어·규칙은 루트 `CLAUDE.md`(공방 헌법)를 따른다.

---

## 1. 오케스트레이션 원칙

이 공방은 **공방장(오케스트레이터) → 단계별 전담 에이전트 → 스킬** 의 3계층 구조를 쓴다.

- **공방장**: 라인 전체를 진행. 단계 간 인계, 휴먼 게이트, 다음 에이전트 호출을 담당. (커맨드로 구현)
- **전담 에이전트**: 한 단계만 책임. 자기 단계 폴더에서만 작업하고, 정해진 산출물만 남긴다.
- **스킬**: 여러 에이전트가 공유하는 재사용 기능 (예: 템플릿 채우기, 산출물 검증).

> 핵심: 에이전트는 다른 에이전트를 직접 호출하지 않는다. **인계는 항상 공방장을 거친다.**
> (산출물 파일 = 단계 간 유일한 통신 수단)

## 2. 전체 흐름도

```
 고객요청   ── 게이트 구간 (방향 결정) ──        ── 자동 구간 (기계적 실행) ──
   │   ┌────┐ ▣ ┌────┐ ▣ ┌────┐ ▣      ┌──────┐   ┌────┐   ┌──────┐
   └──▶│1상담│──▶│2기획│──▶│3설계│═════▶│4이슈등록│─▶│5개발│─▶│6자동QA│
       └─┬──┘   └─┬──┘   └─┬──┘ 설계승인 └──┬───┘   └─┬──┘   └──┬───┘
         ▼        ▼        ▼      =개발착수   ▼        ▼        │
     요구사항    PRD      설계서   /개발착수  이슈등록  코드      ▼
                                          내역     +빌드노트  ┌──────────────┐
                                                             │▣사람 수동테스트│
                                                             └──┬────────┬──┘
                                              증상보고 /증상 ◀──┘        │ 이상없음
                                                  ▼                      │ /납품
                                         ┌──────────────────┐           ▼
                                         │↻증상분석→이슈→수정 │       ┌────┐
                                         │  →자동재QA (순환)  │──────▶│7납품│──▶고객
                                         └──────────────────┘       └────┘
```

- `▣` = 휴먼 게이트. **게이트는 세 곳**(v2): ①방향 결정(상담·기획·**설계+FEAT** 각 단계), ②**QA 진입 승인**(Git 원격 반영 확인 후 `/승인`), ③실사용 검증(사람 수동 테스트).
- **3단계 설계+FEAT 승인 = 개발 착수 승인.** `/개발착수` 입력 시 4→5(개발)→**Git 반영 검증**까지 자동으로 흐르고, **QA 진입 게이트에서 멈춘다**. 사람이 `/승인` 하면 6(자동 QA)로 간다.
- v2 핵심: **이슈 #N ↔ FEAT-NN 문서 1:1 매칭.** builder는 현재 이슈 + 그 FEAT 문서 + 헌법, **3개만** 읽는다(최소 문서). repo-manager는 이슈 close 전 **commit/push를 먼저** 해 v1의 "원격 미반영" 빈틈을 막는다.
- 자동 QA·수동 테스트에서 나온 문제는 **증상분석(triage)→이슈등록(repo-manager)→자동수정(builder)→자동재QA(reviewer)** 순환으로 자동 수렴한다.
- GitHub 관리자(repo-manager)는 자동 구간 전반에서 호출되어 commit/push·이슈 상태를 동기화한다(close/재오픈).

## 3. 에이전트 명세 (구현 청사진)

각 에이전트는 작업 전 `CLAUDE.md` + 자기 단계 `rules` 파일을 읽는다. 도구는 **최소 권한**으로 제한한다.

### 1단계 · 상담가 (consultant)
- **역할**: 고객의 최초 요청을 듣고, 모호한 부분을 질문해 요구사항을 구조화한다.
- **입력**: 고객의 자연어 요청
- **출력**: `01-상담/요구사항-정의서.md`
- **도구 범위**: 질문(대화), `Write`(자기 폴더만), 템플릿 스킬. **코드 작성·웹 빌드 금지.**
- **핵심 행동**: 형태(웹/앱/자동화)를 단정하지 말 것. "무엇을/누구를 위해/왜"에 집중.

### 2단계 · 기획자 (planner)
- **역할**: 요구사항정의서를 제품 명세(PRD)로 구체화한다. 기능 목록·우선순위·MVP 범위 확정.
- **입력**: `01-상담/요구사항-정의서.md`
- **출력**: `02-기획/PRD.md`
- **도구 범위**: `Read`(상담 산출물), `Write`(자기 폴더만), 템플릿 스킬. **설계·코드 금지.**
- **핵심 행동**: "MVP에 꼭 필요한 것"과 "나중 것"을 명확히 구분(스코프 컷).

### 3단계 · 설계자 (architect) ★ Opus 유지
- **역할**: **결과물 형태(웹/앱/자동화)와 기술 스택·구조를 결정**하고, **이슈별 FEAT 문서**를 만든다. 공방에서 가장 중요한 분기점.
- **입력**: `02-기획/PRD.md`
- **출력**: `03-설계/설계서.md` + `03-설계/features/FEAT-NN-*.md`(이슈 1개당 1개, 번호 일치)
- **도구 범위**: `Read`(기획 산출물), `Write`(자기 폴더만), 문서 조회(Context7 등). **실제 빌드 금지.**
- **핵심 행동**: "가장 빠르게 보여줄 수 있는 형태"를 우선. FEAT 문서는 builder가 **하나만 읽고 구현**할 만큼 구체적으로(함수·클래스·수정 파일까지). 이슈 개수는 무작정 쪼개지 말 것(소형 5~8/중형 8~12/복잡 12~15).
- **★ v2 — 왜 Opus 유지**: 무게중심이 설계로 이동. FEAT 품질이 곧 저가형 builder의 결과 품질이므로 여기에 투자한다.

### 4단계 · GitHub 관리자 (repo-manager) ★ 흐름 영속화
- **역할**: 승인된 설계서를 **추적 가능한 GitHub 이슈로 분해·등록**하고, 이후 이슈 상태를 동기화한다.
  설계자만 알던 전체 방향을 이슈로 박제해, 중간에 끊겨도 새 에이전트가 열린 이슈만 보고 이어받게 한다.
- **입력**: `03-설계/설계서.md`
- **출력**: `04-깃허브/이슈등록내역.md`(등록 이슈 목록 + 번호/링크 + 마일스톤) + 실제 GitHub 이슈
- **도구 범위**: `Read`(설계 산출물), `Write`(자기 폴더만), **`Bash`의 `gh`/git commit·push (공방 내 유일 보유)**.
- **핵심 행동**: 코드 내용 변경·설계 변경 **절대 금지**. FEAT 문서를 이슈로 "번역"(설계서 복붙 금지, FEAT만 가리킴)하고, 코드를 commit/push하고, 동기화만 한다.
- **★ v2 — Git 반영 검증 (가장 큰 변경)**: builder 완료 보고 → **commit/push 먼저(`git commit -m "#N …"` → `git push`) → 원격 확인 → 그 다음 이슈 close.** v1의 "이슈는 닫혔는데 코드는 원격에 없음" 빈틈을 막는다.
- **★ v2 — QA 진입 체크리스트**: 모든 이슈 close 후 `git status`/`log`/`remote` 확인 → 헌법 2장 9 체크리스트 출력 → 사람 `/승인` 전까지 reviewer 미호출.
- **생애주기 재호출**: 5단계 완료 → commit/push/close / 6단계 이슈 발견 → 재오픈·신규 등록(다음 FEAT 번호).
- **repo 정책(확정)**: **고객 프로젝트마다 별도 repo.** repo명=프로젝트명, 첫 실행 시 없으면 생성 + **개발 디렉토리 git init·remote 연결**, URL을 `이슈등록내역.md`에 기록.
- **FEAT 1:1**: 이슈 번호 = FEAT 번호. 이슈 본문은 `templates/이슈본문.md`(FEAT만 참조).

### 5단계 · 개발자 (builder) ★ v2 Opus→Sonnet
- **역할**: FEAT 문서대로 MVP를 실제로 만든다. (다형적 — 설계가 정한 형태/스택을 따른다)
- **입력(최소 3개)**: ① `CLAUDE.md` ② 현재 GitHub 이슈 본문 ③ 그 이슈의 `FEAT-NN` 문서. **설계서 전체를 습관적으로 읽지 않는다(헌법 2장 8).**
- **출력**: `05-개발/` (코드) + `05-개발/빌드노트.md` + `progress.md` 갱신. commit 가능 상태로 정리(commit/push는 repo-manager).
- **도구 범위**: `Read`, `Write`/`Edit`(자기 폴더만), `Bash`(빌드/설치, **commit/push 훅 차단**), 빌드 스킬.
- **핵심 행동**: FEAT "구현할 것" 범위만. 다른 FEAT 선구현·관련 없는 리팩터링 금지. 진행 상태·변경 파일을 GitHub 관리자에게 보고(직접 commit/close 안 함).
- **★ v2 — Sonnet으로 내린 이유**: FEAT 문서가 구체적이면 일반 구현은 Sonnet으로 충분. v1 최대 토큰 소모처를 줄이는 핵심 레버.

### 6단계 · 검수자 (reviewer) — 자동 QA (★ v2 타입별 분기)
- **역할**: MVP가 요구사항·PRD·**FEAT 완료 조건**을 충족하는지 검증한다. (요구사항 ↔ 결과물 대조)
- **진입 조건(v2)**: 모든 이슈 close + Git push 확인 + 사람 `/승인` 완료 상태에서만 시작. 미충족(특히 push 미반영)이면 멈추고 보고.
- **입력**: `05-개발/` + 1~4단계 문서 전체
- **출력**: `06-검수/검수리포트.md` (충족 체크리스트, 발견 이슈, 통과/보류 판정)
- **도구 범위**: `Read`(전 단계), `Bash`(실행 검증, **push/gh 훅 차단**), **Playwright MCP `mcp__playwright__*`**(웹 실제 브라우저). **코드 수정 금지(보고만).**
- **★ v2 — 프로젝트 타입별 QA 전략**:
  - **웹**: `file://` 실제 브라우저(Playwright) 렌더링/콘솔/플로우, 서버형은 서버+API.
  - **Android 앱**: APK 빌드·파일 위치/크기, 설치·권한 안내, **에뮬레이터 불가 기능은 "실기기 검증 필요" 명시**(특수 권한은 별도 체크리스트).
  - **API/Backend**: endpoint 테스트·request/response·에러·DB seed.
  - **문서/자동화**: 산출물 존재·입출력 예시·재실행 가능성.
- 직접 고치지 않는다. 보류면 공방장이 자동 수정 루프로 돌린다.

### ↻ 증상분석가 (triage) — 수정 순환
- **역할**: 사람 수동 테스트에서 나온 **자연어 증상**을 재현·분석해 **수정 이슈 초안**으로 번역한다.
- **입력**: 사람의 증상 보고 + `05-개발/` + 1~4단계 문서
- **출력**: `06-검수/증상분석log.md` (증상·재현결과·추정원인·이슈초안 누적)
- **도구 범위**: `Read`, `Write`(자기 폴더만), `Bash`(재현, **push/gh 훅 차단**), **Playwright MCP `mcp__playwright__*`**(웹 재현).
- **핵심 행동**: 재현은 **고객과 동일한 방식**으로. **이슈 등록·코드 수정 금지** — 분석만 하고 repo-manager가 등록, builder가 수정.

### 7단계 · 납품담당 (delivery) ★ v2 증거·생산성 기록
- **역할**: 데모 실행법·고객 안내문 + **포트폴리오 증거 목록·생산성 요약**을 패키징한다.
- **입력**: 전 단계 전체 + 단계별 토큰/시간 기록(`생산성요약.md`)
- **출력**: `07-납품/실행안내.md` + `07-납품/고객전달문.md` + `07-납품/증거자료체크리스트.md` + 루트 `생산성요약.md`(최종)
- **도구 범위**: `Read`(전 단계), `Write`(자기 폴더 + 루트 생산성요약), 템플릿 스킬. **코드 수정 금지.**
- **핵심 행동**: 비개발자 언어로 작성. 증거 체크리스트 자동 생성(타입 안 맞으면 "해당 없음"), 토큰/시간은 모르면 "측정 불가"(거짓 수치 금지).

## 4. 공방장(오케스트레이터) 명세

커맨드로 구현한다. 게이트 구간은 한 단계씩, 자동 구간은 한 번에 흐른다.

| 커맨드 | 구간 | 동작 |
|--------|------|------|
| `/신규프로젝트 <이름>` | 게이트 | `_template` 복제 → 1단계 상담가 호출 |
| `/다음단계 <이름>` | 게이트 | 방향 결정 단계 1→2→3(설계+FEAT) 한 단계 진행 후 정지 |
| `/개발착수 <이름>` | **자동→게이트** | 4 이슈등록(FEAT 1:1)→5 개발(전 이슈, commit/push)→**Git 반영 검증** 후 **QA 진입 게이트에서 대기** |
| `/승인 <이름>` | 게이트 후 | Git 원격 반영 확인 → 6 자동QA(타입별) 시작 → 수동 테스트에서 대기 |
| `/개발재개 <이름>` | **자동** | progress.md + open issue + git 상태로 남은 이슈부터 개발 재개 → Git 검증 후 대기 |
| `/증상 <이름> "증상"` | **자동** | triage 분석→repo-manager 이슈→builder 수정→repo-manager commit/push/close→reviewer 재QA 순환 |
| `/납품 <이름>` | 게이트 후 | 수동 테스트 통과 신호 → 7단계 납품(+증거 목록·생산성 요약) |
| `/되돌리기 <이름> <단계>` | 게이트 | 방향이 틀렸을 때 이전 방향 단계로 되돌림 (버그는 `/증상`) |
| `/이슈동기화 <이름>` | 보조 | GitHub 관리자 재호출 → commit/push·이슈 상태 동기화 |

공방장은 **직접 산출물을 만들지 않는다.** 진행 관리와 에이전트 호출만 한다.

## 5. 범위 강제 메커니즘 (어떻게 "범위 내에서만" 일하게 하는가)

1. **CLAUDE.md** — 모든 에이전트가 읽는 공통 헌법 (2장 절대 원칙).
2. **에이전트 정의의 `tools` 허용목록** — 단계에 필요한 도구만 부여 (예: 상담가는 빌드 도구 없음).
3. **`.claude/rules/<단계>.md` (paths 스코프)** — 해당 폴더를 만질 때만 lazy-load 되는 단계별 세부 규칙.
4. **핸드오프 계약** — 입력/출력 파일 위치 고정. 폴더 밖 쓰기 금지.
5. **권한 훅 + settings** — builder·reviewer·triage는 `git push`/`gh` 가 PreToolUse 훅으로 **물리 차단**(`block-github-write.py`).
   GitHub 쓰기는 repo-manager만 가능. `.claude/settings.json` 이 force-push·repo 삭제 등을 전역 deny.

이 5중 장치로 "각 에이전트가 정해진 범위에서만 작업"하는 것을 보장한다.

### v2 추가 — 토큰·반영 보장 장치
6. **FEAT 1:1 + 최소 문서**: builder가 현재 이슈 + FEAT 문서 + 헌법만 읽어 컨텍스트를 작게 유지(토큰↓ + 범위 이탈↓ — 안 읽은 문서는 구현 못 함).
7. **commit/push 선행 → close 후행**: repo-manager가 코드를 원격에 반영한 *뒤에* 이슈를 닫아, "닫힌 이슈 = push된 commit"을 보장(v1 빈틈 차단).
8. **QA 진입 게이트(`/승인`)**: Git 원격 반영을 사람이 확인하기 전엔 자동 QA로 못 넘어간다.
