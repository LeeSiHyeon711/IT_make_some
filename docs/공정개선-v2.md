# IT상상공방 공정 개선 — v2

PickUpMemo v1 실기기 검증 사이클에서 확인된 병목·빈틈을 바탕으로 공정을 개선한 기록.
이번 변경의 목적은 **새 기능이 아니라 공정 최적화** — 안정성·토큰 효율·추적성을 높인다.

---

## 1. v1에서 확인된 문제

| # | 문제 | v2 해법 |
|---|------|---------|
| 1 | Builder 토큰 소모가 컸음 | builder Opus→**Sonnet** + **최소 문서**(이슈+FEAT+헌법 3개)만 읽기 |
| 2 | 이슈마다 참조 문서 범위가 넓어 탐색 비용 발생 | **FEAT 문서 ↔ 이슈 1:1 매칭**, 이슈 본문은 FEAT만 가리킴(설계서 복붙 금지) |
| 3 | 이슈는 닫혔는데 코드가 원격에 push되지 않은 빈틈 | repo-manager **commit/push 먼저 → 이슈 close 나중** + QA 진입 Git 검증 게이트 |
| 4 | Android와 웹의 QA 방식이 같았음 | reviewer **프로젝트 타입별 QA 분기**(웹/앱/API/문서) |

| 6 | 토큰/시간/중단 기록이 체계 없음 | **생산성요약.md** + **progress.md**(중단/재개) 정식화 |

## 2. 핵심 구조 변경

### 2-1. FEAT 문서 ↔ 이슈 1:1 매칭
- 설계자가 작업 분해를 **이슈 1개당 FEAT 문서 1개**로 펼쳐 쓴다: `03-설계/features/FEAT-NN-*.md`.
- **이슈 번호 = FEAT 번호.** (Issue #7 ↔ FEAT-07)
- 이슈 본문(`templates/이슈본문.md`)은 설계서 전체를 복붙하지 않고 매칭 FEAT 문서만 참조.
- builder는 **현재 이슈 + 그 FEAT 문서 + 헌법, 3개만** 읽는다(헌법 2장 8). 추가 문서는 이유를 기록하고 확인 요청.

### 2-2. 이슈 개수 최적화
- 무작정 잘게 쪼개지 않는다(v1은 호출 횟수·문서 재탐색 비용↑).
- 권장: **소형 5~8 / 중형 8~12 / 복잡 12~15**.
- 묶기 OK: 데이터 클래스+저장소+Repository → "저장 계층 구현" 1개.
- 섞기 금지: UI와 DB 대규모 변경 / 권한과 비즈니스 로직 / 서버 연동과 화면 / 핵심 로직과 디자인.

### 2-3. Git commit/push 검증 게이트
- repo-manager(유일한 Git 쓰기 에이전트)가 이슈마다: `git add -A` → `commit -m "#N …"` → `push` → 원격 확인 → **그 다음** `gh issue close`.
- 모든 이슈 close 후 **QA 진입 전 체크리스트**(헌법 2장 9) 출력 → 사람 `/승인` 전까지 자동 QA 미진입.
- 첫 등록 시 개발 디렉토리를 `git init` + remote 연결.

### 2-4. 프로젝트 타입별 QA 분기
- **웹**: `file://` 실제 브라우저(Playwright) / 서버형은 서버+API.
- **Android 앱**: APK 빌드·위치/크기, 에뮬레이터 불가 기능은 "실기기 검증 필요" 명시(특수 권한 별도 체크리스트).
- **API/Backend**: endpoint·request/response·에러·DB seed.
- **문서/자동화**: 산출물·입출력 예시·재실행 가능성.

### 2-5. 증거자료 + 생산성 기록
- delivery가 `증거자료체크리스트.md`(공정/개발/실행/납품 증거)를 자동 생성.
- 각 단계가 `생산성요약.md`에 토큰/시간을 누적, 납품 시 최종 요약. 측정 불가는 정직하게 표기.

### 2-6. 중단/재개 프로토콜
- `progress.md`(완료/진행/남은 이슈, 마지막 commit/push, 주의사항)를 개발 중 갱신.
- `/개발재개 <프로젝트>` 가 progress.md + GitHub open issue + git 상태로 남은 이슈부터 이어간다.

### 2-7. 모델 배치
- **Opus**: architect, triage (무거운 판단).
- **Sonnet**: consultant·planner·repo-manager·builder·reviewer·delivery.
- **Haiku(실험)**: 체크리스트 검수·로그 요약·증거 목록.
- ★ builder를 내리는 대신 **FEAT 승인 게이트 강화** — 싼 모델일수록 FEAT 품질이 결과 품질.

## 4. 변경된 흐름

```
상담 ─▣─ 기획 ─▣─ 설계+FEAT ─▣─ 이슈등록(FEAT 1:1)
  → 개발 루프[ builder(이슈1개·최소문서) → repo-manager(commit/push→close) ]
  → Git 반영 검증 ─▣(/승인)─ 자동QA(타입별) ─▣(수동테스트)─ 납품(+증거·생산성)
                                         ↘ 증상 → triage → 수정 → commit/push → 재QA (순환)
```

## 6. 다음 프로젝트에서 달라지는 것 (운영 관점)

1. 설계 단계에서 **설계서 + FEAT 문서 N개**가 함께 나오고, 사람이 둘 다 승인한다.
2. `/개발착수` 는 개발+Git push까지만 진행하고 **`/승인` 에서 멈춘다.** GitHub 웹에서 코드 확인 후 승인.
3. 자동 QA가 **프로젝트 타입에 맞는 방식**으로 검증한다(앱은 실기기 필요 항목을 따로 표시).
4. 납품 시 **포트폴리오 증거 목록 + 생산성 요약**이 자동으로 함께 나온다.
5. 끊기면 `/개발재개` 로 progress.md 기준 복원.
