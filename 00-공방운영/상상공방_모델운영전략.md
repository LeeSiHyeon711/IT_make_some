## 기본 원칙
- **무거운 판단은 앞단(설계)으로 몰고, 반복 실행은 싸게.** 설계·FEAT 품질이 높을수록 구현(Builder)은 경량 모델로도 가능해진다.
- 모델 등급은 **역할의 추론 난이도**로 정한다. "똑똑한 모델을 모두에게"가 아니라 "필요한 곳에만 고급".

## 고급(상위) 모델을 권장하는 역할
판단·해석·검증 품질이 결과를 좌우하는 단계:
- **상담가** — 모호한 요구를 정확히 구조화
- **기획자** — 범위·우선순위 판단
- **설계자** — 형태·스택·FEAT 분해(가장 무거운 의사결정)
- **QA(검수)** — "로직은 맞는데 실제로 안 도는" 것까지 잡아내는 검증
- **최종 리뷰/공방장 판단** — 방향·위험 결정

## 경량(저비용) 모델 실험 후보
- **Builder** — 설계·FEAT가 충분히 명확하면 구현은 "옮겨 적기"에 가깝다. (단, 아래 v3 분석 참조 — FEAT 난이도에 따라 갈린다.)
- **Git관리자** — 이슈/커밋 메시지/상태 동기화는 정형 작업.

## Haiku 하향 분석 (PickUpMemo v3 근거) — 검토일 2026-06-24
> 현재 실제 배치: 설계자·증상분석 = **Opus**, 나머지 6역할 = **Sonnet**. (헌법 4-1)
> 이번 검토 질문: **Sonnet → Haiku 하향이 어디까지 안전한가.** 결론은 "일괄 하향"이 아니라 "**Git관리자 우선 + Builder는 FEAT 난이도로 라우팅**".

### v3가 주는 판단 근거 (양면)
- **경량화 가능 신호**: v2에서 Builder를 Opus→Sonnet으로 내려 호출당 토큰 보수적 −27% 달성. 설계·FEAT가 명확하면 구현 모델을 낮춰도 됐다.
- **경량화 제동 신호**: v3 Builder 작업에는 *동시성·엣지케이스 엔지니어링*이 있었다 — `route/RouteService`(5초 타임아웃 + 전달지 4단계 geocode fallback + 실패 원인 분류), `overlay/MemoPopupController`(비동기 경로 갱신을 `AtomicLong` 토큰 세대로 경합 방지), `matcher/AddressExtractor`(2소스 fallback). 이런 FEAT는 Haiku에 위험하다.
- **해석**: 같은 Builder라도 **FEAT마다 난이도가 다르다.** 정형(데이터 클래스·CRUD·UI 결선)은 Haiku 후보, 복잡(동시성·외부 API·fallback 체인)은 Sonnet 유지.

### 권장 모델 매핑표 (제안 — 즉시 적용 아님)
| 역할 | 현재 | 권장 | 변경 여부 | 위험도 |
|------|------|------|-----------|--------|
| 상담가(consultant) | Sonnet | Sonnet | 유지 | — |
| 기획자(planner) | Sonnet | Sonnet | 유지 | — |
| 설계자(architect) | **Opus** | **Opus** | 유지(하향 금지) | — |
| **Git관리자(repo-manager)** | Sonnet | **Haiku 실험(1순위)** | **변경 후보** | 중(완화 가능) |
| **Builder(builder)** | Sonnet | **Sonnet 기본 + 단순 FEAT만 Haiku 실험** | **조건부** | 중~상 |
| QA(reviewer) | Sonnet | Sonnet | 유지(**Haiku 금지**) | 상(금지) |
| 증상분석(triage) | Opus | Opus | 유지 | — |
| 납품(delivery) | Sonnet | Sonnet(후순위 Haiku 실험) | 보류 | 저 |
| (작업 단위) 체크리스트 검수·로그 요약·증거목록 | — | **Haiku** | 헌법 4-1이 이미 허용 | 저 |

### 위험도 분석
- **Git관리자→Haiku**: 커밋 메시지 부정확 / diff 요약 누락 / **secrets 오탐·누락**. 완화 — OPS-01의 commit·push 전 7항목 보고를 체크리스트화, push·close는 어차피 **사람 승인(L1.5)**이라 구조적 완충. → ROI 최고(빈도 높고 가벼움).
- **Builder→Haiku(복잡 FEAT)**: 동시성/엣지케이스 버그 → **회귀·재작업 증가**(v3가 근거). 완화 — FEAT 난이도 라우팅, QA는 Sonnet로 강하게 유지.
- **QA→Haiku**: 치명적(놓친 버그가 납품까지). **하지 않는다.**

### 토큰 절감 효과 (방향만, 실측 필요)
- Haiku는 Sonnet 대비 **토큰당 단가가 크게 낮다**(정확 배수는 당시 단가표 기준 별도 산정).
- Git관리자: 호출 빈도 높음 → 누적 절감 큼·리스크 낮음. Builder: 호출당 토큰 최대(v2 평균 42.9k) → 정형 FEAT 한정 시 절감폭 최대(품질 리스크 동반). **수치는 v3 전환 실험에서 측정해 확정**(현재 v3 토큰 데이터 없음 — 과장 금지).

## PickUpMemo에서 얻은 교훈 (측정 근거)
- **v1 → v2 실측**: Builder 모델을 **Opus → Sonnet**으로 내리고 **FEAT 1:1 매칭 + 최소 문서 읽기 + 이슈 압축**을 적용한 결과, **Builder 호출당 평균 토큰이 보수적 기준 약 −27%** 감소(`docs/공정개선-v2.md` / `docs/PickUpMemo_v1_v2_생산성비교.md`).
- 해석: **설계와 FEAT 문서가 충분히 명확하면 Builder는 저비용/경량 모델로도 구현 가능성이 있다.**
- **PickUpMemo v3(진행 중)**: 경로 거리·시간 통합 버전업. 이 가설을 추가 검증하기 좋은 대상이다. *단, 위 −27% 수치는 어디까지나 **v2 실측**이며 v3의 토큰/품질 데이터는 아직 집계 전이다 — v3 모델 실험 시 위 "기록할 지표"를 새로 측정해 별도 기록한다.*

## 경량화하지 않을 것 (우선)
- **QA는 경량화하지 않는 것을 우선 권장.** 검증을 싸게 만들면 "통과했는데 실제로는 안 되는" 위험이 커진다. 비용 절감의 마지막 후보로 둔다.

## 적용 방식 — 단계적 전환 (권장 순서)
1. **즉시 전면 교체 금지.** 한 번에 한 역할만.
2. **1단계 — Git관리자 Haiku 실험**(1순위, 저위험): 1~2 프로젝트 관찰 + 기계적 작업(체크리스트·로그·증거목록) Haiku 적용.
   - ✅ **적용됨 (2026-06-24)**: `.claude/agents/repo-manager.md` `model: sonnet → haiku`. 모니터링(커밋메시지·diff요약·**secrets 점검 오탐/누락**·이슈 정합성) 후 다음 1~2 프로젝트에서 평가. **롤백 기준**: 재작업 반복 또는 secrets 점검 누락 1회 → 즉시 `sonnet` 복귀.
3. **2단계 — FEAT 난이도 태그 도입**: 설계자가 FEAT에 `단순/복잡`을 표기 → **단순 FEAT만 Builder Haiku 실험**, 복잡 FEAT는 Sonnet.
4. **3단계 — 확대 판단**: 품질·회귀율·재작업이 임계 내면 확대. **QA·설계자·상담가·기획자는 유지.**
5. 각 전환은 같은 종류 작업으로 **A/B 비교**, **롤백 임계(회귀율·재작업 횟수)를 사전 정의**하고 초과 시 즉시 복귀.

## 운영규칙 반영 필요사항 (적용 시 — 이번엔 제안만)
> 아래는 **결정·검증 후** 반영할 대상. 이번 단계에서는 `.claude/agents/*.md`의 `model` 필드나 헌법을 바꾸지 않는다.
- `.claude/agents/repo-manager.md` — Haiku 실험 시 `model: haiku`로 변경(1순위 대상). secrets 점검·커밋메시지 품질 모니터링 문구 추가.
- `.claude/agents/builder.md` — 기본 `sonnet` 유지. "단순 FEAT 한정 Haiku 실험" 라우팅 규칙은 **설계자가 FEAT에 난이도 태그**를 남기는 것과 연동.
- `templates/FEAT.md` — FEAT에 **난이도(단순/복잡)** 필드 추가(Builder 모델 라우팅 근거).
- `CLAUDE.md` 4-1 — 현재 "Haiku 실험 가능"은 이미 있음. repo-manager 1순위·Builder 난이도 라우팅·**QA Haiku 금지**를 명시적으로 보강.
- `.claude/agents/reviewer.md` — QA는 Haiku 하향 금지 명문화(유지).
- 측정·롤백: 모델 변경 결과를 `projects/<프로젝트>/생산성요약.md`에 품질·회귀율·재작업·토큰으로 기록.

## 모델 변경 시 반드시 기록할 것
| 지표 | 의미 |
|------|------|
| 결과 품질 | 산출물이 수용 기준을 충족하는가 |

| 재작업 횟수 | 같은 이슈를 다시 손본 횟수 |
