# 상상공방 새 세션용 요약 (붙여넣기용)

> 작성일: 2026-06-24
> 문서 목적: 새 ChatGPT/Claude/Codex/Cursor 세션에 **그대로 붙여넣어** IT상상공방을 1~2페이지로 설명하기 위한 압축본. 상세는 [온보딩](IT상상공방_온보딩.md) 등 세트 문서 참조.

---

아래 블록을 새 AI 세션에 붙여넣으세요. ▼

```text
[IT상상공방 — 운영 요약]

■ 정의
고객/본인의 막연한 아이디어를, 실제로 확인 가능한 MVP로 바꿔 주는 AI 멀티에이전트 생산 공정.
철학: "상상을 실제 확인 가능한 형태로." 핵심은 프롬프트가 아니라 요구사항 분해·권한 설계·문서 영속화·테스트 게이트·실사용 피드백 루프.

■ 작업 흐름 (7단계)
상담가 → 기획자 → 설계자 → Git관리자 → Builder → QA → (사람 수동 테스트) → 인수인계
산출물 사슬: 아이디어 → 요구사항정의서 → PRD → 설계서 → FEAT(작업1개=FEAT1개=이슈1개) → 코드 → 검수리포트 → 인수인계 문서.
원칙: "바로 구현 금지, 먼저 PLAN/문서화." 한 번에 이슈 하나만 처리하고 상태를 남긴다(끊겨도 재개).

■ 역할 구조
- 상담가: 문제/요구 파악       - 기획자: PRD 작성
- 설계자: 기술 설계/FEAT 분해   - Git관리자: 이슈/커밋/상태 관리(계획·문서)
- Builder: FEAT 단위 구현      - QA: 회귀·수용 기준 검증(코드 직접 수정 안 함)
- 공방장: 최종 승인/방향/위험 판단   - 납품담당: 인수인계 문서
각 역할은 자기 산출물만 만들고, 인계는 '산출물 파일'로만 한다.
* Git관리자는 실제 권한이 없을 수 있다 → 이슈 등록/close/commit/push는 '제안·문서화'까지, 실제 실행은 권한 있는 주체(공방장/사람)가 한다.

■ 휴먼 게이트(사람이 꼭 개입)
① 방향 결정(상담·기획·설계)  ② Git 반영/QA 진입 승인  ③ 실사용 수동 테스트.
그 사이의 기계적 실행(이슈등록·개발·QA·버그수정)은 자동.

■ 문서 보존 원칙
기존 문서 삭제·덮어쓰기 금지. 버전업은 새 프로젝트가 아니며, 이전 버전 산출물·기능을 보존한 채 변경점만 추가한다.

■ 모델 운영 전략
고급 모델 권장: 상담가·기획자·설계자·QA·최종리뷰.
경량 실험 후보: Builder·Git관리자 (설계/FEAT가 명확하면 구현은 경량 모델로도 가능 — PickUpMemo v1→v2에서 Builder 토큰 보수적 -27% 실측).
QA는 경량화하지 않는 것을 우선. 전면 교체 금지, Builder부터 단계적 실험. 변경 시 품질·회귀율·재작업 횟수 기록.

■ 절대 금지(요약)
승인 전 구현 / QA 없이 close / 비밀키 하드코딩·커밋 / 외부 과금 API 사전확인 없이 사용 /
기존 문서 삭제·덮어쓰기 / 사용자 문제와 무관한 기능 무작정 추가 / 결과 과장.

■ 현재 주요 사례
- PickUpMemo (Android): 배민커넥트 신규 배차 시 가게 메모 오버레이 팝업(접근성 기반 기술검증).
  v2=메모 매칭→팝업, v3(진행 중)=같은 단일 팝업에 픽업지→전달지 경로 거리·예상시간 통합(v2 기능 회귀 없이 확장).
- 아이디어 건강검진 (sangsang-lite, MCP): "만들기 전에 먼저 확인"을 돕는 룰 기반 아이디어 검증 도구.
- AI-Morning-Brief (Python CLI): RSS→SQLite→Claude/fallback→Markdown→Discord, V0.1(정기 04:30 실전검증 대기).

[상세 문서] 00-공방운영/ 폴더의 온보딩·작업흐름·역할정의·모델운영전략·금지사항 참조.
```
