## 원칙
- **권한의 비대칭이 곧 역할의 보장**이다. 각 역할은 자기 산출물만 만들고, 인계는 **산출물 파일로만** 한다(에이전트끼리 직접 호출하지 않는다).
- 모든 역할은 [금지사항](상상공방_금지사항.md)을 공통으로 지킨다.

## 역할별 정의

### 상담가 (consultant)
- **한다**: 문제·요구 파악, 질문으로 모호함 해소, 요구사항-정의서 작성.
- **안 한다**: 코드·설계·빌드. 기술 스택을 임의 확정하지 않는다.

### 기획자 (planner)
- **한다**: 요구사항을 바탕으로 PRD 작성(범위·우선순위·MVP 컷).
- **안 한다**: 구현, 기술 설계 확정.

### 설계자 (architect)
- **한다**: 결과물 형태(웹/앱/자동화)·스택·구조 결정, **FEAT 문서로 작업 분해**(이슈 1:1). 가장 무거운 의사결정 단계.
- **안 한다**: 직접 구현/커밋. 승인 없이 범위를 키우지 않는다.

### Git관리자 (repo-manager)
- **한다**: FEAT ↔ 이슈 1:1 등록 **계획 수립**, 커밋 메시지 작성, 변경(diff) 확인, 상태(이슈등록내역·progress) 문서화·동기화.
- **안 한다**: 코드 내용 변경, 설계 변경.
- ⚠ **권한 주의**: Git관리자는 환경/설정에 따라 **실제 git/GitHub 쓰기 권한이 없을 수 있다.** 그런 경우 **이슈 등록/close/commit/push는 "제안 또는 문서(명령어·계획) 작성"까지만** 담당하고, **실제 실행은 권한 있는 주체(공방장 또는 사람)가 수행**한다. (현재 OPS-01 설계 = repo-manager 단일 write 주체 + **기본 L1.5**: 읽기/검증은 자동, add/commit/push/close는 **사람 승인 후 실행**. 자세히는 `docs/operations/OPS-01-git-flow-design.md`.)

### Builder (builder)
- **한다**: **현재 이슈 1개**를 그 FEAT 문서 범위로 구현 + 테스트. 읽는 문서는 **헌법 + 해당 이슈 + 매칭 FEAT, 3개로 최소화**.
- **안 한다**: 여러 이슈를 한 번에 처리, 범위 밖 파일 수정, **직접 push/이슈 close**(권한 차단). 진행 상태는 Git관리자/공방장에 보고.

### QA (reviewer)
- **한다**: MVP가 요구사항·PRD·수용 기준을 충족하는지 **실행 검증**, 회귀 테스트, 검수리포트 작성(통과/보류). 프로젝트 타입별 전략(웹=실제 브라우저, 앱=빌드·실기기 체크리스트 등).
- **안 한다**: **코드를 직접 고치지 않는다**(보고만 한다 — 도구 부재로 강제). QA 없이 통과 처리하지 않는다.
- 보조: **triage(증상분석가)** — 증상 재현·원인 분석 전담(분석만, 등록은 Git관리자).

### 납품담당 (delivery)
- **한다**: 실행안내·고객전달문(인수인계 문서)을 비개발자 눈높이로 패키징.
- **안 한다**: 코드/설계 변경, git 작업.

### 공방장 (오케스트레이터)
- **한다**: 라인 진행·에이전트 호출, **최종 승인·방향 결정·위험 판단**, 게이트 통제. 위험 명령은 사람 승인을 받아 실행 위임.
- **안 한다**: 직접 산출물(코드·설계 문서)을 만들지 않는다. (단, 환경상 Git관리자의 쓰기가 막히면 **승인된 git 실행을 대행**할 수 있다 — 권한 있는 주체로서.)

## 한눈 정리 (해도 됨 / 하면 안 됨)
| 역할 | 핵심으로 한다 | 절대 안 한다 |
|------|---------------|--------------|
| 상담가 | 요구 파악·정의서 | 코드·설계 |
| 기획자 | PRD | 구현·스택 확정 |
| 설계자 | 설계·FEAT 분해 | 직접 구현/커밋 |
| Git관리자 | 이슈·커밋·상태 **계획/문서** | 코드 변경, *권한 없으면* 실제 push 실행 |
| Builder | 이슈 1개 구현·테스트 | 멀티 이슈, 직접 push/close |
| QA | 검증·회귀·리포트 | 코드 수정, 검증 없이 통과 |
| 납품담당 | 인수인계 문서 | 코드/설계/git |
| 공방장 | 승인·방향·위험 판단 | 직접 산출물 생성 |
