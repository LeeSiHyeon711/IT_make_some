# CLAUDE.md — IT상상공방 (IT Make Some) 공방 헌법

이 문서는 IT상상공방에서 일하는 **모든 에이전트가 작업 전 반드시 읽어야 하는 최상위 규칙**이다.
여기에 적힌 규칙은 개별 에이전트의 판단보다 우선한다. 충돌 시 이 문서를 따른다.

---

## 1. 공방이란 무엇인가

IT상상공방은 **고객의 요구사항을 빠르게 확인 가능한 MVP 형태로 만들어주는 AI 자동 생산 공방**이다.

- **목표**: 고객의 상상(아이디어)을 최소 기능 제품(MVP)으로 빠르게 구현해 "이게 맞나요?"를 즉시 검증한다.
- **산출물 형태**: 고정되어 있지 않다. 요구사항에 따라 **웹 / 앱 / 자동화 프로세스** 중 적합한 형태로 만든다.
  형태 결정은 반드시 **3단계(설계)** 에서 이루어진다. 그 이전 단계는 형태를 단정하지 않는다.
- **속도가 품질보다 우선**한다. 완벽한 제품이 아니라 "요구사항이 맞는지 확인할 수 있는 최소 결과물"을 만든다.

---

## 2. 절대 원칙 (모든 에이전트 공통)

1. **범위 엄수**: 각 에이전트는 자신에게 배정된 단계의 작업만 한다. 다른 단계의 산출물을 수정하거나 침범하지 않는다.
2. **계약 기반 인계**: 이전 단계의 산출물(핸드오프 산출물)을 입력으로 받고, 정해진 산출물만 출력한다. (4장 참조)
3. **휴먼 게이트 — 가치 있는 지점에만**: 게이트는 모든 단계가 아니라 **사람만이 줄 수 있는 가치가 있는 세 지점**에만 둔다.
   ① **방향 결정**(상담→기획→설계): 매 단계 사람 승인 필수. 여기서 틀리면 뒤가 다 틀어진다.
   ② **QA 진입 승인**(개발 완료 후): 모든 이슈 close + **Git commit/push 원격 반영 확인** + 빌드 산출물 확인을 사람이 승인해야 자동 QA로 넘어간다. (v2 신설 — 2장 9 참조)
   ③ **실사용 검증**(자동 QA 후 사람 수동 테스트): 자동이 못 잡는 실제 사용 문제를 사람이 확인.
   그 사이 **기계적 실행(이슈등록·개발·자동QA·버그수정)은 게이트 없이 자동**으로 흐른다. 이미 승인된 설계를 옮기는 작업이기 때문이다.
4. **추측 금지, 질문 우선**: 요구사항이 불명확하면 임의로 가정하지 말고 명시적으로 질문한다.
5. **한글로 작성**: 모든 문서·산출물·고객 대면 결과물은 한글로 작성한다. (코드 식별자/명령어는 예외)
6. **프로젝트 격리**: 모든 작업은 `projects/<프로젝트명>/` 안에서만 이루어진다. 공방 공용 파일(`.claude/`, `docs/`, `templates/`)은 임의로 바꾸지 않는다.
7. **작업 원자성 — 1 이슈 = 1 FEAT 문서 = 한 번에 처리 가능한 최소 단위**: 한 번의 작업(호출)에서는 **GitHub 이슈 단 하나만** 처리하고 멈춘다.
   - **1:1 매칭**: 이슈 #N ↔ `03-설계/features/FEAT-NN-*.md` 번호가 반드시 일치한다. 이슈 본문은 설계서 전체를 복붙하지 않고 매칭된 FEAT 문서만 가리킨다.
   - **무작정 쪼개지 않는다**: "한 번의 builder 호출에서 완료 가능하면서 범위가 명확한 최소 단위"로 묶는다. 너무 잘게 쪼개 호출 횟수·문서 재탐색을 늘리지 않는다. 권장 개수 — **소형 5~8 / 중형 8~12 / 복잡 12~15**. (이슈 개수 최적화 = 이 원칙의 일부)
   - 작업이 끊겨도 "닫힌 이슈(완료) + 열린 이슈(남음) + 진행중 1개"로 상태가 명확해, 그 지점부터 재개할 수 있다. 여러 이슈를 한 번에 몰아 처리하지 않는다.
8. **개발 에이전트는 최소 문서만 읽는다**: builder는 ① 공방 헌법(이 문서) ② 현재 처리 중인 GitHub 이슈 ③ 그 이슈와 1:1 매칭된 FEAT 문서, **이 3개만** 읽는 것을 원칙으로 한다. 그 이상이 필요하면 임의로 전체 문서를 탐색하지 말고 **필요한 이유를 먼저 기록**하고 상위(공방장/사용자) 확인을 요청한다. (토큰·탐색 비용 최소화)
9. **Git 반영 없이는 QA·납품으로 넘어가지 않는다**: GitHub 이슈를 닫는 것과 코드가 원격 repo에 push되는 것은 별개다. 모든 개발 이슈 완료 후, repo-manager가 `git status`/`git log`/원격 반영을 검증하고 **QA 진입 전 체크리스트**(아래)를 출력해 사람 승인(`/승인`)을 받기 전까지 자동 QA를 호출하지 않는다.

```
# QA 진입 전 Git 검증 체크리스트
- [ ] 모든 개발 이슈가 closed 상태인가?
- [ ] git status가 clean 상태인가?
- [ ] 최신 commit이 존재하는가?
- [ ] 원격 저장소에 push 되었는가?
- [ ] GitHub 웹에서 실제 파일이 확인되는가?
- [ ] 산출물(APK·빌드 파일 등)이 존재하는가?
- [ ] 사용자가 GitHub 원격 반영을 확인했는가?
```

## 3. 생산 라인 — 표준 7단계

고객 프로젝트는 아래 7단계를 순서대로 통과한다. 각 단계에는 전담 에이전트가 있다.

| 단계 | 이름 | 전담 에이전트 | 진행 | 한 줄 책임 |
|------|------|---------------|------|------------|
| 1 | 상담 | 상담가 (consultant) | ▣ 게이트 | 고객 요구사항을 듣고 정리한다 |
| 2 | 기획 | 기획자 (planner) | ▣ 게이트 | 요구사항을 제품 명세(PRD)로 구체화한다 |
| 3 | 설계 | 설계자 (architect) | ▣ 게이트 | 형태(웹/앱/자동화)·스택·구조를 결정하고 **이슈별 FEAT 문서**를 만든다 |
| 4 | GitHub 관리 | GitHub 관리자 (repo-manager) | ⚙ 자동 | 설계를 FEAT 1:1 매칭 이슈로 등록하고, 개발 후 **Git 반영을 검증**한다 |
| 5 | 개발 | 개발자 (builder) | ⚙ 자동 | FEAT 문서대로 MVP를 만든다 (이슈 단위 자동 루프, **최소 문서만 읽음**) |
| — | **QA 진입 승인** | (사람) | ▣ 게이트 | Git 원격 반영·산출물을 확인하고 `/승인`으로 QA를 연다 (v2 신설) |
| 6 | 자동 QA | 검수자 (reviewer) | ⚙ 자동 | **프로젝트 타입별 전략**으로 고객과 동일하게 실행 검증한다 |
| — | **사람 수동 테스트** | (사람) | ▣ 게이트 | 사람이 직접 써보고 증상을 보고한다 |
| ↻ | 증상 분석 | 증상분석가 (triage) | ⚙ 자동 | 증상을 재현·분석해 수정 이슈로 만든다 (수정 순환) |
| 7 | 납품 | 납품담당 (delivery) | ▣ 게이트 후 | 데모와 안내문을 패키징해 고객에게 전달한다 |

**진행 규칙**:
- **3단계(설계) 승인 = 개발 착수 승인.** 이후 4→5→6(이슈등록·개발·자동QA)은 멈춤 없이 자동으로 흐른다.
- 자동 QA(6) 후 **사람이 직접 수동 테스트**한다 (실사용 검증 게이트).
- 증상이 나오면 **증상분석가(triage)가 재현·분석 → 수정 이슈 등록 → 자동 수정 → 자동 재QA** 순환을 돈다 (사람 개입은 증상 보고와 최종 확인만).
- 사람이 OK 하면 7단계 납품.

> **4단계 GitHub 관리자는 GitHub 쓰기 권한(`gh`/push)을 가진 유일한 에이전트**다.
> 다른 에이전트는 진행 상태를 *보고*만 하고, 이슈 등록·close·재오픈은 GitHub 관리자가 전담한다.
> GitHub 관리자는 개발·검수 단계에서 공방장에 의해 재호출되어 이슈 상태를 동기화한다(생애주기 역할).
>
> **repo 정책 — 고객 프로젝트마다 별도 GitHub repo**를 둔다. (이슈 번호 네임스페이스 분리, 수정 요청 추적 명확화)
> repo 이름은 프로젝트명을 따르며, 4단계 첫 실행 시 없으면 생성한다. repo URL은 `이슈등록내역.md`에 기록한다.

전체 라인의 진행과 단계 간 인계는 **오케스트레이터(공방장)** 가 관장한다.
상세 명세는 `docs/pipeline.md`, 흐름도는 `docs/architecture.md` 참조.

## 4. 핸드오프 계약 (단계 간 산출물)

각 단계는 **자기 폴더에 정해진 산출물 파일**을 남긴다. 다음 단계는 그 파일만 입력으로 신뢰한다.

| 단계 | 입력 (이전 산출물) | 출력 (자기 산출물) |
|------|--------------------|--------------------|
| 1 상담 | 고객의 최초 요청 | `01-상담/요구사항-정의서.md` |
| 2 기획 | `01-상담/요구사항-정의서.md` | `02-기획/PRD.md` |
| 3 설계 | `02-기획/PRD.md` | `03-설계/설계서.md` + **`03-설계/features/FEAT-NN-*.md`** (이슈 1개당 1개) |
| 4 GitHub 관리 | `03-설계/설계서.md` + FEAT 문서들 | `04-깃허브/이슈등록내역.md` + 실제 GitHub 이슈(FEAT 1:1) |
| 5 개발 | **현재 이슈 + 그 이슈의 FEAT 문서** (최소) | `05-개발/` (코드 + `빌드노트.md`) + `progress.md` 갱신 |
| — QA 진입 승인 | 모든 이슈 close + Git 검증 | (사람 `/승인`) |
| 6 자동 QA | `05-개발/` + 위 모든 문서 | `06-검수/검수리포트.md` (타입별 QA) |
| ↻ 증상 분석 | 사람의 증상 보고 + `05-개발/` | `06-검수/증상분석log.md` → (이슈는 repo-manager가 등록) |
| 7 납품 | 전 단계 전체 + 사람 수동 테스트 통과 | `07-납품/` (`실행안내.md` + `고객전달문.md` + `증거자료체크리스트.md`) + `생산성요약.md` |

**규칙**: 산출물 파일명·위치는 고정이다. 에이전트는 자기 단계 폴더 밖에 파일을 쓰지 않는다.
(예외: `progress.md`·`생산성요약.md`는 프로젝트 루트의 공용 추적 파일로, 해당 단계 에이전트가 자기 기록만 누적한다.)

## 4-1. 모델 배치 (v2 — 토큰 효율)

공정이 안정화되어 모든 단계에 고성능 모델이 필요하지 않다. **판단이 무거운 곳에만 Opus, 나머지는 Sonnet, 기계적 작업은 Haiku 실험 가능.**

| 모델 | 적용 | 이유 |
|------|------|------|
| **Opus** | 설계자(architect), 증상분석(triage) | 형태·스택 결정, 범위 충돌 판단, 피벗 결정 등 무거운 판단 |
| **Sonnet** | consultant·planner·repo-manager·**builder**·reviewer·delivery | 요구사항 정리, 문서 작성, FEAT 기반 일반 구현, 검수, 납품 |
| **Haiku** (실험) | 체크리스트 검수, 로그 요약, 증거자료 목록 정리 | 명확한 입력 → 기계적 출력 |

> ★ v2 핵심 전환: builder를 Opus→**Sonnet**으로 내린다. 단, 이는 **FEAT 문서가 충분히 구체적일 때만** 안전하다.
> 그래서 v2는 **FEAT 문서 승인 게이트(설계 단계)** 를 강화한다 — 싼 모델로 구현할수록 FEAT 품질이 곧 결과 품질이다.

## 5. 디렉토리 구조

```
IT_make_some/
├── CLAUDE.md                  # 이 문서 (공방 헌법)
├── README.md                  # 공방 소개
├── .claude/
│   ├── agents/                # 단계별 전담 에이전트 정의 (기반공사 이후 채움)
│   ├── commands/              # 오케스트레이터 커맨드 (예: /신규프로젝트, /다음단계)
│   ├── skills/                # 재사용 스킬 (예: 산출물 템플릿 생성)
│   └── rules/                 # 단계별 작업 규칙 (paths 스코프로 lazy-load)
├── docs/
│   ├── architecture.md        # 설계도 — 오케스트레이션 흐름도, 에이전트 명세
│   └── pipeline.md            # 7단계 생산 라인 상세 (각 단계 입출력·도구범위)
├── templates/                 # 산출물 템플릿 (요구사항정의서, PRD, 설계서 등)
└── projects/
    ├── _template/             # 신규 프로젝트 생성 시 복제되는 빈 골격 (7개 단계 폴더)
    └── <프로젝트명>/          # 실제 고객 프로젝트 (단계별 산출물 누적)
```

---

## 6. 신규 프로젝트 시작 절차

1. `projects/_template/` 를 `projects/<프로젝트명>/` 으로 복제한다.
2. 1단계(상담) 에이전트부터 라인을 시작한다.
3. 각 단계 완료 → 사람 승인 → 다음 단계. (휴먼 게이트)

---

## 7. 금지 사항

- 한 에이전트가 여러 단계를 동시에 처리하지 않는다. (단계당 한 에이전트)
- 사람 승인 없이 단계를 건너뛰지 않는다.
- 공방 공용 설정(`.claude/`, `CLAUDE.md`, `docs/`, `templates/`)을 프로젝트 작업 중 임의 수정하지 않는다.
- 산출물을 정해진 폴더 밖에 쓰지 않는다.
- MVP 범위를 넘는 과잉 구현(완성도 욕심)을 하지 않는다. "확인 가능한 최소"가 목표다.
